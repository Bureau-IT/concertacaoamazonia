## Context

O JetElements Download Button com attachment ID 90286 está em `/cultura/linha-do-tempo/` (blog 2), mas o attachment existe apenas no `wp_posts` (blog 1). O problema é sistêmico: **5 widgets JetElements + 2 handlers** sofrem do mesmo bug cross-blog por chamarem `wp_get_attachment_url()` e `get_attached_file()` sem context switching.

O `gallery-crossblog-fix.php` existente (v1.0.0) resolve parcialmente (apenas `wp_get_attachment_url` para gallery/lightbox do Elementor). A solução é **renomear + expandir** esse plugin em um único arquivo canônico com escopo genérico.

**Widgets JetElements afetados confirmados:**
- `jet-elements-audio.php` — `wp_get_attachment_url()` → áudios não carregam
- `jet-elements-video.php` — `wp_get_attachment_url()` → vídeos self-hosted não carregam
- `jet-elements-download-button.php` — `wp_get_attachment_url()` → nome do arquivo não exibido
- `class-jet-elements-download-handler.php` — `get_attached_file()` ×3 → tamanho não calculado + download falha
- `class-jet-elements-svg-manager.php` — `get_attached_file()` ×2 → dimensions SVG não calculadas

**O NML já cobre `wp_get_attachment_image_src()`** — não duplicar.

## Decisões de design

| Decisão | Escolha | Motivo |
|---|---|---|
| `try/finally` | **Não usar** | Padrão do projeto e do NML (referência): sequência linear com guards |
| `static $resolved/$failed` | **Usar** | Melhoria sobre NML e plugin atual — evita `switch_to_blog()` por ID já processado |
| Escopo do guard | `!== 2` | Site tem exatamente 2 blogs; hardcoded é correto e explícito |
| `wp_cache_set` timing | **Após `restore_current_blog()`** | Confirmado em `object-cache.php`: `blog_prefix` já voltou para 2, escrita vai ao namespace correto |
| `update_meta_cache` | **Dentro do guard de existência** | Evita query desnecessária ao banco do blog 1 para IDs inexistentes |

```php
<?php
/**
 * Plugin Name:  BIT Cross-Blog Attachment Fix
 * Description:  Fixes cross-blog attachment resolution for blog 2 (/cultura/)
 *               accessing attachments stored in blog 1 (site principal).
 *               Covers: wp_get_attachment_url (lightbox, audio, video, download
 *               button), get_attached_file (download handler, SVG manager) e
 *               pré-população de cache para process_download e widget gallery.
 *               Complementa o Network Media Library, que já cobre
 *               wp_get_attachment_image_src().
 * Version:      1.0.0
 * Author:       Bureau IT
 * Network:      true
 */

if ( ! defined( 'ABSPATH' ) ) exit;

// ─────────────────────────────────────────────────────────────────────────────
// 1. Fix wp_get_attachment_url() para attachments cross-blog (blog 2 → blog 1).
//    Cobre: lightbox do Elementor (add_lightbox_data_attributes), JetElements
//    audio, video e download button (exibição do nome do arquivo).
//    Cache estático $resolved/$failed evita switch_to_blog() repetido por ID.
// ─────────────────────────────────────────────────────────────────────────────
add_filter( 'wp_get_attachment_url', function ( $url, $post_id ) {
	static $switched = false;
	static $resolved = [];
	static $failed   = [];

	if ( get_current_blog_id() !== 2 ) return $url;
	if ( $url ) return $url;
	if ( isset( $failed[ $post_id ] ) ) return $url;
	if ( isset( $resolved[ $post_id ] ) ) return $resolved[ $post_id ];
	if ( $switched ) return $url;

	$switched = true;
	switch_to_blog( 1 );
	$b1_url = wp_get_attachment_url( $post_id );
	restore_current_blog();
	$switched = false;

	if ( $b1_url ) {
		$resolved[ $post_id ] = $b1_url;
		return $b1_url;
	}

	$failed[ $post_id ] = true;
	return $url;
}, 999, 2 );

// ─────────────────────────────────────────────────────────────────────────────
// 2. Fix get_attached_file() para attachments cross-blog (blog 2 → blog 1).
//    Cobre: JetElements download handler (get_file_size + process_download) e
//    SVG manager (cálculo de dimensions).
//    Cache estático $resolved/$failed evita switch_to_blog() repetido por ID.
// ─────────────────────────────────────────────────────────────────────────────
add_filter( 'get_attached_file', function ( $file, $attachment_id ) {
	static $switched = false;
	static $resolved = [];
	static $failed   = [];

	if ( get_current_blog_id() !== 2 ) return $file;
	if ( $file ) return $file;
	if ( isset( $failed[ $attachment_id ] ) ) return $file;
	if ( isset( $resolved[ $attachment_id ] ) ) return $resolved[ $attachment_id ];
	if ( $switched ) return $file;

	$switched = true;
	switch_to_blog( 1 );
	$b1_file = get_attached_file( $attachment_id );
	restore_current_blog();
	$switched = false;

	if ( $b1_file ) {
		$resolved[ $attachment_id ] = $b1_file;
		return $b1_file;
	}

	$failed[ $attachment_id ] = true;
	return $file;
}, 999, 2 );

// ─────────────────────────────────────────────────────────────────────────────
// 3. Pré-popula object cache do blog 2 com post + meta do blog 1 antes que
//    JetElements process_download() (init priority 99) chame get_post($id).
//    Sem isso, get_post() retorna null → Fatal Error PHP 8.
//    Hash estrutura: [ 'sha1hash' => post_id ] (gravada no blog 2 pelo widget).
//    update_meta_cache() chamado somente se o post existir no blog 1.
//    wp_cache_set() após restore_current_blog() — escreve no namespace do blog 2.
// ─────────────────────────────────────────────────────────────────────────────
add_action( 'init', function () {
	if ( get_current_blog_id() !== 2 ) return;
	if ( empty( $_GET['jet_download'] ) ) return;

	// Resolve hash → ID no blog 2 (onde a option foi gravada pelo widget)
	$hashes = get_option( 'jet_elements_download_button_hashes', [] );
	$hash   = sanitize_text_field( wp_unslash( $_GET['jet_download'] ) );
	$id     = isset( $hashes[ $hash ] ) ? (int) $hashes[ $hash ] : 0;

	// Fallback: tenta ler do blog 1 (edge case: hash gerado no admin do blog 1)
	if ( ! $id ) {
		switch_to_blog( 1 );
		$hashes_b1 = get_option( 'jet_elements_download_button_hashes', [] );
		restore_current_blog();
		$id = isset( $hashes_b1[ $hash ] ) ? (int) $hashes_b1[ $hash ] : 0;
	}

	if ( ! $id ) return;

	// Busca post do blog 1; warm de meta somente se existir (evita query desnecessária)
	switch_to_blog( 1 );
	$post = get_post( $id );
	$meta = false;
	if ( $post && $post->post_type === 'attachment' ) {
		update_meta_cache( 'post', [ $id ] );
		$meta = wp_cache_get( $id, 'post_meta' );
	}
	restore_current_blog();

	// wp_cache_set aqui usa blog_prefix=2 (restore_current_blog já executou)
	if ( $post && $post->post_type === 'attachment' ) {
		wp_cache_set( $id, $post, 'posts' );
		if ( $meta ) wp_cache_set( $id, $meta, 'post_meta' );
	}
}, 5 );

// ─────────────────────────────────────────────────────────────────────────────
// 4. Antes do render do widget gallery do Elementor, pré-popula o object cache
//    e postmeta cache (namespace blog 2) com dados de attachments do blog 1.
//    Garante que get_post($id) e get_post_meta($id, '_wp_attachment_image_alt')
//    retornem dados corretos para IDs do blog 1 dentro do render da gallery.
// ─────────────────────────────────────────────────────────────────────────────
add_action( 'elementor/widget/before_render_content', function ( $widget ) {
	if ( get_current_blog_id() !== 2 ) return;
	if ( $widget->get_name() !== 'gallery' ) return;

	$settings = $widget->get_settings_for_display();
	$ids = [];
	foreach ( $settings['galleries'] ?? [] as $gallery ) {
		foreach ( $gallery['multiple_gallery'] ?? [] as $item ) {
			$id = absint( $item['id'] ?? 0 );
			if ( $id ) $ids[] = $id;
		}
	}
	if ( empty( $ids ) ) return;

	// IDs ausentes ou com tipo errado em blog 2
	$missing = [];
	foreach ( $ids as $id ) {
		$post = get_post( $id );
		if ( ! $post || $post->post_type !== 'attachment' ) {
			$missing[] = $id;
		}
	}
	if ( empty( $missing ) ) return;

	// Busca posts e meta do blog 1 (batch — uma única switch_to_blog por render)
	switch_to_blog( 1 );
	update_meta_cache( 'post', $missing );
	$b1_posts = [];
	$b1_metas = [];
	foreach ( $missing as $id ) {
		$post = get_post( $id );
		if ( $post && $post->post_type === 'attachment' ) {
			$b1_posts[ $id ] = $post;
			$b1_metas[ $id ] = wp_cache_get( $id, 'post_meta' );
		}
	}
	restore_current_blog();

	// Popula cache do blog 2 (blog_prefix=2 após restore)
	foreach ( $b1_posts as $id => $post ) {
		wp_cache_set( $id, $post, 'posts' );
		if ( ! empty( $b1_metas[ $id ] ) ) {
			wp_cache_set( $id, $b1_metas[ $id ], 'post_meta' );
		}
	}
}, 5, 1 );
```

# Dispatching Parallel Agents

## Overview

You delegate tasks to specialized agents with isolated context. By precisely crafting their instructions and context, you ensure they stay focused and succeed at their task. They should never inherit your session's context or history — you construct exactly what they need. This also preserves your own context for coordination work.

When you have multiple unrelated failures (different test files, different subsystems, different bugs), investigating them sequentially wastes time. Each investigation is independent and can happen in parallel.

**Core principle:** Dispatch one agent per independent problem domain. Let them work concurrently.

## When to Use

```dot
digraph when_to_use {
    "Multiple failures?" [shape=diamond];
    "Are they independent?" [shape=diamond];
    "Single agent investigates all" [shape=box];
    "One agent per problem domain" [shape=box];
    "Can they work in parallel?" [shape=diamond];
    "Sequential agents" [shape=box];

    "Are they independent?" -> "Single agent investigates all" [label="no - related"];
    "Are they independent?" -> "Can they work in parallel?" [label="yes"];
    "Can they work in parallel?" -> "Parallel dispatch" [label="yes"];
    "Can they work in parallel?" -> "Sequential agents" [label="no - shared state"];
}
```

**Use when:**
- 3+ test files failing with different root causes
- Multiple subsystems broken independently
- Each problem can be understood without context from others
- No shared state between investigations

**Don't use when:**
- Failures are related (fix one might fix others)
- Need to understand full system state
- Agents would interfere with each other

## The Pattern

### 1. Identify Independent Domains

Group failures by what's broken:
- File A tests: Tool approval flow
- File B tests: Batch completion behavior
- File C tests: Abort functionality

Each domain is independent - fixing tool approval doesn't affect abort tests.

### 2. Create Focused Agent Tasks

Each agent gets:
- **Specific scope:** One test file or subsystem
- **Clear goal:** Make these tests pass
- **Constraints:** Don't change other code
- **Expected output:** Summary of what you found and fixed

### 3. Dispatch in Parallel

```typescript
// In Claude Code / AI environment
Task("Fix agent-tool-abort.test.ts failures")
Task("Fix batch-completion-behavior.test.ts failures")
Task("Fix tool-approval-race-conditions.test.ts failures")
// All three run concurrently
```

### 4. Review and Integrate

When agents return:
- Read each summary
- Verify fixes don't conflict
- Run full test suite
- Integrate all changes

## Agent Prompt Structure

Good agent prompts are:
1. **Focused** - One clear problem domain
2. **Self-contained** - All context needed to understand the problem
3. **Specific about output** - What should the agent return?

```markdown
Fix the 3 failing tests in src/agents/agent-tool-abort.test.ts:

1. "should abort tool with partial output capture" - expects 'interrupted at' in message
2. "should handle mixed completed and aborted tools" - fast tool aborted instead of completed
3. "should properly track pendingToolCount" - expects 3 results but gets 0

These are timing/race condition issues. Your task:

1. Read the test file and understand what each test verifies
2. Identify root cause - timing issues or actual bugs?
3. Fix by:
   - Replacing arbitrary timeouts with event-based waiting
   - Fixing bugs in abort implementation if found
   - Adjusting test expectations if testing changed behavior

Do NOT just increase timeouts - find the real issue.

Return: Summary of what you found and what you fixed.
```

## Common Mistakes

**❌ Too broad:** "Fix all the tests" - agent gets lost
**✅ Specific:** "Fix agent-tool-abort.test.ts" - focused scope

**❌ No context:** "Fix the race condition" - agent doesn't know where
**✅ Context:** Paste the error messages and test names

**❌ No constraints:** Agent might refactor everything
**✅ Constraints:** "Do NOT change production code" or "Fix tests only"

**❌ Vague output:** "Fix it" - you don't know what changed
**✅ Specific:** "Return summary of root cause and changes"

## When NOT to Use

**Related failures:** Fixing one might fix others - investigate together first
**Need full context:** Understanding requires seeing entire system
**Exploratory debugging:** You don't know what's broken yet
**Shared state:** Agents would interfere (editing same files, using same resources)

## Code Review: `bit-crossblog-attachment-fix.php`

File: `/Users/dcambria/scripts/server-tools/v2/docker-dev/sites/concertacao/wordpress/wp-content/mu-plugins/bit-crossblog-attachment-fix.php`

---

### What was done well

The overall architecture is sound. The two-layer strategy (pre-warming cache + corrective filter at priority 999) correctly handles the case where `wp_get_attachment_url()` reaches the filter with a wrong URL rather than returning `false` outright, because `get_post()` now succeeds due to pre-warmed cache. The use of `static $resolved` / `static $failed` per-ID memoization avoids redundant `switch_to_blog()` round-trips. The helper `bit_crossblog_warm_cache()` correctly batches `update_meta_cache()` inside a single blog switch, which is efficient. Comments are thorough and accurately describe the intent of each hook.

---

### Issues Found

**CRITICAL**

**Issue 1 — Recursion guard `$switched` is not exception-safe (Hooks 1 and 2, lines 70 and 109)**

Both closures use `static $switched = false` as a guard, then set it to `true`, call back into WordPress (`wp_get_attachment_url()` / `get_attached_file()`), and reset it to `false` after `restore_current_blog()`. If the inner WordPress call throws an exception or triggers a PHP fatal before the `restore_current_blog()` line, `$switched` stays `true` permanently for the rest of the request. All subsequent calls from the same request for any attachment ID would then bypass the fix and return the wrong URL or path.

PHP 8 is in use here (confirmed by CLAUDE.md). While PHP does not have `finally`-based exception semantics in the same way some runtimes do, an uncaught `TypeError` or `Error` inside `wp_get_attachment_url()` during the switched context would leave `$switched = true` permanently. The correct pattern:

```php
$switched = true;
try {
    switch_to_blog( 1 );
    $b1_url = wp_get_attachment_url( $post_id );
    restore_current_blog();
} catch ( \Throwable $e ) {
    restore_current_blog();
    $switched = false;
    $failed[ $post_id ] = true;
    return $url;
}
$switched = false;
```

This applies identically to Hook 2 (`get_attached_file`).

---

**Issue 2 — Hook 1: path where `$url` is falsy falls through to `switch_to_blog` unnecessarily (line 81)**

The guard on line 81 is:

```php
if ( $url && strpos( $url, $b2_upload ) === false ) return $url;
```

When `$url` is `false` or empty (the attachment was not found in blog 2's table at all), this condition is false, and execution continues to `switch_to_blog(1)`. This is actually intentional per the comment on line 66 ("Also cobre IDs que chegam sem cache"). However, there is no path where `$url` being `false` is distinguished from `$url` being a correct blog-1 URL that simply does not contain `/sites/2/`. Both cases fall through.

The real edge case: if the attachment genuinely lives in blog 2 (an attachment actually uploaded to blog 2, not a cross-blog reference), its URL will not contain `/sites/2/` either — wait, actually for a multisite subdirectory install in this configuration, uploads to blog 2 go to `/wp-content/uploads/sites/2/`, so a correctly resolved blog-2 attachment URL would contain that string. The check is correct for attachments native to blog 2. But `$url = false` (meaning blog 2 has no record of the post and no pre-warmed cache) will also fall through to the `switch_to_blog` block. This is safe and intentional, just not clearly commented for this case.

**However**, there is a subtle real bug here: if `$url` is an empty string `""` rather than `false` (WordPress can return `""` in some filter chains), the `if ( $url && ... )` check treats it the same as `false`, and the code proceeds to `switch_to_blog(1)` — which is the right behavior. This is not actually a bug, but it differs from what the comment implies.

Severity: low documentation issue only. No code change required, but the comment on line 79 should mention the `$url === false` path explicitly.

---

**IMPORTANT**

**Issue 3 — `bit_crossblog_warm_cache()`: `wp_cache_get( $id, 'post_meta' )` after `update_meta_cache()` may return `false` for legitimately empty meta (lines 42-43, 52-53)**

After `update_meta_cache( 'post', $missing )` runs inside the blog-1 context, meta for each post is populated into the cache. Then:

```php
$b1_metas[ $id ] = wp_cache_get( $id, 'post_meta' );
```

If the attachment has no postmeta rows at all, WordPress stores a sentinel empty array `[]` in the cache to avoid re-querying. This will be retrieved correctly. However, if `update_meta_cache()` was not called before `get_post()`, there is a race: `get_post()` on line 39 may trigger its own meta loading. This is fine because `update_meta_cache()` runs first (line 35). The order is correct.

The actual issue is line 51-53:

```php
if ( ! empty( $b1_metas[ $id ] ) ) {
    wp_cache_set( $id, $b1_metas[ $id ], 'post_meta' );
}
```

The `! empty()` guard means that if an attachment has no postmeta (empty array), its meta is never written into the blog-2 cache namespace. Any subsequent call to `get_post_meta()` for that attachment ID in blog-2 context will hit the DB (or return wrong results from blog-2 meta). For attachments used in download buttons, the `_wp_attached_file` meta key is critical for `get_attached_file()` to construct the path. If this meta is missing from the cache, `get_attached_file()` issues a new DB query — against the blog-2 tables — and returns `false`, which then requires Hook 2 to do a `switch_to_blog` fallback.

The fix: always write the meta into the blog-2 cache, even if it is an empty array, to prevent a cache miss causing a wrong-context DB query:

```php
if ( $b1_metas[ $id ] !== false ) {
    wp_cache_set( $id, $b1_metas[ $id ] ?: [], 'post_meta' );
}
```

This uses `!== false` to distinguish "meta was cached and is empty" from "cache miss" (where `wp_cache_get` returns `false`). The `?: []` ensures an empty array is stored rather than `false`.

---

**Issue 4 — Hook 4 (`before_render_content`): `jet-audio` settings key assumption may not match actual JetElements structure (line 196)**

The code assumes:

## C4 — Fallback hash lookup (hook 3, init)

**Resultado: PROBLEMA — condicao de fallback demasiado restritiva**

A mudanca de `! $id` para `! $id && empty( $hashes )` introduz um falso negativo para o edge case que pretende cobrir.

O edge case descrito no comentario e: "hash gerado no admin do blog 1". Isso ocorre quando um editor configura o widget download button enquanto navegando no blog 1 (`/wp-admin/`) em vez do blog 2 (`/cultura/wp-admin/`). Nesse caso, o hash e gravado na option `jet_elements_download_button_hashes` do blog 1, nao do blog 2.

O problema com `empty( $hashes )` como condicao adicional: o blog 2 pode perfeitamente ter sua propria option `jet_elements_download_button_hashes` com hashes validos de outros download buttons configurados corretamente no admin do blog 2. Nesse cenario, `$hashes` nao estara vazio, a condicao `empty( $hashes )` sera `false`, e o fallback para o blog 1 nunca executara — mesmo que o hash especifico da request seja um hash que foi gravado no blog 1.

Exemplo concreto do problema:
- Blog 2 tem 3 download buttons configurados corretamente: `$hashes = ['abc' => 10, 'def' => 11, 'ghi' => 12]`
- Um quarto download button foi configurado no admin do blog 1: hash `'xyz'` gravado na option do blog 1
- Request: `?jet_download=xyz`
- Resultado com v1.1.0: `$id = 0` (hash nao existe no blog 2), `empty($hashes)` e `false` (blog 2 tem 3 hashes), fallback nao executa, `return` na linha 186 encerra sem processar

A condicao correta para o edge case e: o hash nao foi encontrado no blog 2 (`! $id`), independente de quantos outros hashes o blog 2 possua. A versao v1.0.0 (`! $id`) era mais correta para este caso especifico, com o custo de um `switch_to_blog` extra para qualquer hash invalido.

Uma correcao que equilibra os dois objetivos:

```php
// Fallback: tenta ler do blog 1 apenas quando o hash nao foi encontrado no blog 2.
// Evita switch_to_blog desnecessario para hashes genuinamente invalidos
// verificando primeiro se o hash tem o formato esperado.
if ( ! $id && strlen( $hash ) === 40 ) { // SHA1 sempre tem 40 chars hex
    switch_to_blog( 1 );
    $hashes_b1 = get_option( 'jet_elements_download_button_hashes', [] );
    restore_current_blog();
    $id = isset( $hashes_b1[ $hash ] ) ? (int) $hashes_b1[ $hash ] : 0;
}
```

## Ponto 1 — `static $warmed` em `bit_crossblog_warm_cache()`: escopo de request vs. long-running

**Analise:**

A preocupacao com WP-CLI e cron e valida do ponto de vista teorico, mas nao e um problema real para este caso de uso. Dois argumentos encerram a questao:

Primeiro, o crescimento maximo de `$warmed` e igual ao numero de attachment IDs distintos referenciados nas paginas do blog 2. No contexto deste multisite de 2 blogs com um conjunto finito de widgets JetElements, esse numero e da ordem de dezenas a centenas — nunca milhar. O overhead de memoria de um array PHP com 500 inteiros e desprezivel (abaixo de 50 KB).

Segundo, o proposito de `$warmed` e precisamente evitar chamadas repetidas a `switch_to_blog()` para o mesmo ID durante um processo. Em WP-CLI ou cron de longa duracao, esse comportamento e desejavel: se o mesmo ID ja foi processado corretamente, nao ha motivo para re-verificar. O risco de um ID ter sido promovido/deletado no intervalo e um risco aceito em qualquer cache in-memory.

**Veredito: APROVADO.** O comportamento e correto e intencional para todos os cenarios de uso deste plugin.

## Ponto 2 — `$resolved`/`$failed` (hooks 1 e 2) vs. `$warmed` (warm_cache): redundancia?

**Analise:**

Os dois mecanismos nao sao redundantes — eles operam em camadas diferentes e sao ambos necessarios.

`$warmed` controla quantas vezes o bloco `switch_to_blog(1) ... restore_current_blog()` e executado na funcao de pre-aquecimento. Ele opera na granularidade de IDs e evita que o mesmo ID seja aquecido duas vezes em requisicoes com multiplos widgets (ex.: galeria com 20 imagens — `bit_crossblog_warm_cache` e chamada uma vez com o array completo, mas se chamada novamente com subconjunto, `$missing` filtra os ja processados).

`$resolved`/`$failed` nos hooks 1 e 2 controlam quantas vezes o bloco `switch_to_blog` dentro dos filtros e re-executado para um dado ID. Esses filtros podem ser acionados por outros caminhos que nao passam pelo warm_cache (ex.: um ID chegando diretamente em `wp_get_attachment_url()` sem ter sido pre-aquecido pelo hook 4). Eles tambem armazenam o resultado final (a URL ou path correto), evitando um switch_to_blog completo em chamadas subsequentes para o mesmo ID.

`$switched` (booleano) e diferente dos outros dois: e um guard de reentrancia que evita recursao infinita quando o filtro chama `wp_get_attachment_url($post_id)` com `switch_to_blog(1)` ativo — sem ele, o filtro seria invocado recursivamente dentro de si mesmo. Esse guard tem escopo de chamada, nao de ID.

Em resumo: `$warmed` cuida do pre-aquecimento em lote; `$resolved`/`$failed` cuidam dos filtros reativos; `$switched` cuida da reentrancia. Os tres papeis sao distintos.

**Veredito: APROVADO.** Nenhuma redundancia real. A separacao de responsabilidades e correta.

## Ponto 3 — Hook 3: `init` priority 5 e disponibilidade da option `jet_elements_download_button_hashes`

**Analise:**

Options do WordPress com `autoload = yes` sao carregadas em massa na funcao `wp_load_alloptions()`, que e chamada por `get_alloptions()` internamente. Isso ocorre muito antes do hook `init` — tipicamente durante `wp-settings.php`, na fase de setup do objeto `$wpdb` e inicializacao do cache. A funcao `get_option()` em si e segura a partir do momento em que o banco de dados e o object cache estao inicializados, o que acontece antes de qualquer hook de usuario.

O risco real seria se `jet_elements_download_button_hashes` tivesse `autoload = no`, caso em que a option nao estaria no cache mas ainda seria legivel via query direta (o `get_option()` faz o fallback corretamente). A priority 5 do `init` e conservadora o suficiente — o JetElements processa o download em priority 99 do mesmo hook, dando margem confortavel de 94 niveis de prioridade.

Ha um detalhe sutil, porem nao problematico: o hook 3 chama `get_option()` no contexto do blog 2 (blog atual). Se o JetElements grava a option com `update_option()` durante o request de save do widget no blog 2, ela esta no namespace do blog 2, e a leitura esta correta. O fallback para o blog 1 (linhas 181-186) cobre o edge case de hash gravado no admin do blog 1.

**Veredito: APROVADO.** A priority 5 e suficiente. O mecanismo de fallback para o blog 1 e correto.

## Ponto 5 — `restore_current_blog()` dentro do `try`: risco de duplo restore no `catch`

**Analise:**

A pergunta e: `restore_current_blog()` pode lancar uma excecao?

Olhando a implementacao do WordPress core, `restore_current_blog()` realiza as seguintes operacoes: pop do `$_wp_switched_stack`, chamada a `switch_to_blog()` com o blog anterior, limpeza de algumas variaveis globais. Em circunstancias normais, nenhuma dessas operacoes lanca excecao. Poderia lancar uma excecao se um hook em `switch_to_blog` (como `switch_blog`) chamasse codigo de terceiro que lancasse — o que e teoricamente possivel mas extremamente improvavel em um ambiente controlado.

Se `restore_current_blog()` lancasse dentro do `try`, o `catch` chamaria `restore_current_blog()` novamente. O comportamento de um "duplo restore" no WordPress e: o segundo pop do stack poderia restaurar para um blog anterior ao esperado (o stack seria consumido uma vez a mais). Em um request normal com um unico nivel de switch, o stack teria no maximo o blog original — o segundo restore tentaria pop em stack vazio e retornaria sem fazer nada (o WordPress trata stack vazio graciosamente em `restore_current_blog()`).

Na pratica, o risco real e zero neste ambiente. Em teoria, o padrao mais robusto seria usar `finally`:

```php
try {
    switch_to_blog( 1 );
    $b1_url = wp_get_attachment_url( $post_id );
} catch ( \Throwable $e ) {
    $switched = false;
    $failed[ $post_id ] = true;
} finally {
    restore_current_blog();
}
```

Isso eliminaria o duplo restore de forma estrutural. Porem, adotar `finally` aqui seria uma mudanca de padrao em relacao ao restante do codigo, e o risco mitigado e academico.

**Veredito: SUGESTAO** (baixa prioridade). O codigo atual e seguro para todos os cenarios realistas. O padrao `finally` seria mais elegante e estruturalmente correto, mas nao e obrigatorio.

O guard atual `get_current_blog_id() !== 2` e suficiente para single-site por duas razoes:

Primeiro, em instalacoes single-site, `get_current_blog_id()` retorna `1` (sempre). A condicao `!== 2` e `true`, e todos os hooks retornam imediatamente na primeira linha. Nenhum codigo alem dessa verificacao e executado.

## Seguranca

### Ponto 1 — sanitize_text_field para hash SHA1

Veredito: **suficiente, sem gap.**

`sanitize_text_field` aplicado sobre `wp_unslash` e correto para esse caso. Um hash SHA1 valido e exatamente 40 caracteres hexadecimais `[0-9a-f]`. `sanitize_text_field` remove tags HTML, caracteres de controle e espacos extras — nenhuma dessas remocoes quebra um hash valido. O vetor que importa aqui nao e injecao de HTML, e qualquer outro caractere fora do alfabeto hex. Esse caractere nao seria removido por `sanitize_text_field`, mas isso e irrelevante porque a sanitizacao nao precisa reduzir o valor a hex puro: o valor sanitizado e usado apenas como chave de array em `$hashes[$hash]`. Se um atacante enviar `AAAA<script>` o hash simplesmente nao existira em `$hashes` e o `$id` retornara `0`. Nao ha SQL, nao ha eval, nao ha path traversal a partir desse valor. Uma sanitizacao mais estrita com `preg_replace('/[^0-9a-f]/i', '', $hash)` seria mais expressiva do ponto de vista de intencao do codigo, mas nao muda o comportamento de seguranca em nada.

### Ponto 2 — hash como array key lookup

Veredito: **sem risco, PHP nao tem prototype pollution.**

Prototype pollution e um conceito de JavaScript que nao existe em PHP. Arrays PHP sao estruturas de dados por valor, nao objetos com cadeia de prototipos. Qualquer string pode ser chave de array PHP sem efeitos colaterais na estrutura do array. O cast `(int)` no valor de retorno e a protecao correta: mesmo que `$hashes[$hash]` contivesse um valor nao-numerico por corrupcao do option, o cast o tornaria `0`, e o `if (!$id) return` bloquearia a execucao.

### Ponto 3 — switch_to_blog(1) hardcoded

Veredito: **sem risco de SSRF ou traversal, e a escolha correta.**

SSRF (Server-Side Request Forgery) envolve fazer o servidor emitir requests HTTP para destinos controlados pelo atacante. `switch_to_blog()` nao emite request HTTP algum — ele altera ponteiros internos do WordPress (`$wpdb->prefix`, `$current_blog`, blog_prefix do object cache). O blog ID 1 nunca e exposto ao usuario: o valor `1` esta literalmente hardcoded no PHP server-side e nao ha parametro de entrada que o influencie. Path traversal tambem nao se aplica: nao ha construcao de caminho de arquivo a partir de entrada do usuario neste fluxo. Hardcodar o blog ID 1 e a decisao certa dado que o header do plugin documenta explicitamente que ele e "especifico para multisite de 2 blogs onde o blog 1 detém a mídia." Generalizar isso seria engenharia desnecessaria para o contexto.

### Ponto 4 — wp_cache_set apos restore_current_blog e cache poisoning

Veredito: **comportamento intencional e correto, risco de poisoning e real mas mitigado pela logica de warm_cache.**

A analise completa em tres partes:

**Parte A — o namespace.** Apos `restore_current_blog()`, o WordPress restaura o blog_prefix do object cache para o blog 2. `wp_cache_set($id, $post, 'posts')` grava com prefixo `2:posts`, nao `1:posts`. Isso e intencional: a funcao `bit_crossblog_warm_cache` existe precisamente para pré-popular o namespace do blog 2 com dados do blog 1, para que as funcoes de blog 2 (como `get_post()` chamado por `wp_get_attachment_url()`) encontrem o objeto sem precisar de nova query SQL.

**Parte B — o risco de poisoning.** O risco existe teoricamente: se um attachment ID `X` existir tanto no blog 1 (como attachment) quanto no blog 2 (como qualquer post type), `bit_crossblog_warm_cache` sobrescreve o objeto do blog 2 no cache com o objeto do blog 1. Porem, a pre-condicao da warm_cache para agir sobre um ID e que `wp_cache_get($id, 'posts')` no blog 2 retorne `false` ou retorne um post cujo `post_type !== 'attachment'`. Se o ID existe no blog 2 como post normal, o cache do blog 2 ja tera esse objeto com o tipo correto, a condicao `! $post || $post->post_type !== 'attachment'` sera falsa para esse ID, e ele sera excluido de `$missing`. A warm_cache nao sobrescreve entradas de cache existentes e validas do blog 2.

**Parte C — gap residual.** O gap existe somente na janela de cold cache: se o blog 2 acabou de ter seu cache limpo, e um request simultâneo tenta renderizar um widget com attachment ID `X` que existe no blog 2 como um post normal (nao attachment), esse ID poderia acabar em `$missing` se o cache do blog 2 ainda nao tiver sido populado. Esse cenario e improvavel neste site especifico porque os IDs injetados nos widgets do blog 2 sao sempre attachments do blog 1 — nao haveria sobreposicao de IDs entre os dois blogs. Nao e um gap que requer correcao; e um presuposto documentado no header do plugin.

## Performance

### Ponto 5 — custo real de switch_to_blog() por request

Veredito: **maximo de 2 switch_to_blog() em um request tipico com 1 download button.**

Rastreando os caminhos:

**Hook 3 (init, priority 5):** Ativado apenas quando `$_GET['jet_download']` esta presente, ou seja, somente no request de download, nao em requests de pagina normal. Quando ativado: 1 `switch_to_blog(1)` dentro de `bit_crossblog_warm_cache` + possivelmente 1 `switch_to_blog(1)` no fallback para buscar hashes do blog 1 = maximo 2 switch_to_blog no request de download.

**Hook 4 (elementor/widget/before_render_content, priority 5):** Ativado em requests de pagina normais para widgets de gallery, jet-audio, jet-video, jet-download-button. Para uma pagina com 1 download button widget: invoca `bit_crossblog_warm_cache([id])`, que faz 1 `switch_to_blog(1)`. Se o ID ja estiver em `$warmed` (static), a funcao retorna imediatamente sem switch.

**Hooks 1 e 2 (filtros, priority 999):** Esses filtros executam durante o render do widget. Se a warm_cache do hook 4 teve sucesso, o cache do blog 2 ja tem o post e a URL construida por `wp_get_attachment_url()` ja estara correta (o filtro detecta isso via `strpos($url, '/sites/2/')` e retorna antecipadamente, sem switch_to_blog). Se a warm_cache falhou ou o ID chegou sem passar pelo hook 4, o hook 1 faz 1 `switch_to_blog(1)`, e o resultado e memoizado em `$resolved`. O hook 2 tem logica identica com `$resolved` proprio.

**Total no caso mais comum (pagina com 1 download button, warm_cache bem-sucedida):** 1 switch_to_blog (no hook 4), hooks 1 e 2 retornam por early return. Total: 1 switch_to_blog.

**Total no pior caso (hook 4 falha + hooks 1 e 2 precisam agir + request de download):** 3 switch_to_blog por request de pagina + 2 no request de download. O static `$switched` em hooks 1 e 2 e o static `$warmed` em warm_cache previnem repeticao para o mesmo ID dentro do mesmo request.

O custo de `switch_to_blog()` em si e baixo (sem I/O, apenas reassignment de variaveis globais e troca de prefixo de cache). Nao ha preocupacao de performance aqui.

### Ponto 6 — update_meta_cache quando IDs ja estao em cache

Veredito: **e no-op para IDs ja em cache — confirmado pelo codigo do core.**

A leitura do core em `/wordpress/wp-includes/meta.php` linha 1190 confirma o comportamento:

```php
$cache_values = wp_cache_get_multiple( $object_ids, $cache_group );
foreach ( $cache_values as $id => $cached_object ) {
    if ( false === $cached_object ) {
        $non_cached_ids[] = $id;
    }
}
if ( empty( $non_cached_ids ) ) {
    return $cache;  // retorna sem nenhuma query SQL
}
```

`update_meta_cache` faz `wp_cache_get_multiple` primeiro. Apenas IDs com cache miss (`false`) vao para `$non_cached_ids`. Se todos os IDs ja estao em cache (Redis hit), `$non_cached_ids` fica vazio e a funcao retorna na linha 1201 sem emitir query SQL. E literalmente no-op para o banco de dados.

### Ponto 7 — priority 999 e conflito com outros plugins

Veredito: **sem conflito identificado, e a priority correta para este caso.**

A varredura completa dos plugins instalados encontrou apenas dois filtros em `wp_get_attachment_url` alem do proprio plugin:

1. `tunnel-url-rewrite.php` linha 150: `add_filter('wp_get_attachment_url', 'tunnel_rewrite_url')` — sem quarto argumento, portanto priority padrao **10**. Executa antes do plugin (priority 999), o que e correto: o tunnel rewrite reescreve o hostname, nao o path. O plugin recebe a URL com hostname do tunnel e verifica somente a presenca de `/sites/2/` no path, que e independente do hostname.

2. `elementor-pro/.../external-attachment-trait.php` linha 61: `add_filter('wp_get_attachment_url', ..., 10, 2)` — priority **10**, e registrado somente durante o fluxo de import/export de templates (chamado por `add_external_url_filter()`), nao durante render normal. Sem conflito.

Nao foram encontrados filtros em `get_attached_file` em nenhum plugin instalado. NML nao registra filtros em `wp_get_attachment_url` nem `get_attached_file`.

A priority 999 e a escolha correta aqui porque o plugin precisa ser o ultimo a executar: ele depende de receber a URL final (possivelmente ja com rewrite do tunnel) para verificar se ela contem `/sites/2/` e decidir se precisa agir. Executar antes de outros filtros poderia resultar em uma URL corrigida sendo sobrescrita por outro filtro. Nao ha plugin instalado com priority > 999 nessas funcoes.

## Ponto 2 — `gallery-attachment-title.php` — Completude para Metadados do Lightbox

### Como o Elementor lightbox resolve metadados

O metodo `get_lightbox_image_attributes()` em `/var/www/html/wp-content/plugins/elementor/includes/managers/image.php:131` faz:

```php
$attachment = get_post( $id );  // linha 137
```

O que o lightbox exibe como titulo e descricao depende de `$kit->get_settings('lightbox_title_src')` e `lightbox_description_src` — campos configuraveis no painel do kit. Os valores possiveis sao `alt`, `caption`, `description`, `title`.

**`gallery-attachment-title.php` cobre o filtro `the_title`** — que e invocado quando `lightbox_title_src = 'title'`. **Como funciona de fato:** O hook 4 do plugin principal pre-popula o object cache com o post do blog 1 antes do render do widget. O campo `_wp_attachment_image_alt` tambem e pre-populado via `update_meta_cache()` + `wp_cache_set()` em `bit_crossblog_warm_cache()`.

**Portanto:** Para qualquer combinacao de `lightbox_title_src`/`lightbox_description_src` (alt, caption, description, title), o mecanismo de warm_cache ja garante que os dados corretos do blog 1 estejam disponiveis. O `gallery-attachment-title.php` cobre um caminho adicional onde `get_the_title()` e chamado diretamente (fora do contexto do lightbox), como em temas ou outros plugins.

**Gap em `gallery-attachment-title.php`:** O plugin filtra apenas `the_title`. Se outro codigo chamar `get_post_meta($id, '_wp_attachment_image_alt', true)` diretamente sem passar pelo warm_cache (ex: fora do ciclo de render do widget Elementor), o alt nao estaria coberto.        static $warmed = [];
   ```php
       $missing = array_filter( $ids, function ( $id ) use ( &$warmed ) {
           if ( isset( $warmed[ $id ] ) ) return false;
           $post = wp_cache_get( $id, 'posts' );
           return ! $post || $post->post_type !== 'attachment';
       } );
       if ( empty( $missing ) ) return;
       switch_to_blog( 1 );
       update_meta_cache( 'post', $missing );
       $b1_posts = []; $b1_metas = [];
       foreach ( $missing as $id ) {
           $post = get_post( $id );
           if ( $post && $post->post_type === 'attachment' ) {
               $b1_posts[ $id ] = $post;
               $b1_metas[ $id ] = wp_cache_get( $id, 'post_meta' );
           }
           $warmed[ $id ] = true;
       }
       restore_current_blog();
       foreach ( $b1_posts as $id => $post ) {
           wp_cache_set( $id, $post, 'posts' );
           if ( $b1_metas[ $id ] !== false ) {
               wp_cache_set( $id, $b1_metas[ $id ] ?: [], 'post_meta' );
           }
       }
   }
   ```
