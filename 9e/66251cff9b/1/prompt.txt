# Systematic Debugging

## Overview

Random fixes waste time and create new bugs. Quick patches mask underlying issues.

**Core principle:** ALWAYS find root cause before attempting fixes. Symptom fixes are failure.

**Violating the letter of this process is violating the spirit of debugging.**

## The Iron Law

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

If you haven't completed Phase 1, you cannot propose fixes.

## When to Use

Use for ANY technical issue:
- Test failures
- Bugs in production
- Unexpected behavior
- Performance problems
- Build failures
- Integration issues

**Use this ESPECIALLY when:**
- Under time pressure (emergencies make guessing tempting)
- "Just one quick fix" seems obvious
- You've already tried multiple fixes
- Previous fix didn't work
- You don't fully understand the issue

**Don't skip when:**
- Issue seems simple (simple bugs have root causes too)
- You're in a hurry (rushing guarantees rework)
- Manager wants it fixed NOW (systematic is faster than thrashing)

## The Four Phases

You MUST complete each phase before proceeding to the next.

### Phase 1: Root Cause Investigation

**BEFORE attempting ANY fix:**

1. **Read Error Messages Carefully**
   - Don't skip past errors or warnings
   - They often contain the exact solution
   - Read stack traces completely
   - Note line numbers, file paths, error codes

2. **Reproduce Consistently**
   - Can you trigger it reliably?
   - What are the exact steps?
   - Does it happen every time?
   - If not reproducible → gather more data, don't guess

3. **Check Recent Changes**
   - What changed that could cause this?
   - Git diff, recent commits
   - New dependencies, config changes
   - Environmental differences

4. **Gather Evidence in Multi-Component Systems**

   **WHEN system has multiple components (CI → build → signing, API → service → database):**

   **BEFORE proposing fixes, add diagnostic instrumentation:**
   ```
   For EACH component boundary:
     - Log what data enters component
     - Log what data exits component
     - Verify environment/config propagation
     - Check state at each layer

   Run once to gather evidence showing WHERE it breaks
   THEN analyze evidence to identify failing component
   THEN investigate that specific component
   ```

   **Example (multi-layer system):**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **This reveals:** Which layer fails (secrets → workflow ✓, workflow → build ✗)

5. **Trace Data Flow**

   **WHEN error is deep in call stack:**

   See `root-cause-tracing.md` in this directory for the complete backward tracing technique.

   **Quick version:**
   - Where does bad value originate?
   - What called this with bad value?
   - Keep tracing up until you find the source
   - Fix at source, not at symptom

### Phase 2: Pattern Analysis

**Find the pattern before fixing:**

1. **Find Working Examples**
   - Locate similar working code in same codebase
   - What works that's similar to what's broken?

2. **Compare Against References**
   - If implementing pattern, read reference implementation COMPLETELY
   - Don't skim - read every line
   - Understand the pattern fully before applying

3. **Identify Differences**
   - What's different between working and broken?
   - List every difference, however small
   - Don't assume "that can't matter"

4. **Understand Dependencies**
   - What other components does this need?
   - What settings, config, environment?
   - What assumptions does it make?

### Phase 3: Hypothesis and Testing

**Scientific method:**

1. **Form Single Hypothesis**
   - State clearly: "I think X is the root cause because Y"
   - Write it down
   - Be specific, not vague

2. **Test Minimally**
   - Make the SMALLEST possible change to test hypothesis
   - One variable at a time
   - Don't fix multiple things at once

3. **Verify Before Continuing**
   - Did it work? Yes → Phase 4
   - Didn't work? Form NEW hypothesis
   - DON'T add more fixes on top

4. **When You Don't Know**
   - Say "I don't understand X"
   - Don't pretend to know
   - Ask for help
   - Research more

### Phase 4: Implementation

**Fix the root cause, not the symptom:**

1. **Create Failing Test Case**
   - Simplest possible reproduction
   - Automated test if possible
   - One-off test script if no framework
   - MUST have before fixing
   - Use the `superpowers:test-driven-development` skill for writing proper failing tests

2. **Implement Single Fix**
   - Address the root cause identified
   - ONE change at a time
   - No "while I'm here" improvements
   - No bundled refactoring

3. **Verify Fix**
   - Test passes now?
   - No other tests broken?
   - Issue actually resolved?

4. **If Fix Doesn't Work**
   - STOP
   - Count: How many fixes have you tried?
   - If < 3: Return to Phase 1, re-analyze with new information
   - **If ≥ 3: STOP and question the architecture (step 5 below)**
   - DON'T attempt Fix #4 without architectural discussion

5. **If 3+ Fixes Failed: Question Architecture**

   **Pattern indicating architectural problem:**
   - Each fix reveals new shared state/coupling/problem in different place
   - Fixes require "massive refactoring" to implement
   - Each fix creates new symptoms elsewhere

   **STOP and question fundamentals:**
   - Is this pattern fundamentally sound?
   - Are we "sticking with it through sheer inertia"?
   - Should we refactor architecture vs. continue fixing symptoms?

   **Discuss with your human partner before attempting more fixes**

   This is NOT a failed hypothesis - this is a wrong architecture.

## Red Flags - STOP and Follow Process

If you catch yourself thinking:
- "Quick fix for now, investigate later"
- "Just try changing X and see if it works"
- "Add multiple changes, run tests"
- "Skip the test, I'll manually verify"
- "It's probably X, let me fix that"
- "I don't fully understand but this might work"
- "Pattern says X but I'll adapt it differently"
- "Here are the main problems: [lists fixes without investigation]"
- Proposing solutions before tracing data flow
- **"One more fix attempt" (when already tried 2+)**
- **Each fix reveals new problem in different place**

**ALL of these mean: STOP. Return to Phase 1.**

**If 3+ fixes failed:** Question the architecture (see Phase 4.5)

## your human partner's Signals You're Doing It Wrong

**Watch for these redirections:**
- "Is that not happening?" - You assumed without verifying
- "Will it show us...?" - You should have added evidence gathering
- "Stop guessing" - You're proposing fixes without understanding
- "Ultrathink this" - Question fundamentals, not just symptoms
- "We're stuck?" (frustrated) - Your approach isn't working

**When you see these:** STOP. Return to Phase 1.

## Common Rationalizations

| Excuse | Reality |
|--------|---------|
| "Issue is simple, don't need process" | Simple issues have root causes too. Process is fast for simple bugs. |
| "Emergency, no time for process" | Systematic debugging is FASTER than guess-and-check thrashing. |
| "Just try this first, then investigate" | First fix sets the pattern. Do it right from the start. |
| "I'll write test after confirming fix works" | Untested fixes don't stick. Test first proves it. |
| "Multiple fixes at once saves time" | Can't isolate what worked. Causes new bugs. |
| "Reference too long, I'll adapt the pattern" | Partial understanding guarantees bugs. Read it completely. |
| "I see the problem, let me fix it" | Seeing symptoms ≠ understanding root cause. |
| "One more fix attempt" (after 2+ failures) | 3+ failures = architectural problem. Question pattern, don't fix again. |

## Quick Reference

| Phase | Key Activities | Success Criteria |
|-------|---------------|------------------|
| **1. Root Cause** | Read errors, reproduce, check changes, gather evidence | Understand WHAT and WHY |
| **2. Pattern** | Find working examples, compare | Identify differences |
| **3. Hypothesis** | Form theory, test minimally | Confirmed or new hypothesis |
| **4. Implementation** | Create test, fix, verify | Bug resolved, tests pass |

## When Process Reveals "No Root Cause"

If systematic investigation reveals issue is truly environmental, timing-dependent, or external:

1. You've completed the process
2. Document what you investigated
3. Implement appropriate handling (retry, timeout, error message)
4. Add monitoring/logging for future investigation

**But:** 95% of "no root cause" cases are incomplete investigation.

## Supporting Techniques

These techniques are part of systematic debugging and available in this directory:

- **`root-cause-tracing.md`** - Trace bugs backward through call stack to find original trigger
- **`defense-in-depth.md`** - Add validation at multiple layers after finding root cause
- **`condition-based-waiting.md`** - Replace arbitrary timeouts with condition polling

**Related skills:**
- **superpowers:test-driven-development** - For creating failing test case (Phase 4, Step 1)
- **superpowers:verification-before-completion** - Verify fix worked before claiming success

From debugging sessions:
- Systematic approach: 15-30 minutes to fix
- Random fixes approach: 2-3 hours of thrashing
- First-time fix rate: 95% vs 40%
- New bugs introduced: Near zero vs common

# Dispatching Parallel Agents

## Overview

You delegate tasks to specialized agents with isolated context. By precisely crafting their instructions and context, you ensure they stay focused and succeed at their task. They should never inherit your session's context or history — you construct exactly what they need. This also preserves your own context for coordination work.

When you have multiple unrelated failures (different test files, different subsystems, different bugs), investigating them sequentially wastes time. Each investigation is independent and can happen in parallel.

**Core principle:** Dispatch one agent per independent problem domain. Let them work concurrently.

## When to Use

```dot
digraph when_to_use {
    "Multiple failures?" [shape=diamond];
    "Are they independent?" [shape=diamond];
    "Single agent investigates all" [shape=box];
    "One agent per problem domain" [shape=box];
    "Can they work in parallel?" [shape=diamond];
    "Sequential agents" [shape=box];
    "Parallel dispatch" [shape=box];

    "Multiple failures?" -> "Are they independent?" [label="yes"];
    "Are they independent?" -> "Single agent investigates all" [label="no - related"];
    "Are they independent?" -> "Can they work in parallel?" [label="yes"];
    "Can they work in parallel?" -> "Parallel dispatch" [label="yes"];
    "Can they work in parallel?" -> "Sequential agents" [label="no - shared state"];
}
```

**Use when:**
- 3+ test files failing with different root causes
- Multiple subsystems broken independently
- Each problem can be understood without context from others
- No shared state between investigations

**Don't use when:**
- Failures are related (fix one might fix others)
- Need to understand full system state
- Agents would interfere with each other

## The Pattern

### 1. Identify Independent Domains

Group failures by what's broken:
- File A tests: Tool approval flow
- File B tests: Batch completion behavior
- File C tests: Abort functionality

Each domain is independent - fixing tool approval doesn't affect abort tests.

### 2. Create Focused Agent Tasks

Each agent gets:
- **Specific scope:** One test file or subsystem
- **Clear goal:** Make these tests pass
- **Constraints:** Don't change other code
- **Expected output:** Summary of what you found and fixed

### 3. Dispatch in Parallel

```typescript
// In Claude Code / AI environment
Task("Fix agent-tool-abort.test.ts failures")
Task("Fix batch-completion-behavior.test.ts failures")
Task("Fix tool-approval-race-conditions.test.ts failures")
// All three run concurrently
```

### 4. Review and Integrate

When agents return:
- Read each summary
- Verify fixes don't conflict
- Run full test suite
- Integrate all changes

## Agent Prompt Structure

Good agent prompts are:
1. **Focused** - One clear problem domain
2. **Self-contained** - All context needed to understand the problem
3. **Specific about output** - What should the agent return?

```markdown
Fix the 3 failing tests in src/agents/agent-tool-abort.test.ts:

1. "should abort tool with partial output capture" - expects 'interrupted at' in message
2. "should handle mixed completed and aborted tools" - fast tool aborted instead of completed
3. "should properly track pendingToolCount" - expects 3 results but gets 0

These are timing/race condition issues. Your task:

1. Read the test file and understand what each test verifies
2. Identify root cause - timing issues or actual bugs?
3. Fix by:
   - Replacing arbitrary timeouts with event-based waiting
   - Fixing bugs in abort implementation if found
   - Adjusting test expectations if testing changed behavior

Do NOT just increase timeouts - find the real issue.

Return: Summary of what you found and what you fixed.
```

## Common Mistakes

**❌ Too broad:** "Fix all the tests" - agent gets lost
**✅ Specific:** "Fix agent-tool-abort.test.ts" - focused scope

**❌ No context:** "Fix the race condition" - agent doesn't know where
**✅ Context:** Paste the error messages and test names

**❌ No constraints:** Agent might refactor everything
**✅ Constraints:** "Do NOT change production code" or "Fix tests only"

**❌ Vague output:** "Fix it" - you don't know what changed
**✅ Specific:** "Return summary of root cause and changes"

## When NOT to Use

**Related failures:** Fixing one might fix others - investigate together first
**Need full context:** Understanding requires seeing entire system
**Exploratory debugging:** You don't know what's broken yet
**Shared state:** Agents would interfere (editing same files, using same resources)

1. **Parallelization** - Multiple investigations happen simultaneously

### 2. Bloqueio de xmlrpc.php no WAF vs. nginx

**Recomendação: bloquear no WAF (CloudFront), não apenas no nginx.**

A diferença é arquitetural:

| Camada | O que acontece com a requisição |
|--------|--------------------------------|
| WAF/CloudFront | Request bloqueado **antes** de chegar ao EC2. CPU/memória do servidor não são consumidos. |
| nginx no EC2 | Request chega ao servidor, consome socket, memória de conexão, e ainda encaminha para PHP-FPM antes do rate limit ativar. |

No incidente descrito, o load average chegou a 34 exatamente porque o bloqueio está acontecendo **tarde demais** — o nginx recebe, o PHP-FPM processa, e só então rejeita. Com 200 req/min por IP × 6 IPs = 1.200 req/min chegando ao PHP-FPM, qualquer pool de workers satura.

**Sobre WPML e xmlrpc.php:** WPML a partir da versão 4.x (2019+) migrou completamente para REST API (`/wp-json/wpml/`). A dependência de xmlrpc era do Translation Management via OTGS e foi removida. Para confirmar no ambiente:

```bash
# Verificar versão do WPML instalada
std wp plugin get sitepress-multilingual-cms --field=version

# Verificar se há chamadas reais a xmlrpc em logs recentes
grep "xmlrpc.php" /var/log/nginx/access.log | grep -v "3.172.105\|64.252" | tail -20
```

Se o WPML instalado for >= 4.0 e os logs não mostrarem chamadas legítimas a xmlrpc (fora do range de ataque), **bloquear no WAF sem reservas.**

A regra nginx pode ser mantida como defesa em profundidade, mas o WAF deve ser a linha de frente.

### 3. Proteção contra DDoS distribuído — rate limit por IP falhou

O problema: 6 IPs × 299 req/5min = 1.794 req/5min passando pelo WAF. Solução em três camadas:

#### Camada A — Rate limit por URI (já coberto pela Regra 3 acima)

Agregar rate limit por endpoint sensível, não apenas por IP global. Um scanner que varre `/wp-includes/` com 70-80 req por IP ainda é bloqueado se o threshold por URI for 50/5min.

#### Camada B — Rate limit por ASN (geolocalização de rede)

O WAF v2 não tem rate limit nativo por ASN, mas permite combinar com IP sets de ranges específicos. Para o range `3.172.105.0/24` (6 IPs confirmados como atacantes), criar IP set e bloquear:

```json
{
  "Name": "Block-KnownAttackerRanges",
  "Priority": 3,
  "Statement": {
    "IPSetReferenceStatement": {
      "ARN": "arn:aws:wafv2:us-east-1:ACCOUNT:global/ipset/AttackerRanges/ID"
    }
  },
  "Action": { "Block": {} },
  "VisibilityConfig": {
    "SampledRequestsEnabled": true,
    "CloudWatchMetricsEnabled": true,
    "MetricName": "Block-KnownAttackerRanges"
  }
}
```

#### Camada C — AWS Shield Advanced (pago) ou WAF Fraud Control

Para ataques distribuídos coordenados onde cada IP fica abaixo do threshold, a solução correta de longo prazo é **AWS Shield Advanced** com proteção L7 automática, que detecta padrões de tráfego anômalos por ACL, não por IP individual. Custo: ~$3.000/mês — relevante só se o site for alvo recorrente.

**Alternativa gratuita e eficaz:** usar `Count` + CloudWatch Logs + Lambda para detecção automática de IPs com padrão anômalo (muitas URIs distintas em pouco tempo = scanner), populando o IP set dinamicamente. Isso exige desenvolvimento mas é o padrão usado por WAF gerenciados como Cloudflare Pro.

#### Camada D — Threshold de grupo por URI pattern

Complementar à Regra 3: adicionar rate limit para `/wp-admin/` com threshold ainda mais baixo (20/5min) para IPs não whitelistados, já que nenhum usuário legítimo faz 20 requisições ao wp-admin em 5 minutos de forma não-interativa.

### 4. IP sets recomendados — riscos de bloquear ranges grandes

**`3.172.105.0/24` (Amazon AWS) — Bloquear o /24, com atenção**

Este range é de instâncias EC2 na região `sa-east-1` (São Paulo), confirmável via `whois 3.172.105.68`. Bloquear `/24` implica:

- Não haverá impacto em usuários finais (nenhum usuário legítimo navega com IP de EC2)
- Risco real: outros serviços legítimos rodando em EC2 no mesmo range que chamem a API do site (ex: serviços de integração, webhooks, automações). Verificar nos logs se há outros IPs do /24 com status 200 antes de bloquear.
- Recomendação: bloquear o `/24` especificamente para o endpoint `/xmlrpc.php` (via ScopeDown), não como bloqueio global. Isso elimina o risco de quebrar integrações legítimas.

**`64.252.0.0/16` (Tencent Cloud) — Bloquear com mais cautela**

Um `/16` cobre 65.536 IPs. Tencent Cloud hospeda CDN, serviços SaaS e APIs usados por empresas brasileiras. Bloquear o range inteiro pode causar false positives em:

- Usuários de VPN com saída pela Tencent
- Serviços de terceiros que usam Tencent como backbone (alguns SaaS asiáticos)
- Bots legítimos de indexação que usam infraestrutura Tencent

Recomendação: **não bloquear o /16 globalmente**. Em vez disso:

1. Criar IP set com os `/24` ou `/22` específicos identificados nos logs de ataque
2. Usar rate limit agressivo (Regra 3) para paths de scanner — isso bloqueia o comportamento, não o range inteiro
3. Se o site não tem audiência na China/Ásia, considerar bloqueio geográfico via CloudFront Geo Restriction para `CN` (China) — isso cobre grande parte do Tencent Cloud sem precisar de IP sets manuais

- `REDACTED` — bloqueia comandos perigosos via query string
- `WordPressExploitablePaths_URIPATH` — bloqueia paths conhecidos como vetores de exploit incluindo xmlrpc em contextos de multicall

**Risco de false positive:** baixo para WordPress padrão. O risco principal é em customizações que usam query strings não convencionais. Ativar em modo `Count` por 48h antes de mudar para `Block`:

```json
{
  "Name": "AWS-AWSManagedRulesWordPressRuleSet",
  "Priority": 15,
  "Statement": {
    "ManagedRuleGroupStatement": {
      "VendorName": "AWS",
      "Name": "AWSManagedRulesWordPressRuleSet"
    }
  },
  "OverrideAction": { "Count": {} },
  "VisibilityConfig": {
    "SampledRequestsEnabled": true,
    "CloudWatchMetricsEnabled": true,
    "MetricName": "AWS-WordPress"
  }
}
```

Após 48h sem false positives, trocar `OverrideAction` para `None` (usa ação nativa das regras, que é Block).

#### `AWSManagedRulesKnownBadInputsRuleSet` — Ativar

Bloqueia payloads conhecidos de exploração (Log4Shell, path traversal, SSRF). Sem risco material de false positive para WordPress.

#### `AWSManagedRulesCommonRuleSet` — Ativar com exclusões

É o mais abrangente, mas tem false positives conhecidos em WordPress:

- Regra `SizeRestrictions_BODY` quebra uploads grandes (Elementor, Media Library)
- Regra `GenericRFI_BODY` pode bloquear shortcodes com URLs
- Regra `EC2MetaDataSSRF_BODY` pode bloquear em alguns contextos de API

Estratégia: ativar em modo `Count`, identificar quais regras específicas disparam em requests legítimos, excluir somente essas via `ExcludedRules`, depois mudar para Block:

```json
{
  "Name": "AWS-CommonRuleSet",
  "Priority": 16,
  "Statement": {
    "ManagedRuleGroupStatement": {
      "VendorName": "AWS",
      "Name": "AWSManagedRulesCommonRuleSet",
      "ExcludedRules": [
        { "Name": "SizeRestrictions_BODY" }
      ]
    }
  },
  "OverrideAction": { "Count": {} }
}
```

#### `AWSManagedRulesBotControlRuleSet` — Avaliar (tem custo adicional)

Esta é a única managed rule com custo separado (~$10/mês + $1/milhão de req). Para o scanner Tencent, seria a solução mais direta — detecta fingerprints de bots mesmo quando mudam de IP. Para o volume atual do site, o custo provavelmente não se justifica frente às regras customizadas acima.

# Hardening nginx/PHP-FPM — Análise do Incidente

## 1. xmlrpc.php — Bloquear ou rate limit agressivo?

### O comentário "WPML usa xmlrpc" está errado

WPML abandonou xmlrpc como mecanismo de sync **desde a versão 3.6 (2016)**. A comunicação entre subsites no multisite e entre instalações separadas passou a usar REST API (`/wp-json/wpml/`). Versões modernas (4.x e 5.x, que certamente são as ativas em produção) não tocam xmlrpc para nada funcional.

**Conclusão: o comentário está desatualizado há quase 10 anos.** Nenhuma funcionalidade crítica da stack atual (WPML, JetEngine, Elementor, The Events Calendar) depende de xmlrpc.

### O que fazer com xmlrpc

Há três níveis possíveis:

**Nível 1 — Bloqueio total (recomendado para este site):**
```nginx
location = /xmlrpc.php {
    deny all;
    access_log off;
    log_not_found off;
}
```
Retorna 403 imediatamente, **antes** de qualquer decisão de PHP-FPM. Zero custo de CPU. O brute force bate na parede no nginx.

**Nível 2 — Rate limit agressivo com bloqueio por método:**
```nginx
location = /xmlrpc.php {
    limit_req zone=xmlrpc burst=2 nodelay;
    limit_req_status 444;  # fecha conexão sem resposta HTTP
    # ... PHP-FPM
}
```
`444` fecha o socket sem responder — melhor que 403 para atacantes porque não confirma a existência do endpoint.

**Nível 3 — Manter aberto mas restrito** (apenas se alguma integração legada precisar): whitelist de IPs + rate limit extremamente baixo (`1r/m`).

Para o Concertação, o bloqueio total (Nível 1) é correto. Verifique em produção: `grep -i xmlrpc /var/log/nginx/access.log | grep -v ' 403 ' | head -20` — se só houver 403s ou se não houver requisições legítimas, confirme o bloqueio.

## 2. Dimensionamento PHP-FPM para t3.xlarge com stack pesada

### Por que 20-25 workers saturaram com 4 vCPUs

O t3.xlarge tem 4 vCPUs mas **8GB RAM**. O gargalo aqui não é CPU — é que cada worker PHP com WordPress pesado (Elementor + JetEngine + WPML) consome entre **150-250MB RSS**. Com 25 workers: 25 × 200MB = **5GB RAM só de PHP**. Quando o ataque chegou, os workers estavam bloqueados esperando respostas lentas de MySQL/Redis, acumulando backlog.

### Configuração recomendada

```ini
; www.conf — PHP-FPM pool para WordPress pesado em t3.xlarge
pm = dynamic

; Limite seguro: 8GB - 2GB (SO + nginx + MySQL local) - 1GB (headroom) = 5GB para PHP
; 5000MB / 200MB por worker = ~25 workers máx
; Mas com ataque, queremos deixar headroom para workers legítimos
pm.max_children = 20

; Workers pré-iniciados: reduz cold start em picos legítimos
pm.start_servers = 5

; Mínimo em idle: mantém capacidade sem desperdiçar RAM
pm.min_spare_servers = 3

; Máximo em idle: evitar crescimento excessivo após pico
pm.max_spare_servers = 8

; CRÍTICO: recicla workers para evitar memory leak acumulativo (Elementor vaza)
; ~500 requests por worker antes de recriar o processo
pm.max_requests = 500

; Timeout de execução — mata scripts pendurados (não afeta uploads)
request_terminate_timeout = 60s

; Slow log para identificar gargalos
slowlog = /var/log/php-fpm/slow.log
request_slowlog_timeout = 10s

; Limite de arquivo de log
rlimit_files = 65535
```

### Por que `pm.max_requests = 500` é essencial

WordPress com Elementor tem vazamentos de memória em plugins. Um worker que processa 10.000 requests pode estar consumindo 600-800MB. O `pm.max_requests` garante reciclagem periódica. O custo é insignificante (fork é barato em Linux moderno).

### Complemento: `pm.process_idle_timeout`

Se usar `pm = ondemand` (alternativa mais agressiva para economizar RAM em períodos de baixo tráfego):
```ini
pm = ondemand
pm.max_children = 20
pm.process_idle_timeout = 30s
pm.max_requests = 500
```
Em `ondemand`, workers são criados sob demanda e destruídos após 30s de idle. Mais conservador com RAM, mas adiciona latência no primeiro request após idle. Para site com tráfego constante, `dynamic` é melhor.

## 3. Rate limiting nginx contra ataques distribuídos

### Problema com a config atual

O `limit_req_zone=wp_login` (não visível na config dev, presumivelmente `$binary_remote_addr`) limita **por IP individual**. Com 6 IPs diferentes fazendo 200 req/IP/min, cada IP individualmente pode estar abaixo do burst, mas no total é 1.200 req/min chegando no PHP-FPM.

### Técnicas combinadas

**a) Zone por URI — limitar o endpoint independente de IP:**
```nginx
http {
    # Zone geral por IP (já existente)
    limit_req_zone $binary_remote_addr zone=per_ip:10m rate=10r/s;
    
    # Zone por URI — limita requisições ao endpoint globalmente
    # Teto absoluto de req/s para xmlrpc/login independente de quantos IPs
    limit_req_zone $uri zone=per_uri:1m rate=5r/s;
    
    # Zone combinada IP+URI — mais granular
    limit_req_zone "$binary_remote_addr$uri" zone=ip_uri:10m rate=2r/s;
    
    # Zone para wp-login.php
    limit_req_zone $binary_remote_addr zone=wp_login:10m rate=1r/s;
}
```

**b) `limit_conn` — limitar conexões simultâneas por IP:**
```nginx
http {
    limit_conn_zone $binary_remote_addr zone=addr:10m;
}

server {
    # Máximo 10 conexões simultâneas por IP
    limit_conn addr 10;
    
    location ~ \.php$ {
        # PHP: máximo 5 conexões simultâneas por IP
        limit_conn addr 5;
        limit_req zone=per_ip burst=10 nodelay;
        # ...
    }
}
```

**c) Bloquear User-Agents conhecidos de scanners:**
```nginx
map $http_user_agent $bad_agent {
    default           0;
    ~*masscan         1;
    ~*nikto           1;
    ~*zgrab           1;
    ~*sqlmap          1;
    ~*python-requests 1;
    ~*curl/           1;  # cuidado: CloudFront usa curl em health checks — verificar
    ""                1;  # sem User-Agent = scanner
}

server {
    if ($bad_agent) {
        return 444;
    }
}
```

**d) Rate limit com `delay` em vez de `nodelay` para ataques distribuídos lentos:**
```nginx
# burst=20 permite rajada, mas delay=10 faz as últimas 10 esperarem
# Atacantes com scripts simples ficam "presos" na fila sem desistir
location = /wp-login.php {
    limit_req zone=wp_login burst=20 delay=10;
    # ...
}
```

**e) Bloquear ranges CIDR conhecidos de atacantes:**
```nginx
geo $block_range {
    default         0;
    64.252.0.0/16   1;  # Tencent Cloud — bloco do scanner identificado
    3.172.0.0/14    1;  # AWS us-east range dos IPs de brute force
}

server {
    if ($block_range) {
        return 444;
    }
}
```

O bloco `3.172.105.0/24` é AWS — pode ser legítimo em outros contextos, mas se todos os requests desse range no incidente foram POST para xmlrpc.php, vale bloquear temporariamente via `geo` e remover depois.

## 4. Bloqueio de scanners de wp-includes/ e diretórios WordPress

### Problema: scanners testam arquivos PHP internos do WordPress

Requests como `GET /wp-includes/SimplePie/IRI.php` ou `GET /wp-admin/includes/admin.php` não deveriam nunca ser acessados diretamente — são incluídos por `index.php`, não executados via HTTP.

### Configuração específica para bloquear acesso direto

```nginx
# Bloquear execução direta de PHP dentro de wp-includes e wp-admin/includes
# Esses arquivos só devem ser carregados via include PHP, nunca via HTTP direto
location ~* /wp-includes/.*\.php$ {
    deny all;
    access_log off;
    log_not_found off;
}

location ~* /wp-admin/includes/.*\.php$ {
    deny all;
    access_log off;
    log_not_found off;
}

# Bloquear acesso a arquivos de tema que não deveriam ser acessados diretamente
location ~* /wp-content/themes/[^/]+/(?!style\.css|screenshot\.png).*\.php$ {
    deny all;
    access_log off;
    log_not_found off;
}
```

**Atenção com a ordem das locations:** A config atual tem `location ~ \.php$` que captura tudo. As novas regras de bloqueio devem usar `location ~*` com padrões mais específicos — e em nginx, `location ~*` **não** tem precedência automática sobre `location ~`. A solução é colocar os bloqueios **antes** da location PHP genérica no arquivo, ou usar o modificador `^~` em combinação com try_files + deny.

A forma mais robusta para garantir precedência:
```nginx
# DEVE vir ANTES de location ~ \.php$
location ~* ^/wp-includes/.*\.php$ { deny all; access_log off; log_not_found off; }
location ~* ^/wp-admin/includes/.*\.php$ { deny all; access_log off; log_not_found off; }
```

### Bloquear scanning de extensões suspeitas

```nginx
# Scanners buscam backups, configs, Git artifacts
location ~* \.(sql|bak|bk|swp|orig|old|tar|gz|zip|7z|log)$ {
    deny all;
    access_log off;
    log_not_found off;
}

# Git artifacts — crítico
location ~* /\.git {
    deny all;
    access_log off;
    log_not_found off;
}

# Composer/Node artifacts
location ~* /(composer\.(json|lock)|package\.json|\.env|\.env\.)  {
    deny all;
    access_log off;
    log_not_found off;
}
```

## 5. fail2ban com X-Forwarded-For atrás do ALB

### O problema específico do EC2 atrás de ALB

O nginx recebe `$remote_addr` como o IP do ALB (faixa privada `172.x.x.x`), não do cliente. O IP real vem em `X-Forwarded-For`. Se o fail2ban lê `remote_addr` dos logs, vai banir o IP do próprio ALB — derrubando o site.

### Solução: log format com IP real + fail2ban config ajustado

**Passo 1 — Alterar o log format no nginx para logar o IP real:**
```nginx
# Em nginx.conf, substituir o log_format main atual
log_format main_real '$http_x_forwarded_for - $remote_user [$time_local] "$request" '
                     '$status $body_bytes_sent "$http_referer" '
                     '"$http_user_agent"';
```

Ou, mais elegante, usar `set_real_ip_from` para que o nginx trate o `X-Forwarded-For` como `$remote_addr`:
```nginx
# Aceitar X-Forwarded-For apenas de IPs do ALB (subnets privadas AWS)
set_real_ip_from 10.0.0.0/8;
set_real_ip_from 172.16.0.0/12;
set_real_ip_from 192.168.0.0/16;
real_ip_header X-Forwarded-For;
real_ip_recursive on;
```

Com `real_ip_recursive on`, o nginx extrai o IP mais à esquerda do cabeçalho que não seja um proxy confiável. Isso é o IP real do cliente. A partir daí, `$remote_addr` nos logs já contém o IP correto — fail2ban funciona normalmente sem ajustes.

**Passo 2 — Filtro fail2ban para WordPress:**

```ini
# /etc/fail2ban/filter.d/wordpress-xmlrpc.conf
[Definition]
failregex = ^<HOST> .* "POST /xmlrpc\.php.*" (200|403|429)
ignoreregex =

# /etc/fail2ban/filter.d/nginx-scanner.conf
[Definition]
# Scanner de diretórios: muitos 404 em sequência rápida
failregex = ^<HOST> .* "(GET|POST) /wp-(includes|admin)/.*" 403
            ^<HOST> .* "(GET|POST) .*\.(sql|bak|env|git).*" (404|403)
ignoreregex =
```

**Passo 3 — Jail config:**
```ini
# /etc/fail2ban/jail.local
[wordpress-xmlrpc]
enabled  = true
filter   = wordpress-xmlrpc
logpath  = /var/log/nginx/access.log
maxretry = 10
findtime = 60
bantime  = 3600
action   = iptables-multiport[name=wordpress, port="80,443", protocol=tcp]

[nginx-scanner]
enabled  = true
filter   = nginx-scanner
logpath  = /var/log/nginx/access.log
maxretry = 30
findtime = 60
bantime  = 86400  ; 24h para scanners
action   = iptables-multiport[name=scanner, port="80,443", protocol=tcp]
```

### Por que iptables funciona mesmo com ALB

O ALB entrega tráfego para as instâncias EC2 na porta 80/443. Mesmo que o IP do ALB seja privado, **o `real_ip` já corrigiu `$remote_addr`** no nginx. O fail2ban lê os logs com o IP real e bane via iptables na interface de rede da instância. O ALB não é afetado porque ele está numa camada diferente — o tráfego bloqueado pelo iptables é de conexões TCP saindo do ALB com o IP original preservado no header, mas o iptables na EC2 vai bloquear **todas** as conexões originadas do ALB que contenham `X-Forwarded-For` com aquele IP específico.

**Atenção:** iptables na EC2 não consegue filtrar por `X-Forwarded-For` — iptables opera em L3/L4, não L7. A solução correta é uma de:

1. **Opção A — fail2ban + nginx `deny` dinâmico:** Em vez de iptables, o fail2ban escreve em `/etc/nginx/conf.d/blacklist.conf` (lista de `deny IP;`) e faz `nginx -s reload`. Funciona em L7 onde o IP real já está disponível.

2. **Opção B — AWS WAF no ALB (recomendado para produção):** Rate-based rules no WAF filtram por IP real antes de chegar à EC2. Custo: ~$5/mês base + por request. Elimina a necessidade de fail2ban.

3. **Opção C — fail2ban + AWS CLI para Security Groups:** O fail2ban executa `aws ec2 revoke-security-group-ingress` para bloquear o IP no nível do Security Group. Complexo e com race conditions.

Para este ambiente, a **Opção A** (fail2ban + nginx blacklist dinâmica) é a mais simples de implementar sem custos adicionais.

## Análise de Segurança WordPress — Concertação Amazônica

### 1. xmlrpc.php no WPML — é seguro bloquear?

**Resposta direta: bloquear totalmente é viável, mas exige uma decisão consciente sobre o pickup method do WPML Translation Proxy.**

Lendo o código fonte da versão instalada (WPML 4.8.6):

`sitepress.php` linha 370-373 confirma que `WPML_XMLRPC` só é instanciado quando `defined('XMLRPC_REQUEST') && XMLRPC_REQUEST` — ou seja, **o WPML em si só usa xmlrpc se uma request chega em xmlrpc.php**. O core do site (multilíngue, language switcher, WPML Circle, filtros JetEngine, `bit-wpml-circle.php`) não depende de xmlrpc para funcionar.

O que **realmente usa xmlrpc no stack atual**:

| Plugin | Uso | Impacto do bloqueio |

| **WPML 4.8.6** | `WPML_XMLRPC::xmlrpc_call()` — trata posts criados via xmlrpc para definir `_wpml_language` e `_wpml_trid` | Nenhum impacto no frontend. Impacto apenas se alguém criar posts via MetaWeblog API/xmlrpc remotamente |

| **WPML Translation Feedback** | `WPML_TF_XML_RPC_Hooks` — callback `translationproxy.update_feedback_status` | Só relevante se estiver usando WPML Translation Management com serviço externo de tradução profissional |
| **Network Media Library** | Trata `metaWeblog.newMediaObject`, `wp.getMediaItem`, `wp.getMediaLibrary` para redirecionar ao blog de mídia centralizado | Só impacta se uploads forem feitos via xmlrpc client externo (Desk, MarsEdit, etc.) |

**Ação urgente antes de bloquear:** verificar o pickup method no banco:

```bash
std wp eval 'echo get_option("icl_sitepress_settings")["translation_pickup_method"];'
# 0 = XMLRPC (quebra o Translation Proxy se bloqueado)
# 1 = POLLING (seguro bloquear)
```

Se retornar `1` (polling) ou se não estiver usando WPML Translation Management com serviço externo pago — **bloquear é seguro**. A imensa maioria dos sites que usa WPML sem contrato com tradutores profissionais externos não usa Translation Proxy.

### 2. mu-plugin de bloqueio xmlrpc

Bloquear via mu-plugin é a camada mais rápida dentro do WordPress (carrega antes dos plugins). O hook `xmlrpc_enabled` retorna `false` e causa 403 antes de qualquer autenticação:

```php
<?php
/**
 * Plugin Name: BIT Disable XMLRPC
 * Description: Bloqueia xmlrpc.php completamente. WPML 4.8.6 usa este endpoint apenas
 *              para Translation Proxy pickup (se configurado como XMLRPC) e para posts
 *              criados via MetaWeblog API. Nenhum desses usos é ativo neste site.
 * Version:     1.0.0
 * Author:      Bureau de Tecnologia
 * Network:     true
 */

if ( ! defined( 'ABSPATH' ) ) {
    exit;
}

/**
 * Retorna false para desabilitar completamente o endpoint xmlrpc.php.
 * O WordPress emite HTTP 403 antes de processar qualquer método.
 * Isso não afeta REST API, admin-ajax.php, nem o frontend multilíngue do WPML.
 */
add_filter( 'xmlrpc_enabled', '__return_false' );

/**
 * Remove todos os métodos xmlrpc da lista — defesa em profundidade caso
 * xmlrpc_enabled seja contornado por algum plugin legacy.
 */
add_filter( 'xmlrpc_methods', function() {
    return [];
} );
```

**Importante:** `xmlrpc_enabled` retornando `false` emite `HTTP 403 Forbidden` com body XML de erro — não um 404. Isso ainda consome um worker PHP-FPM por request. Para zerar o custo de CPU, bloquear no nginx (retornando 403 puro) é superior. O mu-plugin é a camada de segurança WordPress, não o substituto do nginx. A combinação ideal é os dois.

Além de xmlrpc, o modo subdirectory multisite expõe endpoints adicionais que não existem em sites single-site:

**`/wp-activate.php`** — ativação de contas criadas via `/wp-signup.php`. Se `registration` for `none`, este endpoint fica inerte mas continua respondendo — melhor bloquear no nginx também.

**`/wp-json/wp/v2/users`** — a REST API expõe lista de usuários por padrão, incluindo usernames. Em multisite, há um endpoint por blog. Com `--url=`, WP-CLI confirma:

### 4. Proteção de wp-login.php na camada WordPress

O nginx atual (linha 209) tem um `location` para `wp-login.php` mas **não tem `limit_req`** — não há rate limiting implementado em nenhuma camada visível nos arquivos do repositório, e não há plugin de limitação de login instalado (verificado — nenhum Wordfence, Cerber, Limit Login Attempts nos plugins).

**Opção A — mu-plugin com rate limit via transients (zero dependência):**

```php
<?php
/**
 * Plugin Name: BIT Login Rate Limit
 * Description: Rate limit em wp-login.php usando transients no Redis.
 *              Bloqueia IP após 5 falhas em 5 minutos. Compatível com WP Rocket + Redis.
 * Version:     1.0.0
 * Author:      Bureau de Tecnologia
 * Network:     true
 */

if ( ! defined( 'ABSPATH' ) ) {
    exit;
}

// Só age no formulário de login, não na página GET
add_action( 'wp_login_failed', function( string $username ) {

    // Em ambiente com Cloudflare Tunnel, usar CF-Connecting-IP
    $ip = $_SERVER['HTTP_CF_CONNECTING_IP']
        ?? $_SERVER['HTTP_X_FORWARDED_FOR']
        ?? $_SERVER['REMOTE_ADDR']
        ?? '0.0.0.0';

    // Normalizar: pegar primeiro IP da lista se X-Forwarded-For tiver múltiplos
    $ip = trim( explode( ',', $ip )[0] );

    $key      = 'bit_login_fail_' . md5( $ip );
    $attempts = (int) get_transient( $key );
    $attempts++;
    set_transient( $key, $attempts, 5 * MINUTE_IN_SECONDS );

    if ( $attempts >= 5 ) {
        wp_die(
            '<p>Muitas tentativas de login. Tente novamente em alguns minutos.</p>',
            'Acesso bloqueado',
            [ 'response' => 429 ]
        );
    }
} );

// Verificar antes do formulário ser processado
add_action( 'login_init', function() {

    if ( 'POST' !== $_SERVER['REQUEST_METHOD'] ) {
        return;
    }

    $ip = $_SERVER['HTTP_CF_CONNECTING_IP']
        ?? $_SERVER['HTTP_X_FORWARDED_FOR']
        ?? $_SERVER['REMOTE_ADDR']
        ?? '0.0.0.0';

    $ip       = trim( explode( ',', $ip )[0] );
    $key      = 'bit_login_fail_' . md5( $ip );
    $attempts = (int) get_transient( $key );

    if ( $attempts >= 5 ) {
        wp_die(
            '<p>Muitas tentativas de login. Tente novamente em alguns minutos.</p>',
            'Acesso bloqueado',
            [ 'response' => 429 ]
        );
    }
} );
```

Este mu-plugin funciona com o Redis Object Cache do site — os transients são armazenados no Redis, não no MySQL, então não adiciona queries ao banco durante brute force.

**Cuidado com o IP:** o nginx passa `HTTP_CF_CONNECTING_IP` via `fastcgi_param` (linha confirmada no `wordpress.conf`), então o PHP recebe o IP real do visitante mesmo atrás do Cloudflare Tunnel — o código acima trata corretamente essa cadeia.

### 6. Plugins de segurança — vale instalar no stack atual?

**Recomendação: não instalar Wordfence ou Sucuri neste stack. Instalar Cerber Security.**

Análise por plugin:

**Wordfence — não recomendado aqui:**
- O firewall WAF do Wordfence roda em PHP (before WordPress bootstrap), o que é menos eficiente que nginx/CloudFront WAF
- O scanner de malware lê todos os arquivos PHP do disco em cada scan — com Elementor (centenas de arquivos) + JetEngine + WPML, um scan pode consumir 2-4 GB de I/O e elevar o load em 8-12 pontos durante ~5 minutos
- O "live traffic" do Wordfence usa sua própria tabela de log (`wfTraffic`) que cresce muito rápido em sites com tráfego — requer manutenção
- Em multisite, o Wordfence cobra por subsite na versão paga
- O maior benefício do Wordfence (WAF + bloqueio de IPs) já está disponível no CloudFront WAF que existe na infraestrutura de produção do site

**Sucuri — não recomendado:**
- O Sucuri SiteCheck é um scanner externo que só vê HTML renderizado, não o código PHP — limitado para detecção de backdoors
- O Sucuri Firewall é um proxy reverso que precisa mudar os DNS — conflita com o CloudFront já configurado
- O plugin de monitoramento do Sucuri usa polling constante que adiciona load desnecessário

**Cerber Security — adequado para este stack:**
- Rate limiting de wp-login.php e xmlrpc nativo, mais sofisticado que o mu-plugin acima (blacklist por IP + subnet)
- Proteção de REST API por IP e por endpoint
- Compatível com WP Rocket (não interfere no cache)
- Leve: não tem scanner de disco por padrão
- Suporta multisite com configuração por rede
- Não usa tabelas de log em disco por default

**Alternativa sem plugin adicional — que é o caminho mais limpo:**
O stack já tem CloudFront (distribuição `E2F1QD7E7YOYEB`) na produção. A solução mais eficiente é criar uma regra no **CloudFront WAF** para:
1. Bloquear `3.172.105.0/24` e `64.252.0.0/16` por IP
2. Rate limit em `/xmlrpc.php` e `/wp-login.php` a nível de CDN — antes de chegar no EC2

Isso elimina o problema na origem sem adicionar carga ao PHP-FPM, que é o recurso que saturou (load 34, 19.2% 5XX).

# Análise de Hardening: nginx + PHP-FPM — concertacaoamazonia.com.br

**Incidente:** 2026-03-31 | EC2 t3.xlarge (4 vCPUs, 8 GB RAM) | Load average 34 durante ataque combinado de brute force em xmlrpc.php + directory scanning Tencent

## Diagnóstico da Situação Atual

### O que o incidente revelou

O script `d1-php-fpm-tuning.sh` calcula `max_children` como `(8192 * 0.80) / 60 = 109 workers`. Se o site estava com 25 workers ativos no pico **e ainda assim entrou em colapso**, dois cenários são possíveis:

1. O tuning ainda não havia sido aplicado no servidor (pool estava com valores padrão menores)
2. Mais provável: 25 workers simultâneos em uma instância que deveria suportar 109 indica que **cada worker estava bloqueado aguardando I/O** (lock no MySQL/Redis durante os requests xmlrpc), não que havia falta de workers. Load average 34 com 3% CPU por processo confirma: gargalo era I/O wait, não CPU.

O problema real foi que **requests chegavam ao PHP-FPM antes de qualquer defesa nginx**, pois a configuração atual não tem `limit_conn` nem zona de rate limit dedicada para xmlrpc (usa a mesma zona `wp_login` que o login, com burst muito generoso de 5).

## 1. xmlrpc.php — Bloqueio vs Rate Limit

### WPML moderno não usa xmlrpc

O comentário no código está desatualizado. WPML desde a versão 4.x usa exclusivamente a REST API (`/wp-json/wpml/`). Para confirmar definitivamente no servidor:

```bash
# Verificar se WPML usa xmlrpc — procurar chamadas ao endpoint
grep -r "xmlrpc" /var/www/concertacaoamazonia.com.br/wp-content/plugins/wpml-multilingual-cms/ \
  --include="*.php" -l 2>/dev/null | head -10

# No dev local
docker exec concertacao-dev-wordpress bash -c \
  "grep -r 'xmlrpc' /var/www/html/wp-content/plugins/wpml-multilingual-cms/ --include='*.php' -l 2>/dev/null | head -10"
```

**Recomendação: bloquear completamente com `return 444`.**

`return 444` é superior a `deny all` (retorna 403) ou `return 403` por três razões:
- Fecha a conexão TCP sem enviar resposta HTTP → economiza bandwidth
- Não informa ao atacante que o endpoint existe
- Reduz tempo do worker nginx (não aguarda envio de body de resposta)

```nginx
# xmlrpc.php: bloqueio total — WPML usa REST API desde v4.x, não xmlrpc
# return 444: fecha TCP sem resposta (não vaza informação, economiza recursos)
location = /xmlrpc.php {
    access_log /var/log/nginx/blocked-xmlrpc.log combined;
    return 444;
}
```

**Se existir qualquer integração legada que precise de xmlrpc** (verificar com o diagnóstico acima), a alternativa é whitelist por IP:

```nginx
location = /xmlrpc.php {
    # Permitir apenas IPs conhecidos do WPML/Yoast (substituir pelos IPs reais)
    allow 203.0.113.10;  # IP do servidor de sync WPML se aplicável
    deny all;
    access_log /var/log/nginx/blocked-xmlrpc.log combined;
    try_files $uri =404;
    include /etc/nginx/snippets/php-fpm.conf;
    fastcgi_param HTTPS on;
    fastcgi_read_timeout 30;
    fastcgi_send_timeout 30;
}
```

## 3. Nginx Hardening — Rate Limiting e Conexões

### Problema na configuração atual

A zona `wp_login` tem `rate=5r/m` com `burst=5 nodelay` para xmlrpc. Isso significa que **até 5 requests chegam imediatamente ao PHP-FPM** antes de qualquer throttling. Com 200 req/min por IP e múltiplos IPs do /24, a fila de burst se esgota rapidamente mas já causou dano.

### Configuração recomendada para `nginx.conf` (bloco http)

```nginx
# --- Rate Limiting: múltiplas zonas por endpoint ---

# Zone para wp-login: anti-brute-force agressivo
limit_req_zone $binary_remote_addr zone=wp_login:10m rate=5r/m;

# Zone dedicada para xmlrpc: mais restritiva (ou remover se bloquear)
limit_req_zone $binary_remote_addr zone=xmlrpc_strict:5m rate=1r/m;

# Zone para REST API: permite uso normal mas limita abuso
limit_req_zone $binary_remote_addr zone=wp_rest:20m rate=30r/m;

# Zone para PHP em geral: proteção contra flood genérico
limit_req_zone $binary_remote_addr zone=php_general:20m rate=60r/m;

# --- Limit Connections: máximo simultâneo por IP ---
# Previne que um único IP abra centenas de conexões simultâneas
limit_conn_zone $binary_remote_addr zone=per_ip:10m;
limit_conn_zone $server_name zone=per_server:10m;

# --- Logging de requests bloqueados pelo rate limit ---
# $limit_req_status: "PASSED", "DELAYED", "REJECTED", "BYPASSED"
log_format security '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'rt=$request_time '
                    'limit_req=$limit_req_status '
                    'limit_conn=$limit_conn_status '
                    'xff="$http_x_forwarded_for"';
```

### Configuração recomendada para o vhost de produção

Estas alterações devem ser aplicadas no `03-nginx-sites.sh`, no bloco do site gerado:

```nginx

limit_conn per_ip 20;

# wp-login.php: rate limit agressivo anti-brute-force
```

## 4. Proteção Contra Directory Scanning

### Por que requests chegavam ao PHP-FPM

Analisando o fluxo atual em `wordpress.conf` (docker-dev, modelo do prod):

1. `location ~ /\.` → nega dotfiles. OK.
2. `location ~* \.(js|css|...)$` → serve estáticos, sem PHP.
3. **`location /`** → `try_files $uri $uri/ /index.php?$args` — **este é o problema**.

Quando o scanner Tencent acessa `/wp-includes/inexistente/`, nginx não encontra o arquivo ou diretório, então cai no `index.php?$args` — chegando ao PHP-FPM. O `try_files` com fallback para PHP é necessário para WordPress permalinks, mas permite que qualquer 404 acione PHP.

### Solução: bloquear diretórios sensíveis antes do fallback PHP

```nginx
# Bloquear acesso direto a diretórios de core WordPress
# Scanner não deve acionar PHP-FPM ao varrer wp-includes/
location ~* ^/(wp-includes|wp-admin/includes)/ {
    # Arquivos PHP nestes diretórios: apenas wp-includes/ms-files.php é legítimo
    # (já coberto pelo rewrite multisite acima)
    location ~ \.php$ {
        # Apenas ms-files.php é permitido (multisite uploads)
        if ($fastcgi_script_name !~ "^/wp-includes/ms-files\.php") {
            return 444;
        }
        try_files $uri =404;
        include /etc/nginx/snippets/php-fpm.conf;
        fastcgi_param HTTPS on;
        fastcgi_read_timeout 30;
    }
    # Arquivos estáticos: servir normalmente
    try_files $uri =404;
}

# Bloquear scanners de plugins/themes (não devem acessar PHP de plugins diretamente)
location ~* ^/wp-content/(?:plugins|themes)/[^/]+/(?:.*\.php)$ {
    # Apenas arquivos explicitamente chamados pelo WordPress são legítimos
    # Scanners tentam acessar admin.php, config.php, etc. diretamente
    try_files $uri =404;
    # Nota: não passar para PHP-FPM — acesso direto a PHP de plugins é incomum
    # e quando legítimo é via WordPress core, não URL direta
    return 444;
}
```

**Observação importante:** O `try_files $uri =404` no `location ~ \.php$` já previne que PHP inexistente vá ao FPM — o `=404` retorna 404 sem consultar o FPM. O problema é que `location /` com fallback para `index.php` aciona o FPM mesmo quando o arquivo não existe. A solução é adicionar uma location específica para wp-includes antes da location geral.

### Adição ao log para identificar scanning

```nginx
# Log separado para monitorar acessos a wp-includes (scanner detection)
location ~* ^/wp-includes/ {
    access_log /var/log/nginx/wp-includes-access.log security;
    try_files $uri =404;
    # PHP dentro de wp-includes só via rewrite do WordPress core
    location ~ \.php$ {
        return 444;
    }
}
```

## 5. Detecção de Ataque em Tempo Real

### Log format com `$limit_req_status`

A variável `$limit_req_status` está disponível desde nginx 1.17.6 e retorna:

```nginx
# Log dedicado para requests bloqueados
# Configurar por location nos endpoints sensíveis:
# access_log /var/log/nginx/security.log security if=$limit_req_status;
```

### Diagnóstico central do incidente

O ataque de brute-force em `/xmlrpc.php` chegou diretamente ao PHP-FPM porque **não há nenhum bloqueio antes do backend**. O health check `/healthz` respondia normalmente (é nginx puro, sem tocar o FPM), então o ALB nunca parou de enviar tráfego.

---

### Roadmap priorizado

**P1 — Hoje (impacto máximo, complexidade mínima)**

1. **Bloquear xmlrpc.php no nginx** — `location = /xmlrpc.php { deny all; return 403; }` no `wordpress.conf`. Nginx reload, zero downtime. Elimina 100% do vetor do ataque ocorrido.

2. **PHP-FPM tuning em produção:**
   - `pm.max_children = 50` (era 25 — com 8GB RAM + Aurora externo, comporta ~68 workers)
   - `request_terminate_timeout = 30s` (era 300s — esse é o principal vilão)
   - `request_slowlog_timeout = 3s`

**P1 — Esta semana**

3. **Página de manutenção como fallback nginx** — configurar `@php_down` com `error_page 502 503 504` servindo o `manutencao.html` já existente. Reduzir `fastcgi_connect_timeout` de 60s para 5s — com FPM saturado, nginx para de acumular fila em 5s em vez de 60s.

**P2 — Próximas 2 semanas**

5. **Custom error response no CloudFront** — configurar 502/504 para retornar 503 com página estática (ErrorCachingMinTTL: 10s) em vez de timeout ao usuário final.

6. **Health check real no ALB** — criar `health.php` que verifica PHP-FPM + MySQL + Redis. Unhealthy threshold = 2 checks (30s) em vez de 3 (90s).

7. **CloudWatch alarms** — CPU > 75% por 3min → SNS alerta; CPU > 90% → SNS crítico. 5XX > 5% no ALB → alerta. Custo: zero (free tier).

### 2. WordPress Application-Level Hardening

**a) wp-login.php em Multisite** — usar `limit_req` do nginx (não HTTP Basic Auth, que quebra o fluxo de login do Multisite):

```nginx
# Em http{}: limit_req_zone $binary_remote_addr zone=wplogin:10m rate=5r/m;
# Em server{}: limit_req zone=wplogin burst=3 nodelay;
```

**b) REST API** — NÃO desabilitar globalmente. `disable_wp_rest_api` quebraria WPML, JetEngine listings e o Elementor Editor. Restringir apenas `/wp/v2/users` para não autenticados (implementado no mu-plugin).

**c) User enumeration** — dois vetores bloqueados no mu-plugin:
- `/wp-json/wp/v2/users` — restringe para usuários com `list_users` capability
- `/?author=N` — redireciona para home (302), páginas de arquivo de autor retornam 404

**d) Arquivos sensíveis** — o nginx atual já cobre `wp-config.php`, `.env` (via `location ~ /\.`), `readme.html` e `license.txt`. Faltam `debug.log` e `error_log`:

```nginx
location ~* /(debug\.log|error_log|\.htaccess|wp-cli\.phar)$ {

}
```

### 3. Plugin de Segurança — Recomendação

**Não instalar Wordfence nem Solid Security.** Conflitos documentados:

- **Wordfence:** block-level scanning gera falsos positivos em queries dinâmicas do JetEngine; modo "live traffic" cria lock contention com Redis Object Cache
- **Solid Security:** módulo "WordPress Tweaks" pode desabilitar XML-RPC sem verificar dependências; lockout conflita com o login flow do WPML Language Switcher

**Recomendação: hardening manual em 3 camadas:**
1. **CloudFront WAF** — rate limiting na borda (bloqueia antes do EC2)
2. **nginx** — bloqueio por localização + rate limiting
3. **mu-plugin `bit-security-hardening.php`** — application-level

Zero overhead de plugin em cada request, zero conflitos com JetEngine/WPML/Redis.

## A — Implicações do bloqueio de xmlrpc.php

**A.1 — `return 444` gera 5XX no CloudFront** `MÉDIO`

O código 444 fecha a conexão TCP sem resposta HTTP. Para o CloudFront isso é uma falha de origin — ele faz 3 retentativas antes de retornar 502 ao cliente. Durante ataque volumétrico, o EC2 recebe 3× o volume de conexões TCP para esse path. O WAF (Block-XMLRPC) descarta antes de chegar ao origin — então o problema só ocorre em tráfego que bypassa o CloudFront.

Recomendação: substituir `return 444` por `return 403`. O 444 faz sentido para economizar banda em conexões de scanner leve, mas amplifica carga em ataques volumétricos.

**A.2 — `blocked-xmlrpc.log` sem logrotate** `BAIXO`

Pode crescer vários GB em poucas horas sob ataque volumétrico. Se `/var/log` encher, PHP-FPM começa a falhar silenciosamente. Configurar logrotate com `daily / rotate 7 / compress`.

**A.3 — Regra WAF `ENDS_WITH` e Multisite subdirectory** `INFO`

Não há gap real, mas o comportamento é contra-intuitivo: `/cultura/xmlrpc.php` é capturado pelo rewrite nginx (`^/[_0-9a-zA-Z-]+/(wp-[^/]+\.php)$` com flag `last`) e reescrito para `/xmlrpc.php`, que cai no bloco de bloqueio. A regra WAF `ENDS_WITH` captura ambos os paths antes do rewrite. Documentar no `wordpress.conf` para não remover o rewrite acidentalmente.

**A.4 — Acesso direto ao EC2 bypassa nginx** `MÉDIO`

O mu-plugin (`xmlrpc_enabled → false`) cobre este caminho — o PHP retorna XML de erro. O gap real é: o Security Group do EC2/ALB permite tráfego direto de `0.0.0.0/0`? Se sim, toda a cadeia WAF é bypassável.

## B — Implicações do IPSet de IPs atacantes

**B.1 — Bloqueio de `/24` de nós CloudFront origin-facing** `CRÍTICO`

Os ranges `3.172.105.0/24` e `64.252.66-68.0/24` são IPs de origin-facing nodes da AWS. Bloquear um `/24` inteiro é seguro para este site (ALB exclusivo), mas os IPs rotativos da AWS podem ser desalocados e realocados para tráfego legítimo de outras distribuições. O risco aumenta com o tempo — o IPSet precisa de TTL de 90 dias.

Mais importante: esta proteção é contornável em 5 minutos — o atacante reconfigura a distribuição CloudFront para usar outros nós de origin. Bloquear IPs específicos não resolve o problema estrutural.

**B.2 — WAF no CloudFront não protege requests diretas ao ALB** `ALTO`

Se o Security Group do ALB aceita `0.0.0.0/0` nas portas 80/443, todo o WAF é bypassável. O atacante do incidente claramente conhece a arquitetura CloudFront + ALB — este vetor será tentado. Restringir o SG para aceitar apenas o bloco de IPs do CloudFront (publicado em `ip-ranges.amazonaws.com`).

**B.3 — IPSet sem TTL definido** `MÉDIO`

Ranges de IPs cloud rotativos acumulados sem revisão podem bloquear tráfego legítimo silenciosamente após meses. Documentar TTL de 90 dias e criar revisão recorrente.

## C — Implicações do mu-plugin

**C.1 — `is_author()` → 404 para todos os perfis de autor** `ALTO`

A política é demasiado ampla. Se o site tem perfis de autores públicos indexados no Google (pesquisadores, organizações parceiras), o Google vai receber 404 nas próximas crawls e desindexar esses perfis. Verificar se há perfis públicos intencionais — se sim, adicionar whitelist de roles ou IDs.

**C.2 — Remoção de `?ver=` é cirúrgica e não afeta WP Rocket** `INFO`

O filtro só remove `ver=` quando o valor é exatamente a versão do WordPress core — não remove hashes de plugins. O WP Rocket usa cache busting próprio (hash no path ou concatenação). Não há impacto.

**C.3 — `header_remove('Server')` no PHP é ineficaz em nginx+FPM** `BAIXO`

O header `Server:` é adicionado pelo nginx após receber a resposta do PHP-FPM — o PHP nunca o controla. A instrução é inócua. O `server_tokens off` no nginx já remove a versão. Para remover o header `Server` completamente precisaria do módulo `headers_more` no nginx.

**C.4 — `ini_set('expose_php', '0')` não funciona em PHP-FPM** `MÉDIO`

`expose_php` é `PHP_INI_SYSTEM` — não pode ser alterado em runtime. A linha é silenciosamente ignorada. O `header_remove('X-Powered-By')` logo acima no mesmo plugin já resolve o problema em runtime. Remover a linha `ini_set` e adicionar `expose_php = Off` ao `php.ini` do pool FPM.

**C.5 — Log de falhas de login pode causar flooding + log injection** `ALTO`

Dois problemas:
1. Em brute force de 1.000 req/s, o `error_log()` escreve ~150MB/10min no error_log do PHP-FPM sem limite de tamanho.
2. O `$username` passado ao `sprintf` não recebe `sanitize_text_field()` — um atacante pode injetar newlines no log enviando username com `\n`. Os campos `$ip` e `$ua` têm sanitização, mas `$username` não.

## D — Gaps não cobertos

**D.1 — wp-login.php sem proteção WAF** `ALTO` — Gap mais urgente

O incidente foi CDN fronting attack via xmlrpc.php. O próximo passo óbvio é repetir o mesmo ataque em wp-login.php. O rate limit nginx de 5r/m por IP é ineficaz contra ataques distribuídos via CDN fronting (cada nó CloudFront tem IP diferente). Não há regra WAF para wp-login.php. Criar regra WAF de rate limiting: POST em `/wp-login.php`, 50 req/5min por IP.

**D.2 — admin-ajax.php exposto sem rate limiting** `MÉDIO`

JetEngine, Elementor Pro e TEC registram ações `wp_ajax_nopriv_*` públicas. Sem rate limiting, qualquer bot pode chamar essas actions indefinidamente (scraping de listagens, queries pesadas de JetEngine). Adicionar rate limiting WAF para POST em `/wp-admin/admin-ajax.php`.

**D.3 — REST API além de `/wp/v2/users`** `MÉDIO`

O endpoint `/wp-json/` expõe a estrutura completa da API (discovery), facilitando reconhecimento de plugins. Endpoints de JetEngine, WPML e Elementor Pro ficam visíveis. Verificar exposição de rascunhos via `/wp/v2/posts?status=draft`. Considerar restringir o discovery endpoint para não autenticados.

**D.4 — CDN fronting attack: proteção atual não resiste a novos IPs** `ALTO` — Gap estrutural

O IPSet bloqueia os ranges específicos do incidente. O atacante troca para outros nós CloudFront em minutos. A defesa estrutural correta é o **custom header secreto**: o CloudFront injeta `X-Origin-Verify: UUID-secreto` em todas as requests, e o nginx rejeita com 403 qualquer request sem esse header. Esta única mudança elimina CDN fronting attacks estruturalmente, independente dos IPs usados.

### 1.1 Log `/var/log/nginx/blocked-xmlrpc.log`

**Severidade: BAIXA → MÉDIA se ataque persistir**

**Cobertura pelo logrotate:**
O padrão `/etc/logrotate.d/nginx` em Ubuntu/Debian cobre `    /var/log/nginx/*.log`, portanto o novo arquivo **está coberto automaticamente** — rotação diária com 14 dias de retenção (configuração padrão). Nenhuma ação necessária, mas vale confirmar:

### 1.3 Regra WAF P8 `Block-XMLRPC` — `ENDS_WITH /xmlrpc.php`

## 2. Gaps de Resiliência Não Endereçados

### 2.1 Health Check Superficial do ALB

**Severidade: ALTA**

Este é o gap que permitiu o ataque causar 30+ minutos de degradação com o ALB **continuando a enviar tráfego ao nó saturado**. O health check na porta 8080 (nginx puro) verifica apenas que o processo nginx está vivo — não verifica PHP-FPM, não verifica latência, não verifica fila de backlog.

**Impacto real observado no ataque:**

```
PHP-FPM: todos os 55 workers ocupados
```

```php
// Retornar 200 apenas se tudo OK, 503 caso contrário
```

### 2.2 Ausência de Circuit Breaker no nginx → PHP-FPM

**Severidade: ALTA**

**Comportamento atual:**
```
fastcgi_connect_timeout 60s
fastcgi_read_timeout 60s
→ nginx segura conexão por até 60s esperando FPM
→ Com 25 req simultâneas × 60s = acumulação de 1500 conexões
→ Cada conexão consome ~8KB de buffer nginx = ~12MB só em buffers
→ worker_connections esgota → nginx começa a retornar 502 para TODAS as requests, incluindo estáticas
```

**Configuração de fail-fast recomendada:**

```nginx
# Em /etc/nginx/conf.d/wordpress.conf
fastcgi_connect_timeout 5s;      # Era 60s — se FPM não aceita em 5s, está saturado
fastcgi_read_timeout 30s;        # Reduzir para páginas normais
fastcgi_send_timeout 30s;

# Circuit breaker via upstream (se múltiplos workers, mas mesmo com um serve):
upstream php_fpm {
    server unix:/run/php/php8.2-fpm.sock;
    keepalive 8;
}

# Retorno rápido quando FPM não responde
fastcgi_next_upstream error timeout;
fastcgi_next_upstream_timeout 5s;
```

O impacto prático: ao invés de acumular 1500 conexões esperando 60s cada, nginx retorna 502 imediatamente ao detectar FPM saturado, liberando recursos para servir cache estático do WP Rocket.

## 5. Impacto do `return 444` nas Métricas CloudFront

**Severidade: MÉDIA** (risco de falso positivo em alertas)

**Comportamento técnico:**
Quando nginx retorna 444 (TCP close sem resposta HTTP), o CloudFront interpreta como falha de conexão com o origin. O campo `x-edge-result-type` nos logs CloudFront registra `Error` (não `Miss` ou `Hit`).

**Impacto nas métricas CloudWatch:**
- `5xxErrorRate` **NÃO é inflada** — 444 não é uma resposta HTTP 5xx, é uma falha de conexão TCP. O CloudFront registra como `OriginError` mas a métrica `5xxErrorRate` conta apenas respostas HTTP com status 5xx recebidas do origin.
- A métrica afetada é `OriginLatency` (timeout) e o campo `origin-fbl` nos logs.
- **Porém:** `TotalErrorRate` **pode ser inflada** — CloudFront pode retornar 502 ao cliente final quando o origin fecha a conexão sem resposta, e 502 conta em `TotalErrorRate`.

**Conclusão prática:** O alerta `CloudFront 5XX Rate > 10%` existente **pode sim ser triggereado** se o volume de xmlrpc bloqueados for alto e o CloudFront retornar 502 para o usuário final em vez de absorver o 444 silenciosamente.

**Recomendação:** Verificar o comportamento na prática monitorando `5xxErrorRate` vs `BlockedRequests` WAF nas primeiras 48h. Se correlacionados, adicionar Custom Error Response no CloudFront para 502 com TTL baixo (item 2.4), o que evita que o 502 seja reportado como erro de origem.

Alternativa: considerar `return 403` no nginx em vez de 444 — retorna resposta HTTP válida, CloudFront não registra como OriginError, e WAF já bloqueia a maioria antes de chegar ao origin de qualquer forma.

## Quando usar

Sempre que um relatório for gerado durante uma conversa — seja de saúde de infraestrutura, análise de performance, diagnóstico, otimização, migração, ou qualquer documento de entrega técnica.

## Estrutura de diretórios obrigatória

```
~/scripts/reports/[área bureau]/[nome do programa]/relatorio-[tipo]-YYYY-MM-DD.[ext]
```

| Código | Nome |
|--------|------|
| `01 - Governança` | Políticas, processos, compliance |

| `08 - Segurança` | Auditorias, firewall, incidentes, pentest |

## Regras de nomenclatura

- Formato obrigatório: `relatorio-[tipo]-YYYY-MM-DD.[ext]`
- Tipos comuns: `saude`, `migracao`, `otimizacao`, `diagnostico`, `auditoria`, `performance`
- Extensões: `.html` (principal), `.pdf` (quando possível), `.md` (alternativo)
- Usar data real do dia em que o relatório foi gerado (não a data do período analisado)


1. **Identificar área bureau** com base no contexto do relatório

3. **Criar diretório** se não existir:
   ```bash
   mkdir -p "~/scripts/reports/[área]/[programa]"
   ```
4. **Salvar HTML** com design consistente Bureau (fundo escuro, glassmorphism):
   - Fundo: `#0a0a0b`
   - Glassmorphism: `backdrop-filter: blur(12px)`
   - Fontes: Inter + JetBrains Mono
   - Floating orbs, animações CSS
   - Footer com "Bureau de Tecnologia"
5. **Converter para PDF** via Playwright MCP quando disponível:
   ```
   mcp__playwright__playwright_navigate → file:///Users/dcambria/scripts/reports/[área]/[programa]/[arquivo].html
   mcp__playwright__playwright_save_as_pdf → [mesmo caminho].pdf
   ```

6. **Confirmar** ao usuário com os caminhos dos arquivos gerados

#### 2f. Hipótese para o problema de CPU com TEC

**Hipótese principal: cache miss por proliferação de query strings únicas do TEC.**

A cadeia do problema:

1. **CloudFront cache policy `WP-Dynamic-ShortTTL` tem `QueryStrings behavior: all`** — cada combinação única de `?tribe-bar-date=YYYY-MM-DD` gera uma entrada de cache separada no CloudFront.

2. **nginx bypass rule:** `if ($query_string != "")` → `rocket_bypass = 1`, **exceto** pelos parâmetros `eixo`, `jsf`, `jet-engine`, `tax`. `tribe-bar-date` e `eventDisplay` **não estão na lista de exceção** → cada request com QS do TEC vai direto para o PHP.

**Agravante:** O cron `tribe_events_pro_process_recurring_events` roda a cada 30 minutos — em sites com muitos eventos recorrentes isso adiciona carga periódica de DB.

- Adicionar `tribe-bar-date` e `eventDisplay` à lista `cache_query_strings` do WP Rocket
- Adicionar a regra de exceção no nginx: `if ($query_string ~* "^(eixo|jsf|jet-engine|tax|tribe-bar-date|eventDisplay)=")` → `rocket_bypass = 0`
- Verificar se CloudFront deve ou não variar cache por `tribe-bar-date` (atualmente varia, o que aumenta o hit rate no CF mas só após o primeiro miss por data)</result>

# Skill: save-report

## Quando usar

Sempre que um relatório for gerado durante uma conversa — seja de saúde de infraestrutura, análise de performance, diagnóstico, otimização, migração, ou qualquer documento de entrega técnica.

## Estrutura de diretórios obrigatória

```
~/scripts/reports/[área bureau]/[nome do programa]/relatorio-[tipo]-YYYY-MM-DD.[ext]
```

## Áreas bureau válidas

| Código | Nome |
|--------|------|
| `01 - Governança` | Políticas, processos, compliance |
| `02 - Infraestrutura` | AWS, servidores, redes, banco de dados |
| `03 - Web` | Sites, deploys, frontend, CDN |
| `04 - Consultoria e Suporte` | Tickets, suporte a clientes |
| `05 - Ambiente: Educação, Cultura e Treinamento` | Documentação educacional, tutoriais |
| `06 - Comercialização Software e Hardware` | Licenças, produtos, cotações |
| `07 - Automação` | Scripts, pipelines, CI/CD, bots |
| `08 - Segurança` | Auditorias, firewall, incidentes, pentest |

## Regras de nomenclatura

- Formato obrigatório: `relatorio-[tipo]-YYYY-MM-DD.[ext]`
- Tipos comuns: `saude`, `migracao`, `otimizacao`, `diagnostico`, `auditoria`, `performance`
- Extensões: `.html` (principal), `.pdf` (quando possível), `.md` (alternativo)
- Usar data real do dia em que o relatório foi gerado (não a data do período analisado)

## Exemplos de caminhos válidos

```
~/scripts/reports/02 - Infraestrutura/plesk/relatorio-saude-ec2-2026-03-15.html
~/scripts/reports/02 - Infraestrutura/plesk/relatorio-saude-ec2-2026-03-15.pdf
~/scripts/reports/08 - Segurança/cloudflare/relatorio-auditoria-waf-2026-03-20.html
~/scripts/reports/03 - Web/mfb/relatorio-performance-2026-04-01.html
~/scripts/reports/07 - Automação/backup-s3/relatorio-diagnostico-2026-03-10.md
```

## Processo de execução

1. **Identificar área bureau** com base no contexto do relatório
2. **Identificar nome do programa** — usar o diretório do projeto em `~/scripts/` (ex: `plesk`, `mfb`, `elos-dev`)
3. **Criar diretório** se não existir:
   ```bash
   mkdir -p "~/scripts/reports/[área]/[programa]"
   ```
4. **Salvar HTML** com design consistente Bureau (fundo escuro, glassmorphism):
   - Fundo: `#0a0a0b`
   - Glassmorphism: `backdrop-filter: blur(12px)`
   - Fontes: Inter + JetBrains Mono
   - Floating orbs, animações CSS
   - Footer com "Bureau de Tecnologia"
5. **Converter para PDF** via Playwright MCP quando disponível:
   ```
   mcp__playwright__playwright_navigate → file:///Users/dcambria/scripts/reports/[área]/[programa]/[arquivo].html
   mcp__playwright__playwright_save_as_pdf → [mesmo caminho].pdf
   ```
6. **Confirmar** ao usuário com os caminhos dos arquivos gerados

Os relatórios devem seguir o design das páginas do Plesk (default, maintenance, 404, suspended):
- Background escuro (`#0a0a0b`) com gradientes animados
- Glassmorphism (`backdrop-filter: blur`)

- Tabelas com hover sutil
- Badges de área e data no header
- Responsivo para mobile
