<result>## Análise técnica: `aria-hidden`+`blur()` vs `inert`

### 1. Compatibilidade real (Baseline 2026)
`inert` é **Baseline Newly available desde abril/2023**: Chrome/Edge 102+, Safari 15.5+, Firefox 112+. Cobertura global ~96%. Para a matriz exigida (Chrome 100+, Firefox 100+, Safari 15+), há um gap mínimo: Firefox 100-111 e Safari 15.0-15.4 não têm suporte nativo. Polyfill da WICG (`wicg-inert`) cobre o gap em ~3KB gzipped, mas na prática o uso real desses ranges hoje é estatisticamente irrelevante.

### 2. Comportamento semântico — "right tool for the job"
- `aria-hidden=true`: **só** afeta a árvore de acessibilidade. Não remove de tab order — daí o bug. É um *anti-pattern* em containers focáveis.
- `inert`: tripla ação — esconde de AT, remove tab order, desfoca descendentes automaticamente, e bloqueia eventos de input. Foi desenhado exatamente para o caso "modal fechado mas DOM presente".
- `display:none`: correto semanticamente, mas mata transições CSS (o painel do bureau-a11y desliza lateralmente).

`inert` é o *right tool* aqui.

### 3. Trade-offs
- **A (blur+aria-hidden)**: frágil. Se TTS dispara `focus()` async num botão "parar leitura", ou se a tab ativa restaurar foco em mount, o bug ressurge. Cada novo handler precisa lembrar do contrato.
- **B (inert)**: autocontido. Animações funcionam (não toca `visibility`/`display`). Custo: pequeno polyfill condicional para Safari 15.0-15.4.

### 4. WAI-ARIA APG Dialog Pattern
O APG recomenda **`inert` no conteúdo *fora* do dialog** quando aberto (focus trap por exclusão), e foco de retorno ao trigger. O bureau-a11y faz o **inverso** (esconde o próprio painel quando fechado). Isso é legítimo para um *non-modal disclosure panel* (não bloqueia o resto da página), mas se a intenção é dialog modal real (`role="dialog" aria-modal="true"` implícito), há refactor maior pendente: trap de foco, `Escape` para fechar, retorno de foco ao botão de abrir.

### 5. Preocupações adjacentes (não causadoras do bug, mas correlatas)
- `speechSynthesis.cancel()` no close: **obrigatório** — `inert` não para Web Speech API.
- Reading-ruler listeners: desanexar no close — `inert` não desliga listeners no `document`/`window`.
- Tratar como cleanup no mesmo `closePanel()`.

### 6. Recomendação: **B (`inert`)**, com polyfill condicional
É o padrão moderno, resolve o bug por construção, sobrevive a foco programático async (TTS) e preserva animações. Risco baixo: testar tab order pós-close e garantir que `openPanel()` zere `inert` *antes* de chamar `focus()` no primeiro item — chamar `focus()` em descendente de elemento `inert` é silenciosamente ignorado.

### 7. Bug latente — `aria-hidden=true` no estado inicial
Está **correto em intenção, errado em mecanismo**. O painel inicia fechado, então deve estar inacessível a AT. Mas com `aria-hidden=true` estático no HTML *e* descendentes focáveis (botões das tabs), qualquer foco programático inicial (autofocus, deep-link, restore de scroll) reproduz o mesmo warning do Chrome no load. Migrar para `inert` no atributo inicial resolve ambos os estados (inicial e pós-close) com a mesma primitiva — coerência semântica e elimina a classe inteira de bugs.

### Como funciona

O relatório comercial é gerado por um script Python (`gerar-relatorio-comercial.py`) que lê um arquivo YAML com os dados e produz um `.docx` nativo. Isso garante:

- **Imagens embutidas** no documento (não precisa de arquivos separados)

- **Compatibilidade total** com Word, Pages e LibreOffice

# Auditoria A3 — `waf_collector` worker + spike detection

## Findings

### 🔴 #1 — PK `NOT NULL` viola insert quando `rule_name` ou `ip_address` ausentes
**Arquivo:** `backend/workers/tasks/waf_collector.py:62-76` + `backend/app/models/security_event.py:38-39`

```python
"rule_name": fields.get("terminatingRuleId"),  # pode ser None
"ip_address": fields.get("httpRequest.clientIp"),  # pode ser None
```

`SecurityEvent.rule_name` e `ip_address` são `nullable=False` e participam da PK composta. CloudWatch Insights pode retornar campos vazios (registro malformado, `terminatingRuleId` ausente em ações `CHALLENGE`/`CAPTCHA` parcialmente bloqueadas, IP ausente em logs com cliente atrás de múltiplos hops sem `clientIp` populado). O `pg_insert` vai estourar `IntegrityError NOT NULL violation`, abortando a transação INTEIRA da conta — perdendo TODOS os eventos parsed, não só o malformado. Não há try/except no `db.execute(stmt)`.

**Fix:** filtrar parsed antes do upsert (`if not e["ip_address"] or not e["rule_name"]: continue` em `_parse_events`), OU defaults sentinela (`"unknown"`). A primeira é mais conservadora.

---

### 🔴 #2 — `user_agent` nunca preenchido (key mismatch CW Insights → parsed dict)
**Arquivo:** `backend/app/services/waf_collector.py:33` + `backend/workers/tasks/waf_collector.py:54,69`

A query Insights usa `httpRequest.headers.0.value as user_agent` — o alias é só estético no Insights console; o `field` retornado nos `results` é o **nome literal do field**, que pode vir como `user_agent` (alias preservado) OU `httpRequest.headers.0.value`. Não é determinístico via boto3 — testes empíricos mostram que `as` aliases nem sempre se propagam ao `results`. Pior: `fields = {f["field"]: f["value"]}` em linha 54 chaveia pelo nome cru. Se o alias não pegar, `fields.get("user_agent")` retorna `None` silenciosamente — coluna sempre `NULL` em prod sem warning.

**Fix:** ler ambos: `fields.get("user_agent") or fields.get("httpRequest.headers.0.value")`. Ou validar com log debug nas primeiras execuções.

---

### 🟡 #3 — Sem lock entre workers concorrentes → CW Insights billed 2x
**Arquivo:** `backend/workers/celery_app.py:124-128` (sem lock); `waf_collector.py:154-171`

Beat `*/5` + 2 workers no pool `cloudwatch` (queue routing) → ambos podem pegar o mesmo `collect_waf_events`. Cada worker lê o mesmo `last_ts`, dispara `start_query` no mesmo log_group/window → **CloudWatch Insights cobra 2x por bytesScanned** (não é insignificante: 10k records matched × N source × 2 contas pode somar centavos/ciclo, dólares/mês). `on_conflict_do_nothing` cobre DB, mas não AWS billing. PR comment diz "aceitável" — mas não há single-instance lock + nem flag `singleton`.

**Fix:** decorator `@singleton_task` via Redis lock (padrão usado em `evaluate_infra_alerts` se existir, verificar). Ou `task_acks_late=True` + `worker_prefetch_multiplier=1` + Redis `SETNX` com TTL 280s no início da task.

---

### 🟡 #4 — Baseline de 7 dias com buckets vazios não-contados → false positives
**Arquivo:** `backend/workers/tasks/waf_collector.py:297-325`

`GROUP BY floor(extract(epoch from timestamp) / 900)` só retorna buckets COM eventos. Source de tráfego esparso (ex: 3 BLOCKs/dia distribuídos) gera ~3 buckets de count=1. Mean=1, stddev≈0, threshold=mean+3σ≈1. Qualquer rajada minúscula (15 BLOCKs) dispara alerta CRITICAL. O guard `current_count &gt; 10` ajuda mas não elimina — `top_rule`/`top_ip` mostrarão ruído. Spec original (PR description) provavelmente esperava `generate_series` ou `LEFT JOIN bucket calendar` para preencher zeros.

**Fix:** ou usar `generate_series(baseline_start, window_start, '15 minutes')` + LEFT JOIN, ou adicionar floor mínimo (`threshold = max(mean + 3*stddev, 50)`). Documentar como decisão MVP se mantiver atual.

---

### 🟡 #5 — `datetime.utcnow()` naive vs `datetime.now(timezone.utc)` (mistura)
**Arquivo:** `backend/app/services/waf_collector.py:85`

```python
endTime=int(datetime.utcnow().timestamp()),
```

`utcnow()` retorna naive datetime, `.timestamp()` em naive interpreta como LOCAL timezone do host → se container roda em UTC tá ok, se acidentalmente em BRT é -3h de janela. `since_timestamp` vem com tz aware. Inconsistência defeituosa.

**Fix:** `int(datetime.now(timezone.utc).timestamp())`.

---

### 🟡 #6 — Realtime publish fora da session, mas com `new_event_count` sem reflexo de duplicatas
**Arquivo:** `backend/workers/tasks/waf_collector.py:231,240-246`

`new_event_count += len(account_parsed)` conta o que foi ENVIADO, não o que foi INSERIDO. Com `on_conflict_do_nothing`, em coleta repetida (deploy sem watermark mudou), `new_event_count &gt; 0` mas zero rows novas — frontend invalida cache à toa. Não corrompe, mas viola promessa do `data` do realtime ("há novos eventos").

**Fix:** usar `result.rowcount` do `db.execute(stmt)` (PG retorna rowcount real).

**Fix:** cooldown key composto `(rule_id, resource_id)` — mudança no `_check_cooldown`.

## Edge cases não cobertos (documentação ausente)

- **Watermark &gt; 90d retention:** se worker fica parado 100 dias, `last_ts` aponta para timestamp deletado por TimescaleDB compression/retention. `start_query` com `startTime` muito antigo retorna vazio ou erro. Sem reset automático. Documentar: "watermark &gt; retention → resetar manualmente via SQL".
- **Migration up + worker rodando:** `alembic upgrade` adiciona coluna → `INSERT` legado falha. Risco mitigado por Celery retry, mas não documentado. Recomenda playbook: parar workers WAF antes de migrar `security_events`.
- **Redis cair durante `enrich_batch`:** `redis.from_url` em `__init__` não levanta até primeira op. `get_cache` levanta `ConnectionError` → bubbla pra task → Celery retry → **eventos não inseridos por causa de Redis down**. Bug latente: enrichment é best-effort, mas falha bloqueia upsert. Wrap em try/except no worker.

# Auditor C5 — TimescaleDB / Postgres / Concorrência DB

PR #195 · `feat/security-attack-monitoring-v2` · `500c407`

## 1. ALTER TYPE COMMIT — 🟡

A sequência (linhas 140-142) é:
```python
connection.execute(sa.text("COMMIT"))   # fecha a tx aberta pelo Alembic
op.execute("ALTER TYPE triggertype ADD VALUE IF NOT EXISTS 'security_attack'")
```
**Problemas reais:**
- **Posicionamento:** vem **depois** de `create_hypertable` (PASSO 3) e **antes** do `autocommit_block` do PASSO 5. Como o `COMMIT` "vaza" da tx envelope do Alembic, todo trabalho do PASSO 2 + 3 (add_column, create_table, indexes, hypertable) é commitado nesse ponto. Se o PASSO 5 ou 6 falhar, **os passos anteriores ficaram persistidos** — `alembic upgrade` aborta mas a migration registrou progresso parcial sem alembic_version atualizado → próximo `upgrade head` falha em `cloud_accounts` (coluna duplicada) ou `security_events` (já existe). Não há `IF NOT EXISTS` no `add_column`/`create_table`. **Recomendação:** mover o bloco ALTER TYPE para **antes** do PASSO 2, ou usar `IF NOT EXISTS` em `op.add_column` (Postgres ≥ 9.6, mas Alembic não tem flag direta — usar `op.execute` raw).
- **Nova tx após COMMIT:** o `op.execute` seguinte roda em autocommit no SQLA core até o próximo statement. O `autocommit_block` do PASSO 5 lida por sua conta. Mas o PASSO 6 (linhas 183-215) volta a usar `op.execute` **sem** abrir tx explicitamente — em psycopg2/asyncpg via Alembic, isso é OK porque o engine reabre tx no próximo statement non-DDL, mas é frágil. **Recomendação:** após o ALTER TYPE, emitir `connection.execute(sa.text("BEGIN"))` para alinhar com o que o Alembic espera no `__exit__`.
- **Falha no COMMIT:** raríssimo, mas se acontecer (deadlock, conn loss), exception sobe; Alembic faz rollback do nada (já fora de tx) e marca migration como falhada. Aceitável.

## 2. PK composta + write performance — 🟢/🟡

`pg_insert(SecurityEvent).values(account_parsed).on_conflict_do_nothing(...)` em **lote único multi-row**:
- Postgres faz **um único INSERT statement**, mas valida unicidade row-a-row contra btree do PK. Em chunk recente não-comprimido, isso é eficiente. **10k events** = 10k btree lookups + ~10k inserts em ~hundreds ms.
- `ON CONFLICT DO NOTHING` em multi-row **não** valida em batch: cada tuple é avaliada individualmente, mas duplicatas dentro do **mesmo INSERT** levantam `ERROR: ON CONFLICT DO UPDATE command cannot affect row a second time` quando há **DO UPDATE**. Com **DO NOTHING**, duplicatas internas no batch são aceitas silenciosamente (PG ≥ 9.5). 🟢
- **Risco latente:** `_parse_events` não dedup interno por (timestamp, cloud_account_id, source, ip_address, rule_name). Se CloudWatch Insights retornar 2 rows idênticas em 1ms (raro mas possível em CDN com replicação), batch tem duplicata. `ON CONFLICT DO NOTHING` cobre, **mas** `result.rowcount` em PG nesse caso inclui as duplicatas como "inserted" em algumas versões. Para o broadcast WS, `new_event_count` pode estar levemente inflado — **recomendação:** dedup local no Python antes do `pg_insert`.

## 3. CAGG refresh policy — 🟢

- `start_offset=3h, end_offset=0s, schedule_interval=5min`: **bootstrap janela morta** existe. Primeiros ~5min após migration: `WITH NO DATA` + nenhum refresh ainda → `_timeline` query retorna 0 rows. Fallback em `security_service._timeline` (linhas 113-118) executa raw aggregation no `except`. **Mas o except só dispara se a query *falhar***, não se retornar 0 rows. Linha 108: `if rows:` — bem, isso cobre. 🟢
- **Performance:** 100 events/sec × 3h × 12 ciclos/h = 1.3M rows agregados a cada 5min. CAGG incremental do TimescaleDB só processa **chunks mudados** — não reagrega 3h inteiras. Carga real ~5min × 100 = 30k rows. OK. 🟢

## 4. Compression policy — 🟡

- `segmentby='cloud_account_id, source'` — boa escolha para queries multi-tenant. `_timeline` (3h) e `detect_attack_spike` baseline (7d) ambos filtram por `(cloud_account_id, source)`.
- **`detect_attack_spike` toca chunks no limite dos 7d** (linha 359: `baseline_start = now - timedelta(days=7)`). A política `add_compression_policy('security_events', INTERVAL '7 days')` comprime chunks **&gt;7d**. O CASO LIMITE: chunk de 7d-0h ainda não comprimido na primeira passada da query, mas é elegível na próxima. Query sobre chunks comprimidos é transparente mas **mais lenta** (~3-5x). Considerar `INTERVAL '8 days'` para garantir margem de segurança. 🟡
- **Index `ix_se_account_ts` em chunks comprimidos:** TimescaleDB **não usa** os índices originais em chunks comprimidos — usa metadata sparse + `segmentby`. Como a query agrupa por `(cloud_account_id, source)`, `segmentby` cobre. OK. 🟢

## 5. Retention + CAGG — 🔴

- **`add_retention_policy('security_events', INTERVAL '90 days')`** deleta base events &gt;90d. **CAGG não tem retention própria configurada** — ela retém indefinidamente. Bom para dashboards de 1 ano.
- **MAS:** a partir do PG 14 + TimescaleDB ≥ 2.6, retention policy em hypertable com CAGG dependente **falha** com `ERROR: cannot drop chunks ... because they are dependent on continuous aggregate`. Para liberar drop de chunks abaixo do CAGG, precisa configurar `materialized_only=true` (default) **mais** `ignore_invalidations_older_than` ou usar `add_retention_policy('security_events', INTERVAL '90 days', drop_after_materialized =&gt; true)` (TSDB 2.13+). **Recomendação:** testar `alembic upgrade head` num DB onde já existem chunks de 91d para confirmar comportamento — esse caminho **não está coberto pela validação manual descrita no header**.

## 6. Decompress em downgrade — 🟢

Loop varre `timescaledb_information.chunks WHERE is_compressed = TRUE` e decomprime. Em DB fresh: 0 iterações, instantâneo. Em prod com 30 dias: ~23 chunks comprimidos × ~5-20s cada = 2-7min de downgrade. Aceitável. **Falta:** `SET statement_timeout = 0` antes do loop — se cluster tiver timeout default (ex: 5min), decompression de chunk grande aborta. **Recomendação:** adicionar `op.execute("SET LOCAL statement_timeout = 0")` no início do DO block.

## 7. UPDATE com PK composta — 🟢

Implementação atual enriquece in-memory antes do INSERT (linhas 273-283). Sem UPDATE downstream. Bem coberto pelo author no comentário. Para o futuro: PK composta dificulta UPDATE-by-id; documentar antes de adicionar fluxo de re-enrich assíncrono.

## 8. Singleton lock + Redis falha — 🟡

`_singleton_lock` Redis: se Redis cair entre acquire e DB write, lock perdido. Worker B pega lock, dispara INSERT concorrente.
- **DB dedup via `on_conflict_do_nothing`:** cobre. 🟢
- **Realtime publish duplicado:** `new_event_count &gt; 0` em ambos workers → 2 broadcasts. Frontend invalida cache 2x. Wasteful mas inofensivo.
- **detect_attack_spike:** mais grave. Sem lock, dois workers podem disparar `SyncAlertService.trigger_alert` simultaneamente. Há **cooldown Redis** (`waf-spike-cooldown:{cloud_account_id}:{source}`, TTL 30min, linhas 493-509), MAS:
