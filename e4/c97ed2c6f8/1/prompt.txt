# Brainstorming Ideas Into Designs

Help turn ideas into fully formed designs and specs through natural collaborative dialogue.

Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design and get user approval.

<HARD-GATE>
Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity.
</HARD-GATE>

## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. A todo list, a single-function utility, a config change — all of them. "Simple" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval.

## Checklist

You MUST create a task for each of these items and complete them in order:

1. **Explore project context** — check files, docs, recent commits
2. **Offer visual companion** (if topic will involve visual questions) — this is its own message, not combined with a clarifying question. See the Visual Companion section below.
3. **Ask clarifying questions** — one at a time, understand purpose/constraints/success criteria
4. **Propose 2-3 approaches** — with trade-offs and your recommendation
5. **Present design** — in sections scaled to their complexity, get user approval after each section
6. **Write design doc** — save to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and commit
7. **Spec self-review** — quick inline check for placeholders, contradictions, ambiguity, scope (see below)
8. **User reviews written spec** — ask user to review the spec file before proceeding
9. **Transition to implementation** — invoke writing-plans skill to create implementation plan

## The Process

**Understanding the idea:**

- Check out the current project state first (files, docs, recent commits)
- Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first.
- If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle.
- For appropriately-scoped projects, ask questions one at a time to refine the idea
- Prefer multiple choice questions when possible, but open-ended is fine too
- Only one question per message - if a topic needs more exploration, break it into multiple questions
- Focus on understanding: purpose, constraints, success criteria

**Exploring approaches:**

- Propose 2-3 different approaches with trade-offs
- Present options conversationally with your recommendation and reasoning
- Lead with your recommended option and explain why

**Presenting the design:**

- Once you believe you understand what you're building, present the design
- Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced
- Ask after each section whether it looks right so far
- Cover: architecture, components, data flow, error handling, testing
- Be ready to go back and clarify if something doesn't make sense

**Design for isolation and clarity:**

- Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
- For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
- Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
- Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.

**Working in existing codebases:**

- Explore the current structure before proposing changes. Follow existing patterns.
- Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
- Don't propose unrelated refactoring. Stay focused on what serves the current goal.

## After the Design

**Documentation:**

- Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
  - (User preferences for spec location override this default)
- Use elements-of-style:writing-clearly-and-concisely skill if available
- Commit the design document to git

**Spec Self-Review:**
After writing the spec document, look at it with fresh eyes:

1. **Placeholder scan:** Any "TBD", "TODO", incomplete sections, or vague requirements? Fix them.
2. **Internal consistency:** Do any sections contradict each other? Does the architecture match the feature descriptions?
3. **Scope check:** Is this focused enough for a single implementation plan, or does it need decomposition?
4. **Ambiguity check:** Could any requirement be interpreted two different ways? If so, pick one and make it explicit.

Fix any issues inline. No need to re-review — just fix and move on.

**User Review Gate:**
After the spec review loop passes, ask the user to review the written spec before proceeding:

> "Spec written and committed to `<path>`. Please review it and let me know if you want to make any changes before we start writing out the implementation plan."

Wait for the user's response. If they request changes, make them and re-run the spec review loop. Only proceed once the user approves.

**Implementation:**

- Invoke the writing-plans skill to create a detailed implementation plan
- Do NOT invoke any other skill. writing-plans is the next step.

## Key Principles

- **One question at a time** - Don't overwhelm with multiple questions
- **Multiple choice preferred** - Easier to answer than open-ended when possible
- **YAGNI ruthlessly** - Remove unnecessary features from all designs
- **Explore alternatives** - Always propose 2-3 approaches before settling
- **Incremental validation** - Present design, get approval before moving on
- **Be flexible** - Go back and clarify when something doesn't make sense

## Visual Companion

A browser-based companion for showing mockups, diagrams, and visual options during brainstorming. Available as a tool — not a mode. Accepting the companion means it's available for questions that benefit from visual treatment; it does NOT mean every question goes through the browser.

**Offering the companion:** When you anticipate that upcoming questions will involve visual content (mockups, layouts, diagrams), offer it once for consent:
> "Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)"

**This offer MUST be its own message.** Do not combine it with clarifying questions, context summaries, or any other content. The message should contain ONLY the offer above and nothing else. Wait for the user's response before continuing. If they decline, proceed with text-only brainstorming.

**Per-question decision:** Even after the user accepts, decide FOR EACH QUESTION whether to use the browser or the terminal. The test: **would the user understand this better by seeing it than reading it?**

- **Use the browser** for content that IS visual — mockups, wireframes, layout comparisons, architecture diagrams, side-by-side visual designs
- **Use the terminal** for content that is text — requirements questions, conceptual choices, tradeoff lists, A/B/C/D text options, scope decisions

A question about a UI topic is not automatically a visual question. "What does personality mean in this context?" is a conceptual question — use the terminal. "Which wizard layout works better?" is a visual question — use the browser.

If they agree to the companion, read the detailed guide before proceeding:
`skills/brainstorming/visual-companion.md`

# Skill: bit-canvas

> **v3.0.0 — BREAKING CHANGE.** O roteamento de arestas foi extraido para o [elkjs](https://github.com/kieler/elkjs) (Eclipse Layout Kernel). A partir desta versao, edges exigem que o script `elk.bundled.js` esteja carregado globalmente e que `canvas.autoLayout()` seja chamado apos adicionar nos e edges. O pathfinder interno (Dijkstra + bezier shortcut) foi removido. Ver secao **Auto-layout com elkjs** abaixo.

## Quando usar

Sempre que o usuario pedir para criar um canvas interativo, grafo visual, node editor, fluxograma editavel, ou qualquer interface que envolva nos draggaveis conectados por edges. O bit-canvas e a biblioteca interna do Bureau de Tecnologia para isso.

## Renderer padrao (OBRIGATORIO)

**REGRA: SEMPRE usar `renderer: 'standard'`.** NAO criar renderers customizados. O renderer padrao garante consistencia visual (fontes, cores, word-wrap, tema, badges) em todos os diagramas Bureau IT. Ele integra automaticamente com o sistema de temas e suporta todos os campos de dados abaixo.

### Campos de dados suportados pelo renderer padrao

| Campo | Tipo | Descricao |
|-------|------|-----------|
| `data.label` | string | Texto principal do card (word-wrap automatico) |
| `data.sub` | string | Texto secundario abaixo do label (word-wrap automatico) |
| `data.color` | string hex | Cor do card (borda, badge, sombra). Ex: `'#4ade80'` |
| `data.phase` | string | Badge no topo-esquerdo (ex: `'01'`, `'DEPLOY'`) |
| `data.icon` | string | Texto no topo-direito (ex: `'[DB]'`, `'[API]'`) |
| `data.version` | string | Label no canto inferior-direito (ex: `'v1.2'`) |
| `data.comment` | string | Texto de comentario. Exibe icone "i" clicavel que abre tooltip |
| `data.shape` | string | Forma do card (ver tabela abaixo). Default: `'rect'` |
| `data.isGroup` | boolean | Se `true`, renderiza como container tracejado (grupo) |
| `data.groupTitle` | string | Titulo do grupo ou swimlane |

### Shapes (formas de card)

O campo `data.shape` determina a forma visual do card. Se omitido, assume `'rect'`.

| `data.shape` | Visual | Quando usar |
|--------------|--------|-------------|
| `rect` | Retangulo arredondado (padrao) | Processos, acoes, steps, qualquer no generico |
| `diamond` | Losango (diamante) | Decisoes, condicoes, if/else, validacoes, gates |
| `terminal` | Retangulo com bordas muito arredondadas (pilula) | Inicio/fim de fluxo, triggers, terminadores |
| `subprocess` | Retangulo com borda dupla | Subprocessos, chamadas externas, modulos referenciados |
| `document` | Retangulo com borda inferior ondulada | Documentos, relatorios, outputs, logs |
| `cylinder` | Cilindro (com tampa eliptica) | Bancos de dados, storage, cache, filas |
| `swimlane` | Container com header solido | Fases, departamentos, responsaveis (container) |

**REGRA DE MAPEAMENTO SEMANTICO** — ao criar diagramas de fluxo, use esta tabela para escolher o shape correto:

| Conceito no diagrama | `data.shape` |
|---|---|
| Decisao, condicao, if/else, validacao, gateway | `diamond` |
| Inicio, fim, start, end, trigger, terminator | `terminal` |
| Subprocesso, chamada externa, modulo, referencia | `subprocess` |
| Documento, relatorio, output, log, arquivo | `document` |
| Banco de dados, storage, cache, fila, bucket | `cylinder` |
| Fase, departamento, responsavel, lane | `swimlane` |
| Qualquer outro (processo, acao, step, task) | `rect` (ou omitir) |

Exemplo:
```js
// Decisao
canvas.addNode('d1', {
  label: 'Aprovado?', shape: 'diamond', color: '#f59e0b',
}, { x: 200, y: 100, width: 120, height: 100 });

// Inicio
canvas.addNode('start', {
  label: 'Inicio', shape: 'terminal', color: '#4ade80',
}, { x: 50, y: 100, width: 100, height: 50 });

// Banco de dados
canvas.addNode('db1', {
  label: 'PostgreSQL', shape: 'cylinder', icon: '[DB]', color: '#60a5fa',
}, { x: 350, y: 100, width: 120, height: 80 });
```

### Edge labels (rotulos de conexao)

Edges suportam `edge.data.label` para rotular conexoes (ex: "Sim", "Nao", "Erro"):

```js
canvas.addEdge('e1', 'd1', 'out', 'n2', 'in', { label: 'Sim' });
canvas.addEdge('e2', 'd1', 'bottom', 'n3', 'top', { label: 'Nao' });
```

O label e renderizado no ponto medio do caminho ortogonal com fundo translucido.

**Importante:** labels em arestas longas (4+ segmentos, dorsais ELK) podem ficar posicionados fora do campo visual esperado, aparentemente "flutuando" sobre outros elementos. Se uma aresta precisa de label informativo, garanta um caminho curto (portas opostas, nos proximos); caso contrario, prefira anotar a informacao no card de origem via `data.sub` ou `data.comment`.

### Portas registradas automaticamente

O renderer padrao registra 4 portas em cards normais:
- `'out'` — direita (centro vertical)
- `'in'` — esquerda (centro vertical)
- `'bottom'` — baixo (centro horizontal)
- `'top'` — topo (centro horizontal)

Portas do diamond ficam nos vertices do losango. Grupos e swimlanes nao tem portas.

### Containers (grupos e swimlanes)

**Grupos** (`data.isGroup: true`): retangulo tracejado com fundo sutil. Titulo via `data.groupTitle` (fieldset-legend).

**Swimlanes** (`data.shape: 'swimlane'`): retangulo com header solido e titulo no topo. Titulo via `data.groupTitle` ou `data.label`.

Ambos:
- Arrastar move todos os nos contidos dentro
- Sem resize handles nem portas
- Renderizados atras dos nos normais

```js
// Grupo
canvas.addNode('g1', {
  isGroup: true, groupTitle: 'PHASE 1 — Build', color: '#60a5fa',
}, { x: 50, y: 50, width: 500, height: 200 });

// Swimlane
canvas.addNode('lane1', {
  shape: 'swimlane', groupTitle: 'DevOps', color: '#a78bfa',
}, { x: 50, y: 50, width: 500, height: 200 });
```

### Comentarios

Nos com `data.comment` exibem um icone "i" no card. Ao clicar:
- **Edit mode**: abre textarea editavel (salva no blur)
- **View mode**: abre tooltip read-only

## Regras de modelagem de arestas

Siga estas 4 regras ao escolher `from`/`fromPort`/`to`/`toPort` de cada aresta. Mesmo com ELK roteando tudo, decisoes semanticas erradas (resposta voltando ao cliente, portas convergentes) pioram a leitura do diagrama.

### Regra 1 — HTTP redirects, erros e respostas terminais NAO sao arestas de volta ao cliente

O edge routing nao tem semantica de "resposta" — ele desenha uma linha fisica entre dois pontos. Conectar `cf-func.bottom -> browser.bottom` forca o ELK a tracar uma dorsal longa contornando o diagrama, resultando em leitura confusa.

**Modele como:**
- `data.comment` ou `data.sub` no card de origem, **ou**
- Um no terminal isolado (`shape: 'terminal'`, cor vermelha/amarela) que **recebe** a seta do no de origem e **termina ali** — sem voltar ao cliente.

```js
// ERRADO (gera a "pipa"):
canvas.addEdge('e-301', 'cf-func', 'bottom', 'browser', 'bottom',
  { label: 'www? 301 -> apex' });

// CERTO (anotacao in-card):
canvas.addNode('cf-func', {
  label: 'CF Function', sub: '301 www -> apex', /* ... */
});

// OU CERTO (terminal dedicado):
canvas.addNode('resp-301', {
  label: '301 Redirect', shape: 'terminal', color: '#eab308',
}, { x: COL.cf, y: 210, width: 160, height: 50 });
canvas.addEdge('e-301', 'cf-func', 'bottom', 'resp-301', 'top');
```

### Regra 2 — NUNCA convergir 2+ arestas no mesmo port do mesmo no

Em v3 o ELK ja espacha arestas colineares automaticamente (sem cumulativos), mas duas arestas chegando no mesmo port geram dois stubs sobrepostos no ponto de entrada — visualmente indistinguiveis. Alem disso, leitura do diagrama fica ambigua: nao da pra seguir "qual aresta veio de onde".

Cada conexao entrando em um mesmo no deve usar um port distinto: `in`, `top`, `bottom` (ou um port extra de `out` caso haja espaco no layout). Se precisar de 3+ entradas, use um no "juncao" intermediario.

```js
// ERRADO (gera o losango/flicker):
canvas.addEdge('e-apex-cf', 'r53-apex', 'out', 'cf-func', 'in');
canvas.addEdge('e-www-cf',  'r53-www',  'out', 'cf-func', 'in');  // <- mesmo port

// CERTO (portas diferentes):
canvas.addEdge('e-apex-cf', 'r53-apex', 'out', 'cf-func', 'in');
canvas.addEdge('e-www-cf',  'r53-www',  'out', 'cf-func', 'top');

// CERTO (no juncao quando 3+ entradas):
canvas.addNode('j1', { label: '', shape: 'rect' },
  { x: 400, y: 200, width: 20, height: 20 });
canvas.addEdge('e-a', 'a', 'out', 'j1', 'in');
canvas.addEdge('e-b', 'b', 'out', 'j1', 'in');
canvas.addEdge('e-c', 'c', 'out', 'j1', 'in');
canvas.addEdge('e-j1', 'j1', 'out', 'target', 'in');
```

### Regra 3 — Combinacao de portas importa: nao use `bottom -> bottom` nem `top -> top`

Mesmo com ELK, portas na mesma direcao forcam a dorsal a "dar a volta" no lado oposto do no, gerando L-shapes longos. Use portas opostas sempre que possivel para que a rota cruze a minima distancia.

Combinacoes recomendadas (portas opostas, fluxo natural):

| Direcao do fluxo | `fromPort` | `toPort` |
|---|---|---|
| Esquerda -> Direita | `out` | `in` |
| Direita -> Esquerda | `in` | `out` |
| Cima -> Baixo | `bottom` | `top` |
| Baixo -> Cima | `top` | `bottom` |

Se precisa realmente conectar `bottom -> bottom` (ex: retorno em U), avalie primeiro se aquilo e mesmo uma aresta (ver Regra 1).

### Regra 4 — Arestas NUNCA devem atravessar nos nao-container

Em v3, quando `canvas.autoLayout()` e chamado, **cruzamentos sobre nos sao impossiveis** — o ELK usa o algoritmo `layered` com `elk.edgeRouting: ORTHOGONAL`, que garante ortogonalidade + zero overlap entre rotas e bboxes. Containers (swimlanes, grupos) nao entram no grafo do ELK (nao tem portas), entao arestas **podem** atravessa-los — isso e intencional, e o comportamento de Bureau IT.

Se voce ainda ver uma linha atravessando um card (nao container), investigue nesta ordem:

1. **`autoLayout()` nao foi chamado** — o render esta usando a polilinha fallback (L-path direto por portas). Chame `await canvas.autoLayout()` apos `addEdge()` (ou passe `edges: { autoLayout: true }` no construtor).
2. **`elk.bundled.js` nao carregou** — verifique no console: `window.ELK` deve existir. Em falha de rede ou ordem errada de `<script>`, o `autoLayout()` lanca erro claro.
3. **ELK timeout/erro** — grafos muito grandes (500+ nos) podem exceder o default de `layout`. Nesse caso, lance um erro no `.catch()` de `autoLayout()` e abra os logs do ELK.

### Regra 5 — Todo no deve estar dentro dos limites da sua swimlane

Swimlanes sao **visuais**, nao contem os nos logicamente. Se um no e posicionado com `x, y` fora do retangulo da swimlane que deveria agrupa-lo, ele aparece "solto" e as arestas que o conectam atravessam outras swimlanes, cruzando dezenas de nos no caminho.

**Padrao recomendado:** definir as swimlanes primeiro, extrair suas coordenadas em constantes, e derivar `x, y` dos nos a partir delas (nunca hardcodar `x` global).

```js
// Limites da swimlane
const LANE_CF = { x: 500, y: 20, w: 680, h: 1380 };

// No deve ficar dentro:
const s3 = {
  x: LANE_CF.x + 40,            // margem interna
  y: LANE_CF.y + 400,
  width: 260, height: 110,
};
// Verificar: s3.x + s3.width <= LANE_CF.x + LANE_CF.w
//           s3.y + s3.height <= LANE_CF.y + LANE_CF.h
```

## Auto-layout com elkjs

Em v3, o roteamento de todas as arestas e delegado ao [elkjs](https://github.com/kieler/elkjs) (Eclipse Layout Kernel, algoritmo `layered` + `ORTHOGONAL`). O bit-canvas **nao tem** pathfinder interno — o `EdgeManager` so renderiza a polilinha de `edge.data.points`. A API publica `canvas.autoLayout()` encapsula tudo.

### Setup

1. **Carregar o bundle UMD** antes do modulo ES:
   ```html
   <script src="./elk.bundled.js"></script>
   ```
   Copiar `~/scripts/bit-canvas/elk.bundled.js` para o diretorio servido junto com `bit-canvas.js`. Verificar `window.ELK` no console.

2. **Construir o canvas normalmente** — adicione nos e edges sem `points` manuais:
   ```js
   canvas.addNode('n1', { label: 'A' }, { x: 0, y: 0, width: 130, height: 110 });
   canvas.addNode('n2', { label: 'B' }, { x: 300, y: 0, width: 130, height: 110 });
   canvas.addEdge('e1', 'n1', 'out', 'n2', 'in');
   ```

3. **Chamar `await canvas.autoLayout()`** depois de adicionar tudo:
   ```js
   await canvas.autoLayout();           // roteia edges, mantem posicoes dos nos
   await canvas.autoLayout({ preservePositions: true });   // default
   await canvas.autoLayout({ preservePositions: false });  // ELK tambem recoloca nos
   ```

### Modo declarativo

Passar `edges: { autoLayout: true }` ao construtor agenda `autoLayout()` automaticamente no proximo frame apos `canvas:ready` — util em boot scripts.

```js
const canvas = new BitCanvas({
  container: document.getElementById('wrap'),
  renderer: 'standard',
  edges: { autoLayout: true },
  grid: { style: 'dots', size: 30 },
});
```

`edges: false` continua desligando arestas por completo. `edges: true` (ou omitido) = arestas sem auto-layout — voce precisa chamar `canvas.autoLayout()` manualmente se quiser rotas ELK.

### `layoutOptions` custom

`autoLayout()` aceita um objeto `layoutOptions` que e mesclado sobre os defaults do bit-canvas:

```js
await canvas.autoLayout({
  layoutOptions: {
    'elk.spacing.edgeEdge': 30,
    'elk.spacing.edgeNode': 30,
    'elk.direction': 'DOWN',
  },
});
```

Defaults internos: `{'elk.algorithm': 'layered', 'elk.edgeRouting': 'ORTHOGONAL', 'elk.spacing.edgeEdge': 20, 'elk.spacing.edgeNode': 20, 'elk.layered.considerModelOrder.strategy': 'PREFER_EDGES'}`.

### Caso avancado — `edge.data.points` manual

Se voce precisa de uma rota especifica (ex: forcar passagem por um corredor), pode passar `points: [...]` ao `addEdge()` em coordenadas de mundo. O renderer **confia** nesses pontos sem validar. Depois de um `autoLayout()` chamado novamente, os `points` sao sobrescritos pela rota do ELK.

```js
canvas.addEdge('e-manual', 'a', 'out', 'b', 'in', {
  points: [
    { x: 500, y: 100 },   // world coords; precisa bater com as portas
    { x: 500, y: 400 },
    { x: 800, y: 400 },
  ],
});
```

Use este modo com moderacao — o caminho feliz e `autoLayout()`.

## Interacoes do usuario

### Mouse
- **Scroll** — pan
- **Ctrl+Scroll** — zoom (ancora no cursor)
- **Drag no** — mover (com snap de grid + guides)
- **Drag canto/borda de no selecionado** — resize (8 handles: 4 cantos + 4 bordas, min 40x30)
- **Drag area vazia** — rubber-band selection
- **Alt+Drag / Botao direito drag** — pan
- **Clique em edge** — seleciona edge (destaque laranja)
- **Drag porta** — criar edge (preview ortogonal tracejada)
- **Duplo-clique area vazia** — emite `canvas:dblclick`
- **Duplo-clique no** — emite `node:dblclick`
- **Duplo-clique guide** — remove guide
- **Drag da regua** — criar guide horizontal ou vertical
- **Botao direito em edge** — context menu com Delete

### Teclado (edit mode)
- **Delete/Backspace** — remover nos selecionados OU edge selecionada
- **Ctrl+A** — selecionar todos visiveis
- **ESC** — limpar selecao (nos e edges)
- **Ctrl+D** — emitir `node:duplicate` (app cria a copia)

### View mode
- Somente pan e zoom
- Delete, Ctrl+D, drag de no, criacao de edge bloqueados
- Context menu desabilitado
- ObjectToolbar oculta

## Guides (linhas-guia)

- Arrastar da regua superior cria guide horizontal
- Arrastar da regua esquerda cria guide vertical
- Duplo-clique sobre guide remove
- Nos se alinham a guides dentro de 20px ao arrastar
- Guides sao salvas no JSON (`canvas.guides: { h: [], v: [] }`)
- Cor: laranja tracejado (`rgba(251,146,60,0.7)`)

## Layers (camadas por cor)

O LayerPanel agrupa nos automaticamente por `node.data.color`. Cada cor gera uma checkbox para mostrar/ocultar todos os nos daquela cor. Nao e necessario configurar — basta definir `data.color` nos nos.

## Print

Botao na ZoomBar. Ao imprimir:
- Auto-fit: viewport ajustado para caber todos os nos antes de imprimir
- Grid, rulers, guides, UI ocultados
- Somente nos e edges renderizados
- Fundo branco via CSS `@media print`
- Viewport restaurado apos impressao
- Restaura apos `afterprint`

## Persistencia

Com `persistence: { key, debounce }`:
- Auto-save no localStorage a cada alteracao (debounced)
- `loadFromStorage()` chamado no construtor — sobrescreve initialNodes
- JSON inclui: pan, scale, guides, nodes (com data), edges

Para export/import manual:
```js
const json = canvas.exportJSON();
// salvar em arquivo, enviar para servidor, etc.
canvas.importJSON(json);
```

## Erros comuns a evitar

1. **Criar renderer customizado** — SEMPRE usar `renderer: 'standard'`. NAO inventar renderers.
2. **Esquecer o `renderer`** — e obrigatorio, sem ele o BitCanvas lanca erro
3. **Usar `addEdge` para import** — usar `_addEdgeDirect()` que bypassa canConnect
4. **Esperar que `node:duplicate` clone automaticamente** — o evento so notifica, a app deve criar o no
5. **CSS do container sem `position: relative`** — o canvas e `position: absolute`, precisa de pai posicionado
6. **Nao colocar `overflow: hidden`** — o canvas pode vazar do container
7. **Tentar usar em Node.js** — e browser-only (depende de HTMLCanvasElement, localStorage)
8. **Usar `snap.size` sem `snap.enabled: true`** — snap desativado por padrao
9. **Hardcodar cores** — usar `data.color` nos nos, o renderer e tema resolvem o resto
10. **HTTP redirects/respostas como arestas de volta** — modele como comment/terminal, nao edge bidirecional. Ver Regra 1.
11. **2+ arestas no mesmo port** — stubs sobrepostos + leitura ambigua. Use ports distintos ou no juncao. Ver Regra 2.
12. **Combinacao bottom->bottom, top->top, in->in, out->out** — forca caminho ortogonal longo. Use portas opostas. Ver Regra 3.
13. **Linha atravessando o corpo de um no** — sinal de que `autoLayout()` nao foi chamado ou `elk.bundled.js` nao carregou. Ver Regra 4.
14. **No posicionado fora dos limites da sua swimlane** — edges atravessam outras lanes. Derive x,y das constantes da lane. Ver Regra 5.
15. **Esquecer de carregar `elk.bundled.js`** — `canvas.autoLayout()` lanca erro claro. Adicione `<script src="./elk.bundled.js"></script>` antes do modulo ES.
16. **Esquecer de chamar `await canvas.autoLayout()`** — sem isso o render cai no fallback de L-path direto, ignorando obstaculos. Use `edges: { autoLayout: true }` para automatizar.

# Relatório A5 — Cutover blue-green CloudFront-OAC para `/wp-content/uploads/*`

## 1. Estado X-Test-Green em CF — gaps identificados

Cache policies que já incluem `X-Test-Green` no cache key:
- `wp-cache-default-hostaware` (8e1062b8…) — DefaultBehavior. Headers: `X-Test-Green`, `Host`. Cookies WP, QS whitelist `jet_download`.
- `wp-cache-default` (5af30473…) — não usado atualmente.
- `WP-Dynamic-ShortTTL-TestGreen` (0fba5fe1…) — não usado atualmente.

Origin Request Policy `WP-Dynamic-ForwardHostQS` (a0be8784…) também encaminha `X-Test-Green` + `Host` para origem.

**Gap crítico para o cutover:**
- O behavior `wp-content/uploads/*` usa cache policy `WP-Uploads-Static-1y` (0397e9f0…) que tem `HeaderBehavior: none`. **`X-Test-Green` NÃO está no cache key desse path.** Isto significa que respostas blue/green do mesmo URI seriam compartilhadas. Para validar OAC com X-Test-Green é necessário criar nova cache policy.
- Origin Request Policy do behavior `wp-content/uploads/*` é `Managed-AllViewer` — encaminha tudo. Para S3+OAC isto vira problema (S3 rejeita Authorization quando reescrito por OAC; encaminhar Host quebra SigV4).

## 2. CloudFront Functions existentes — extensibilidade

O padrão atual (`concertacao-cross-host-redirect.js` + `deploy-cf-function.sh`) está acoplado ao DefaultCacheBehavior (1 função / viewer-request). O script de deploy é idempotente e suporta `--detach` (rollback).

**Padrão recomendado para A5**: NÃO usar CF Function para escolher origin (CF Functions não podem mudar `OriginId`; só Lambda@Edge faz isso e adiciona latência/custo). Usar **2 behaviors com path patterns mutuamente exclusivos** ou **header-based cache key** + dois behaviors com prioridade de avaliação. A abordagem mais simples e rollbackável é a **Opção B abaixo: 2 origins distintos com cache key híbrido**, mas só faz sentido com Lambda@Edge.

## 3. Métricas e gaps

Métricas CW disponíveis para `E2F1QD7E7YOYEB` (us-east-1): `4xxErrorRate`, `5xxErrorRate`, `TotalErrorRate`, `BytesDownloaded`, `BytesUploaded`, `Requests`, `FunctionInvocations`, `FunctionComputeUtilization`.

- `CacheHitRate` requer **Additional Metrics** habilitadas (paid: ~$0.30/distribution/mês). Não habilitado.

## 4. Estratégia de rollback recomendada — **Opção A (variante)**

**Decisão: Opção A modificada** — manter behavior `wp-content/uploads/*` apontando para ALB e criar **NOVO behavior mais específico** que apenas swap em runtime via "promote/demote" do PathPattern.

## Investigação concluída — bit-elementor-local-cache.php v1.1.0

### Versão Elementor
**3.35.8** (confirmado via header de `/var/www/concertacaoamazonia.com.br/wp-content/plugins/elementor/elementor.php`).

### Causa raiz: combinação de **(a) cache estático** + **(d) timing do path** + **(c) método ausente da whitelist**

**Evidência file:line:**

1. **`Base::$wp_uploads_dir` é cache estático populado UMA vez por request** — `core/files/base.php:18` declara `private static $wp_uploads_dir = []`, e `base.php:296-303` (`get_wp_uploads_dir()`) só chama `wp_upload_dir()` se a chave do blog estiver vazia. **A primeira chamada a `wp_upload_dir()` é feita pelo S3-Uploads (ou frontend, ou qualquer plugin) muito antes do filtro de escrita ser relevante** — daí em diante, o array é cacheado e nenhum filtro `upload_dir` é disparado.

2. **O path é resolvido no `__construct`, não no `write()`** — `base.php:81-99`: o construtor chama `set_path()` (linha 309-317), que chama `get_base_uploads_dir()` → `get_wp_uploads_dir()`. **Quando `write()` (linha 193) executa `file_put_contents( $this-&gt;path, … )`, o `$this-&gt;path` já foi calculado antes — provavelmente em outro momento e provavelmente sem o backtrace incluir `write/save/update`.**

3. **A whitelist do mu-plugin não cobre o caller real.** Os métodos da whitelist (`write`, `update`, `update_file`, `delete`) **chamam `file_put_contents` direto sobre `$this-&gt;path` já memoizado** — não chamam mais `wp_upload_dir`. O filtro nem é invocado durante o `write()`. E quando o construtor chama `set_path()`, o frame relevante do backtrace é `set_path` (não whitelisted) ou `__construct` (não whitelisted).

4. **`Base::create()` é static factory** (linha 73), executa `Plugin::$instance-&gt;files_manager-&gt;get(...)` que via Reflection chama `__construct` (`manager.php:40-50`). O frame "create" pode até aparecer no backtrace, **mas o construtor é executado antes que `wp_upload_dir` seja chamado pela primeira vez no request** (S3-Uploads tipicamente dispara em wp-init).

**Em resumo:** o filtro `upload_dir` quase nunca é chamado durante uma operação de escrita — o cache estático do Elementor + o cache pré-aquecido pelo S3-Uploads no boot fazem com que `wp_upload_dir()` retorne o valor S3-cached antes da hora.

### Correção recomendada — abandonar `upload_dir` e usar `elementor/files/file_name`

Esse é o **único filtro disponível em 3.35.8** dentro do pipeline (`base.php:92`, dispara no construtor antes de `set_path`). Mas ele só nomeia o arquivo, não muda o diretório base. **Não há filtro nativo para path/url no Elementor 3.35.8** (confirmado pelo grep — só existem: `file_name`, `temp-file-path`, `temp-dir`, `zip/extraction-directory`, `selectors`, `property`).

**Estratégia viável:** **substituir a classe `Post_CSS` via `elementor/css-file/post/class`** — esse hook não existe em 3.35.x. **Logo, o caminho correto é override pelo `elementor/files/file_name` + sobrescrever `set_path` é impossível** (private).

**Solução robusta sugerida:**

```php
// A) Pré-aquecer o cache estático do Elementor com path local ANTES do S3-Uploads aquecê-lo
add_action( 'plugins_loaded', function () {
    if ( ! class_exists( '\Elementor\Core\Files\Base' ) ) return;
    add_filter( 'upload_dir', 'bit_elc_force_local', 5 ); // priority 5: ANTES do S3-Uploads (10)
    \Elementor\Core\Files\Base::get_base_uploads_dir(); // força populate do cache estático
    remove_filter( 'upload_dir', 'bit_elc_force_local', 5 );
}, 20 );

function bit_elc_force_local( $dirs ) {
    $blog_id = get_current_blog_id();
    $suffix  = ( is_multisite() &amp;&amp; $blog_id &gt; 1 ) ? "/sites/{$blog_id}" : '';

    $dirs['path']    = $base;
    $dirs['url']     = $dirs['baseurl'];
    return $dirs;
}
```

## Diagnóstico: ERR_CONNECTION_CLOSED em CSS files (CF→ALB→green)

### 1. Diff de timeouts (browser → CF → ALB → EC2)

| Hop | Setting | Valor atual |
|---|---|---|
| Browser → CF | viewer idle | ~60s (CF default) |
| **CF → ALB** | **OriginReadTimeout** | **30s** |
| **CF → ALB** | **OriginKeepaliveTimeout** | **5s** |
| **ALB** | **idle_timeout** | **180s** |
| **ALB** | **client_keep_alive** | **3600s** |
| ALB → EC2 (green) | TG protocol | HTTP:80 (healthy) |

### 2. Ponto de fechamento

**Smoking gun em métricas**: `TargetResponseTime` p50/p95/p99 = **4.79s — 5.28s — colapsado** em alguns datapoints (todos os percentis idênticos = origem respondeu de forma assimétrica, fila travando). E `5xxErrorRate` no CF tem picos de **100%** no Maximum (mediana 0% mas spikes em 4 buckets de 5 min na última hora). `TargetConnectionErrorCount = 0` e `HTTPCode_Target_5XX = 0` → **o ALB não está vendo erro do target**, mas o CF está reportando 5xx para o cliente.

Combinação clássica de **mismatch keepalive**:

- **CF → ALB OriginKeepaliveTimeout = 5s** (default conservador)
- **ALB idle_timeout = 180s**
- Quando CF reaproveita conexão keep-alive e o ALB já fechou o pool no lado dele, o request entra em socket meio-fechado → **CF entrega `ERR_CONNECTION_CLOSED` ao browser** (manifesta-se em CSS porque são chunks pequenos, paralelizados sobre HTTP/2, multiplicando a chance de cair numa conexão zumbi).

Nota: ALB tem `routing.http2.enabled=true` e CF tem `HttpVersion=http2` — HTTP/2 está OK em ambos. **O problema não é negociação de protocolo, é pool de conexões keep-alive.**

### 3. Métricas relevantes

- `5xxErrorRate` Maximum = 100% em buckets 15:38, 15:58, 16:03, 16:08, 16:13 (last 1h)
- `TargetResponseTime` p99 = 4.79s/5.28s em buckets onde houve poucos requests (queue-of-one)
- `TargetConnectionErrorCount` = 0, `HTTPCode_Target_5XX` = 0 → erros são **de conexão entre CF↔ALB**, não do EC2

# Brainstorming Ideas Into Designs

Help turn ideas into fully formed designs and specs through natural collaborative dialogue.

Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design and get user approval.

<HARD-GATE>
Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity.
</HARD-GATE>

## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. A todo list, a single-function utility, a config change — all of them. "Simple" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval.

## Checklist

You MUST create a task for each of these items and complete them in order:

1. **Explore project context** — check files, docs, recent commits
2. **Offer visual companion** (if topic will involve visual questions) — this is its own message, not combined with a clarifying question. See the Visual Companion section below.
3. **Ask clarifying questions** — one at a time, understand purpose/constraints/success criteria
4. **Propose 2-3 approaches** — with trade-offs and your recommendation
5. **Present design** — in sections scaled to their complexity, get user approval after each section
6. **Write design doc** — save to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and commit
7. **Spec self-review** — quick inline check for placeholders, contradictions, ambiguity, scope (see below)
8. **User reviews written spec** — ask user to review the spec file before proceeding
9. **Transition to implementation** — invoke writing-plans skill to create implementation plan

## Process Flow

## The Process

**Understanding the idea:**

- Check out the current project state first (files, docs, recent commits)
- Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first.
- If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle.
- For appropriately-scoped projects, ask questions one at a time to refine the idea
- Prefer multiple choice questions when possible, but open-ended is fine too
- Only one question per message - if a topic needs more exploration, break it into multiple questions
- Focus on understanding: purpose, constraints, success criteria

**Exploring approaches:**

- Propose 2-3 different approaches with trade-offs
- Present options conversationally with your recommendation and reasoning
- Lead with your recommended option and explain why

**Presenting the design:**

- Once you believe you understand what you're building, present the design
- Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced
- Ask after each section whether it looks right so far
- Cover: architecture, components, data flow, error handling, testing
- Be ready to go back and clarify if something doesn't make sense

**Design for isolation and clarity:**

- Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
- For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
- Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
- Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.

**Working in existing codebases:**

- Explore the current structure before proposing changes. Follow existing patterns.
- Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
- Don't propose unrelated refactoring. Stay focused on what serves the current goal.

## After the Design

**Documentation:**

- Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
  - (User preferences for spec location override this default)
- Use elements-of-style:writing-clearly-and-concisely skill if available
- Commit the design document to git

**Spec Self-Review:**
After writing the spec document, look at it with fresh eyes:

1. **Placeholder scan:** Any "TBD", "TODO", incomplete sections, or vague requirements? Fix them.
2. **Internal consistency:** Do any sections contradict each other? Does the architecture match the feature descriptions?
3. **Scope check:** Is this focused enough for a single implementation plan, or does it need decomposition?
4. **Ambiguity check:** Could any requirement be interpreted two different ways? If so, pick one and make it explicit.

Fix any issues inline. No need to re-review — just fix and move on.

**User Review Gate:**
After the spec review loop passes, ask the user to review the written spec before proceeding:

> "Spec written and committed to `<path>`. Please review it and let me know if you want to make any changes before we start writing out the implementation plan."

Wait for the user's response. If they request changes, make them and re-run the spec review loop. Only proceed once the user approves.

**Implementation:**

- Invoke the writing-plans skill to create a detailed implementation plan
- Do NOT invoke any other skill. writing-plans is the next step.

## Key Principles

- **One question at a time** - Don't overwhelm with multiple questions
- **Multiple choice preferred** - Easier to answer than open-ended when possible
- **YAGNI ruthlessly** - Remove unnecessary features from all designs
- **Explore alternatives** - Always propose 2-3 approaches before settling
- **Incremental validation** - Present design, get approval before moving on
- **Be flexible** - Go back and clarify when something doesn't make sense

## Visual Companion

A browser-based companion for showing mockups, diagrams, and visual options during brainstorming. Available as a tool — not a mode. Accepting the companion means it's available for questions that benefit from visual treatment; it does NOT mean every question goes through the browser.

**Offering the companion:** When you anticipate that upcoming questions will involve visual content (mockups, layouts, diagrams), offer it once for consent:
> "Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)"

**This offer MUST be its own message.** Do not combine it with clarifying questions, context summaries, or any other content. The message should contain ONLY the offer above and nothing else. Wait for the user's response before continuing. If they decline, proceed with text-only brainstorming.

**Per-question decision:** Even after the user accepts, decide FOR EACH QUESTION whether to use the browser or the terminal. The test: **would the user understand this better by seeing it than reading it?**

- **Use the browser** for content that IS visual — mockups, wireframes, layout comparisons, architecture diagrams, side-by-side visual designs
- **Use the terminal** for content that is text — requirements questions, conceptual choices, tradeoff lists, A/B/C/D text options, scope decisions

A question about a UI topic is not automatically a visual question. "What does personality mean in this context?" is a conceptual question — use the terminal. "Which wizard layout works better?" is a visual question — use the browser.
