# Systematic Debugging

## Overview

Random fixes waste time and create new bugs. Quick patches mask underlying issues.

**Core principle:** ALWAYS find root cause before attempting fixes. Symptom fixes are failure.

**Violating the letter of this process is violating the spirit of debugging.**

## The Iron Law

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

If you haven't completed Phase 1, you cannot propose fixes.

## When to Use

Use for ANY technical issue:
- Test failures
- Bugs in production
- Unexpected behavior
- Performance problems
- Build failures
- Integration issues

**Use this ESPECIALLY when:**
- Under time pressure (emergencies make guessing tempting)
- "Just one quick fix" seems obvious
- You've already tried multiple fixes
- Previous fix didn't work
- You don't fully understand the issue

**Don't skip when:**
- Issue seems simple (simple bugs have root causes too)
- You're in a hurry (rushing guarantees rework)
- Manager wants it fixed NOW (systematic is faster than thrashing)

## The Four Phases

You MUST complete each phase before proceeding to the next.

### Phase 1: Root Cause Investigation

**BEFORE attempting ANY fix:**

1. **Read Error Messages Carefully**
   - Don't skip past errors or warnings
   - They often contain the exact solution
   - Read stack traces completely
   - Note line numbers, file paths, error codes

2. **Reproduce Consistently**
   - Can you trigger it reliably?
   - What are the exact steps?
   - Does it happen every time?
   - If not reproducible → gather more data, don't guess

3. **Check Recent Changes**
   - What changed that could cause this?
   - Git diff, recent commits
   - New dependencies, config changes
   - Environmental differences

4. **Gather Evidence in Multi-Component Systems**

   **WHEN system has multiple components (CI → build → signing, API → service → database):**

   **BEFORE proposing fixes, add diagnostic instrumentation:**
   ```
   For EACH component boundary:
     - Log what data enters component
     - Log what data exits component
     - Verify environment/config propagation
     - Check state at each layer

   Run once to gather evidence showing WHERE it breaks
   THEN analyze evidence to identify failing component
   THEN investigate that specific component
   ```

   **Example (multi-layer system):**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **This reveals:** Which layer fails (secrets → workflow ✓, workflow → build ✗)

5. **Trace Data Flow**

   **WHEN error is deep in call stack:**

   See `root-cause-tracing.md` in this directory for the complete backward tracing technique.

   **Quick version:**
   - Where does bad value originate?
   - What called this with bad value?
   - Keep tracing up until you find the source
   - Fix at source, not at symptom

### Phase 2: Pattern Analysis

**Find the pattern before fixing:**

1. **Find Working Examples**
   - Locate similar working code in same codebase
   - What works that's similar to what's broken?

2. **Compare Against References**
   - If implementing pattern, read reference implementation COMPLETELY
   - Don't skim - read every line
   - Understand the pattern fully before applying

3. **Identify Differences**
   - What's different between working and broken?
   - List every difference, however small
   - Don't assume "that can't matter"

4. **Understand Dependencies**
   - What other components does this need?
   - What settings, config, environment?
   - What assumptions does it make?

### Phase 3: Hypothesis and Testing

**Scientific method:**

1. **Form Single Hypothesis**
   - State clearly: "I think X is the root cause because Y"
   - Write it down
   - Be specific, not vague

2. **Test Minimally**
   - Make the SMALLEST possible change to test hypothesis
   - One variable at a time
   - Don't fix multiple things at once

3. **Verify Before Continuing**
   - Did it work? Yes → Phase 4
   - Didn't work? Form NEW hypothesis
   - DON'T add more fixes on top

4. **When You Don't Know**
   - Say "I don't understand X"
   - Don't pretend to know
   - Ask for help
   - Research more

### Phase 4: Implementation

**Fix the root cause, not the symptom:**

1. **Create Failing Test Case**
   - Simplest possible reproduction
   - Automated test if possible
   - One-off test script if no framework
   - MUST have before fixing
   - Use the `superpowers:test-driven-development` skill for writing proper failing tests

2. **Implement Single Fix**
   - Address the root cause identified
   - ONE change at a time
   - No "while I'm here" improvements
   - No bundled refactoring

3. **Verify Fix**
   - Test passes now?
   - No other tests broken?
   - Issue actually resolved?

4. **If Fix Doesn't Work**
   - STOP
   - Count: How many fixes have you tried?
   - If < 3: Return to Phase 1, re-analyze with new information
   - **If ≥ 3: STOP and question the architecture (step 5 below)**
   - DON'T attempt Fix #4 without architectural discussion

5. **If 3+ Fixes Failed: Question Architecture**

   **Pattern indicating architectural problem:**
   - Each fix reveals new shared state/coupling/problem in different place
   - Fixes require "massive refactoring" to implement
   - Each fix creates new symptoms elsewhere

   **STOP and question fundamentals:**
   - Is this pattern fundamentally sound?
   - Are we "sticking with it through sheer inertia"?
   - Should we refactor architecture vs. continue fixing symptoms?

   **Discuss with your human partner before attempting more fixes**

   This is NOT a failed hypothesis - this is a wrong architecture.

## Red Flags - STOP and Follow Process

If you catch yourself thinking:
- "Quick fix for now, investigate later"
- "Just try changing X and see if it works"
- "Add multiple changes, run tests"
- "Skip the test, I'll manually verify"
- "It's probably X, let me fix that"
- "I don't fully understand but this might work"
- "Pattern says X but I'll adapt it differently"
- "Here are the main problems: [lists fixes without investigation]"
- Proposing solutions before tracing data flow
- **"One more fix attempt" (when already tried 2+)**
- **Each fix reveals new problem in different place**

**ALL of these mean: STOP. Return to Phase 1.**

**If 3+ fixes failed:** Question the architecture (see Phase 4.5)

## your human partner's Signals You're Doing It Wrong

**Watch for these redirections:**
- "Is that not happening?" - You assumed without verifying
- "Will it show us...?" - You should have added evidence gathering
- "Stop guessing" - You're proposing fixes without understanding
- "Ultrathink this" - Question fundamentals, not just symptoms
- "We're stuck?" (frustrated) - Your approach isn't working

**When you see these:** STOP. Return to Phase 1.

## Common Rationalizations

| Excuse | Reality |
|--------|---------|
| "Issue is simple, don't need process" | Simple issues have root causes too. Process is fast for simple bugs. |
| "Emergency, no time for process" | Systematic debugging is FASTER than guess-and-check thrashing. |
| "Just try this first, then investigate" | First fix sets the pattern. Do it right from the start. |
| "I'll write test after confirming fix works" | Untested fixes don't stick. Test first proves it. |
| "Multiple fixes at once saves time" | Can't isolate what worked. Causes new bugs. |
| "Reference too long, I'll adapt the pattern" | Partial understanding guarantees bugs. Read it completely. |
| "I see the problem, let me fix it" | Seeing symptoms ≠ understanding root cause. |
| "One more fix attempt" (after 2+ failures) | 3+ failures = architectural problem. Question pattern, don't fix again. |

## Quick Reference

| Phase | Key Activities | Success Criteria |
|-------|---------------|------------------|
| **1. Root Cause** | Read errors, reproduce, check changes, gather evidence | Understand WHAT and WHY |
| **2. Pattern** | Find working examples, compare | Identify differences |
| **3. Hypothesis** | Form theory, test minimally | Confirmed or new hypothesis |
| **4. Implementation** | Create test, fix, verify | Bug resolved, tests pass |

## When Process Reveals "No Root Cause"

If systematic investigation reveals issue is truly environmental, timing-dependent, or external:

1. You've completed the process
2. Document what you investigated
3. Implement appropriate handling (retry, timeout, error message)
4. Add monitoring/logging for future investigation

**But:** 95% of "no root cause" cases are incomplete investigation.

## Supporting Techniques

These techniques are part of systematic debugging and available in this directory:

- **`root-cause-tracing.md`** - Trace bugs backward through call stack to find original trigger
- **`defense-in-depth.md`** - Add validation at multiple layers after finding root cause
- **`condition-based-waiting.md`** - Replace arbitrary timeouts with condition polling

**Related skills:**
- **superpowers:test-driven-development** - For creating failing test case (Phase 4, Step 1)
- **superpowers:verification-before-completion** - Verify fix worked before claiming success

## Real-World Impact

From debugging sessions:
- Systematic approach: 15-30 minutes to fix
- Random fixes approach: 2-3 hours of thrashing
- First-time fix rate: 95% vs 40%
- New bugs introduced: Near zero vs common

## 2. Defesas atuais vs vetores ainda abertos

| Camada | Defesa implementada | Cobre fake-Firefox? |
|--------|---------------------|---------------------|
| nginx `$cleaned_args` (strip QS) | Sim — TEC/jet_download/UTM | Parcial (apenas se URL atacada tiver QS conhecida) |
| nginx `cf_crawlers` rate-limit (2r/s, burst=5) | UA = "Amazon CloudFront" ou vazio | **NÃO** — fake-Firefox não casa o map |
| nginx `wp_login` rate-limit (5r/m) | Apenas `/wp-login.php` e `/xmlrpc.php` | **NÃO** — para HTML público |
| nginx `$deny_meta_html` 429 | Apenas UA `meta-externalagent` | **NÃO** |
| nginx day-view 301 + REST tribe rate-limit | Captura URLs específicas TEC | Parcial |
| robots.txt físico (`bit-robots v1.9.0`) | Disallow TEC + `jet_download` + `/wp-json/tribe` | **NÃO** — bots maliciosos ignoram robots |

| Alertas BIT Monitoring | CloudFront 4XX/5XX, EC2 CPU 80%/90% (10 min) | Detecta, não bloqueia |

**Diagnóstico de gaps:**

1. **UA forjado**: nenhuma camada inspeciona UA suspeito (versão Firefox inexistente, ausência de Sec-CH-UA, headers atípicos). Toda a defesa por UA é deny-list de strings literais.
2. **Rate-limit por IP cliente independente de UA**: inexistente. `cf_crawlers` só dispara quando UA = `Amazon CloudFront` ou vazio. Um IP único enviando 100 req/min com UA "Mozilla/5.0 Firefox/142.0" passa direto.
3. **Anomaly detection no WAF**: AWS WAF tem `Bot Control` e rate-based rules (por IP, com threshold em N/5min). Nenhuma está ativa.
4. **Cloudflare WAF**: site usa Cloudflare Tunnel apenas em DEV; PROD vai direto ALB → CloudFront → ALB → EC2. Não há camada Cloudflare em prod.
5. **Robots.txt regrediu** (incidente 2026-04-20): após blue-green, WP gerava virtual sem Disallows TEC. Corrigido com versão idempotente, mas **não há teste smoke pós-deploy** que valide presença das diretivas.
6. **FPM em 20 (não 27)**: decisão conservadora — RAM permite ~27 workers, mas vCPUs*10=20 é teto seguro com WP Rocket bypass. Não é gargalo nesse incidente (workers não estavam OOM, estavam ocupados em queries lentas).
7. **CloudWatch**: existem alertas de CPU/4XX/5XX por instância (PRs #129–137), mas **não há alerta para FPM saturação direto** (workers em estado R, queue depth) — débito técnico já registrado.

## 3. Padrão comum aos 3 (4) incidentes

Todos atacam **endpoints dinâmicos sem cache CloudFront**: TEC archives (Month/Week View com `usort()` sobre `tribe_events`), JetEngine listings com filtros, `/?jet_download`. Carga deriva de **produto cartesiano de QS funcionais** (data × view × ical × eixo × tax). Workers FPM ficam 30–60s presos cada; saturam com ~1.4 reqs/s. WP Rocket bypass não ajuda porque QS funcional pula o cache.

## 4. Recomendações priorizadas

### Curto prazo (24–48h) — fechar fake-Firefox

1. **Rate-limit por IP cliente independente de UA** em `nginx.conf`: `limit_req_zone $binary_remote_addr zone=client_html:10m rate=10r/s;` aplicado em `location /` com `burst=20 nodelay`. Browsers reais não passam disso; bot single-IP quebra. Compatível com CloudFront porque `real_ip_module` já resolve `X-Forwarded-For`.
2. **AWS WAF rate-based rule por IP**: adicionar regra `RateLimit-Per-IP` no `ACL-WPAdminHML`, threshold 100 reqs/5min em `URI` que case `^/eventos/|^/?$|^/conhecimento/`. Bloqueio temporário (10 min). Custo trivial.

### Médio prazo (1–2 semanas) — detecção de UA forjado

3. **AWS WAF Bot Control (Common Bot Pattern set)**: módulo gerenciado da AWS detecta UA inconsistente (Firefox sem Sec-CH-UA, Chrome sem Accept-CH, etc.). Cobre o vetor "fake-Firefox/142.0". Custo ~US$10/mês + tráfego — viável.

### Longo prazo (1 mês+) — observabilidade

4. **OTel `hostmetrics` receiver com scraper `load`** + alerta load-avg &gt;4 sustentado 5 min (débito técnico registrado em incidente 2026-04-20). Detecta saturação antes do CloudFront 5XX.
5. **Smoke test pós-deploy** validando: (a) `robots.txt` físico contém Disallows TEC; (b) `pm.max_children` no valor esperado; (c) `fastcgi_read_timeout` 20s no PHP default location; (d) WAF `Block-AggressiveBots` contém os 9 UAs canônicos.

## Diagnóstico

A configuração **PERMITE** cache da espiral. O bot bater no origin com URLs cacheable não deve estar acontecendo na maioria dos casos — exceto em duas situações claras:

1. **Cache key explosiva por `eixo + tax + jsf` (combinatória)**: a cache policy whitelista 4 QSs. Para uma única tax (ex.: `eixos`), há ~30 combinações `eixo` × ~150 valores de `tax` (`eixos:2401`, `eixos:2402`, …) × `jsf=jet-engine:estudos|estudos-cultura|...`. **Produto cartesiano de milhares de cache entries**. Bot que itera pares `eixo+tax` distintos gera MISS em sequência — cada um cai no ALB.

2. **Origin Shield em sa-east-1** mitiga em parte (consolidação regional), mas **não bloqueia floods de combinações novas**. POPs distantes podem repetir MISSes para o shield, que repassa novas combinações ao ALB.

3. **Bug latente**: `_label` **não está na cache key** mas é semanticamente relevante para o JetEngine listing (define o rótulo do filtro). Como CF não diferencia `_label` distintos, **a primeira resposta cacheada é servida para todos os labels** — isso é um bug funcional que pode aparecer no UX se `_label` afetar o conteúdo renderizado pelo JetEngine. Confirmar se é apenas decorativo (provável) ou se altera filtros realmente.

4. **Sem `Cache-Control` no response**, o CF cacheia pelo DefaultTTL=86400s (1 dia). WP-Rocket está ativo mas não está emitindo `Cache-Control: public, max-age=...` para essas URLs — provável que estejam **excluídas do page cache** do WP-Rocket por regex de QS dinâmico, então cada MISS do CF custa **render PHP completo** (JetEngine query JSF + listing). Quando o bot itera milhares de combos, cada MISS custa CPU/FPM no ALB.

## Recomendações concretas

1. **Adicionar `_label` à cache key** ou explicitamente removê-lo via CF Function. Alternativa preferida: viewer-request function que strip `_label` antes do CF gerar a cache key, reduzindo redundância (já que não muda conteúdo). Custo zero, ganho de hit-rate.

2. **Cache Control headers no nginx para listings JetEngine**: emitir `Cache-Control: public, s-maxage=300, max-age=60` (ou similar) para `*/conhecimento/espiral-de-conhecimento/*` quando a request **NÃO** tem cookie de logged-in. Isso permite que WP-Rocket bypass + CF cacheie por 5 min — derrubando MISSes consecutivos do bot na MESMA combinação.

3. **WAF rate limit por path-prefix (Priority 7 já existe `RateLimit-300-Block`)**: validar se está com escopo `URI starts_with /conhecimento/`. Se for global (300 req/5min em qualquer path), o bot pode pulverizar entre páginas e nunca disparar. Adicionar regra `RateLimit-Listings-Strict` com **threshold ~100/5min** específico para `/conhecimento/*` ou `/eventos/*` quando `User-Agent contains bot|crawl|spider|http`. Reaproveitar padrão `Block-AggressiveBots` (Priority 6 já existe via `OrStatement`).

Origin Shield já está habilitado em sa-east-1 — bom. O modelo cache-friendly atual é correto, mas com 4 QSs whitelistadas e zero `Cache-Control` do origin, a superfície de MISSes é grande demais para resistir a um crawler determinado.</result>

## Achados estruturais (causas raiz)

### 1. WP Rocket page cache estava AUSENTE no momento do ataque

Diretórios `/conhecimento/espiral-de-conhecimento/`, `/sobre-nos/4-amazonias/`, `/conhecimento/publicacoes/` **não tinham `index-https.html`** — foram gerados DURANTE meus testes (mtime 04:39-04:41 UTC = momento da medição). Após geração, TTFB cai para 0.001s. **Cache foi invalidado/limpo recentemente** e bot atacou justo no janela cold.

### 2. Redirects 301 demoram 5-27s — bug crítico

NGINX retorna 301 mas leva DEZENAS de segundos. Hipótese: o `redirect_canonical` do WordPress carrega o WP completo (boot + JetEngine + Elementor) só para emitir um redirect de trailing slash. **301 deveria ser servido pelo NGINX a 1ms**, mas está caindo no PHP. Configurar redirect canônico no NGINX (`rewrite ^(.+[^/])$ $1/ permanent;`) eliminaria 100% desse custo.

### 3. URLs com query string bypass o WP Rocket por design

`/eventos-calendario/lista/?shortcode=*` passa por `$cleaned_args` (mapeamento em `nginx.conf:40`). Os shortcodes `f5de2e49`, `05031a23` não estão na whitelist de strip → tratados como "query_string" → `$rocket_skip_reason=query_string` → bypass cache. Bot explorou exatamente isso.

### 4. REST API `/wp-json/*` excluído do cache por regex

`$rocket_cache_reject_uri` exclui `wp-json` explicitamente. `/wp-json/wp/v2/pages/2461` (236 KB JSON) custa 8s e SEMPRE cai no PHP. Atacante visou essa URL deliberadamente.

### 5. /eventos-calendario/category/* e categoria/* duplicados

Slugs PT-BR e EN coexistem (`/category/webinar` + `/categoria/webinar`), ambos retornando 301. **Custo dobrado** sem benefício.

## Mitigações propostas

### 1. Trailing slash em nginx-level (HIGH IMPACT, low risk)
Adicionar **antes** do `try_files` em `location /`:
```nginx
# Trailing slash redirect — só para rotas sem extensão e sem arquivo físico
if (!-e $request_filename) {
    rewrite ^(/(?!wp-admin|wp-content|wp-includes|wp-json|feed)[^.?]*[^/])$ $scheme://$host$1/ permanent;
}
```
**Impacto**: elimina 24% dos 301s (~116/dia). Custo cai de 3-13s para &lt;50ms.
**Risco**: precisa excluir `/wp-admin`, `/wp-json`, `/wp-cron.php`, arquivos físicos. Testar em HML primeiro com X-Test-Green. **Atenção**: não pode interferir com rewrites multisite existentes (linha 23-25 do config) — colocar **antes** do `if (!-e)` multisite.

### 2. Bloquear bots agressivos para URLs 301-prováveis (HIGH IMPACT)
DotBot + BacklinksExtendedBot + meta-externalagent emitem ~140 reqs/dia que viram 301. Adicionar ao `map $deny_meta_html` (já existe v1.12.2) os UAs DotBot/BacklinksExtendedBot retornando 429:
```nginx
map $http_user_agent $deny_seo_bots { default 0; "~*DotBot|BacklinksExtended|MJ12bot|AhrefsBot" 1; }
```
**Risco**: nenhum (bots SEO sem valor de negócio). Aplica só ao HTML; libera `/wp-content/uploads/`.

### 3. Cache de 301 no CloudFront (MEDIUM IMPACT)
Hoje 301 não retorna `Cache-Control`. Adicionar no nginx:
```nginx
# Cache 301s no CDN por 1h
location / {
    error_page 301 = @cache_redirect;
    ...
}
```
Ou via WP filter `wp_redirect` adicionar header. **Impacto**: 301 viaja ao CF apenas 1x; subsequentes do mesmo URL são CDN-cached. Reduz repetição de bots.
**Risco**: baixo. CF já honra `Cache-Control: max-age=3600` em 3xx se configurado.

### 4. Mapa explícito de slug-renamed redirects em nginx (MEDIUM IMPACT)
Os redirects "slug renomeado" (`/sobre-nos/agenda-integradora` → `/agenda-integradora/`) custam 13-16s mas são **estáveis** (não mudam). Capturar os top 30 do log e materializar em nginx `map`:
```nginx
map $request_uri $slug_redirect {
    /sobre-nos/agenda-integradora /agenda-integradora/;
    /atuacao/projetos-estruturantes/itinerarios-amazonicos/ /itinerarios-amazonicos/;
    /publicacoes /conhecimento/publicacoes/;
    /eixos/eixo-1/page/3/ /page/3/;
}
```
**Impacto**: ~50-80 reqs/dia bypassando WP. **Risco**: drift se WP mudar slug — mitigar com revisão trimestral. Não escala bem.

### 5. TEC slug consolidação (MEDIUM IMPACT, requires testing)
`/eventos-calendario/categoria/X` é redundância de slug PT (TEC tem `categoria` E `category`). Adicionar nginx-rewrite para normalizar:
```nginx
location ~ ^/eventos-calendario/(category|categoria)/([^/]+)/?$ {
    return 301 /eventos/categoria/$2/;  # TEC base atual
}
```
**Risco**: médio — confirmar `/eventos/categoria/X/` é a URL canonical antes de aplicar.

## 8. Cálculo de capacidade ideal — t3.large (2 vCPU, 7.6 GB)

**Limite memory** (250 MB/worker real, não 50 MB):  
`(7600 - 1000 SO - 500 OPcache - 500 buffer) / 250 ≈ 22 workers`

**Limite CPU** (CPU-bound — listing JetEngine ~10s/req):  
`2 vCPU × 5 = 10 workers efetivos`. Workers além disso só **enfileiram** (latência sobe, throughput não).

**Conclusão matemática:** com requests CPU-bound, **mais workers ≠ mais throughput**. Aumento de 20→25 workers serve apenas como **buffer de fila** durante spike de bot — *não* resolve o problema raiz (URLs lentas).

# Review 2 — Segurança e Blast Radius

**Spec:** `2026-05-02-spiral-502-mitigation.md`
**Reviewer:** Senior Code Reviewer (security/blast-radius lens)
**Data:** 2026-05-02

## Resumo executivo

O spec é tecnicamente sólido, com rollback documentado por item e tiers temporais bem definidos. No entanto, identifico **3 riscos altos**, **6 riscos médios** e diversas lacunas de hardening que devem ser tratadas antes do sign-off. O blast radius está majoritariamente contido a uma única instância EC2 / um WebACL / uma distribuição CloudFront — não há propagação multi-site nem mudanças em IAM/Security Groups, o que reduz superfície sistêmica.

---

## Riscos aceitáveis (blast radius bem contido)

- **Item 2 (cron warmer)** — atinge somente `localhost` via curl, não escala carga externa. Falha = ausência de pré-aquecimento (degradação suave). Rollback é remover linha do crontab.
- **Item 8 (slowlog FPM)** — overhead &lt; 1ms, log local, logrotate configurado (14 dias). Não afeta tráfego.
- **Item 12 (map redirects estáticos)** — escopo: 5 paths conhecidos. Falha = 301 errado em path específico, detectável por smoke.
- **Item 15 (CloudWatch alarms)** — observability-only, zero efeito sobre tráfego.
- **Item 9 (OPcache file_cache)** — restart 1-2s em janela noturna; rollback restaura `.ini.bak` e limpa cache. Disco isolado.

---

## RISCO ALTO

### 1. Item 1 (Redis prefix) — colisão silenciosa com HML pode corromper sessões/transients

**Cenário:** o spec já reconhece que `WP_REDIS_PREFIX='hml:'` é drift de import. Isso significa que **HML e PROD compartilham o mesmo Redis** (ou compartilharam em algum momento). Trocar para `'prd:'` sem auditar HML pode colidir caso HML também já esteja usando esse prefix. Pior: **keys com prefix antigo `hml:` ficam órfãs no Redis** ocupando memória até o TTL natural expirar — em transients sem TTL explícito, podem persistir indefinidamente (memory leak silencioso).

**Mitigação proposta:**
- Antes do Item 1, executar: `redis-cli -s /var/run/redis/redis-server.sock --scan --pattern 'hml:*' | wc -l` e `redis-cli --scan --pattern 'prd:*' | wc -l`.
- Se `prd:*` retornar &gt; 0, escolher prefix único `concertacao_prd:` (sugestão 1467 do próprio spec — elevar de "se vazio" para **default**).
- Após cutover, agendar `redis-cli --scan --pattern 'hml:*' | xargs redis-cli del` em janela de 30 dias para limpar órfãs.

### 2. Item 3 (nginx Referer block) — falsos positivos em ferramentas de monitoria/uptime

**Cenário:** o spec afirma "tráfego legítimo NUNCA tem Referer literal sem path", mas isso é falso para vários casos reais:
- **Pingdom, UptimeRobot, StatusCake** podem enviar Referer literal do domínio configurado quando o probe é setado por URL raiz.
- **Slack/Discord/WhatsApp link unfurl bots** ocasionalmente enviam Referer simplificado.
- **AMP cache do Google** e **Cloudflare's Radar** são casos limítrofes.
- **Bots internos do BIT** (smoke tests, link checkers) podem ter sido configurados sem Referer com path.

**Mitigação proposta:**
- Adicionar **modo COUNT antes de BLOCK**: usar `return 444` apenas após 24h de logging com `access_log /var/log/nginx/referer-naked.log` + análise dos UAs/IPs que fariam match.
- Excluir IPs do range AWS sa-east-1 (probes internos), Pingdom/UptimeRobot ranges públicos.
- Documentar runbook: como o cliente vai diagnosticar se um terceiro reclamar de "site não abre" (resposta 444 = TCP reset, sem body, debug difícil).

### 3. Item 4 (trailing-slash redirect) — quebra OAuth callbacks e webhooks com URL hardcoded

**Cenário:** O regex `^([^.]*[^/])$` redireciona qualquer path sem ponto e sem `/` final. Casos reais que quebram:
- **OAuth callbacks** registrados em terceiros (Google, Facebook, WPML.org) com URL `https://concertacaoamazonia.com.br/oauth/callback` — terceiro envia POST, redirect 301 transforma em GET e perde o body.
- **Webhooks** de Stripe, Mailchimp, SES events — POST com payload JSON é redirecionado e o body é perdido.
- **REST API custom endpoints** sem extensão (ex.: `/wp-json/wp/v2/posts` é coberto pela exclusão do `wp-json`, mas custom routes podem não ser).
- **Slug "/2026.04"** já citado no risco 4 do spec — mas o spec não lista quais slugs reais existem.

**Mitigação proposta:**
- Auditoria prévia obrigatória: `wp post list --post_status=publish --fields=ID,post_name | grep -E '^\d+\s+[^.]+$'` + dump completo de `permalink_structure` + lista de webhooks externos.
- Excluir explicitamente: `wp-admin`, `wp-json`, `wp-login.php`, `xmlrpc.php`, `oauth`, `webhook`, `callback`, `tribe-bar`.
- Testar **POST** em URL sem trailing-slash em dev antes de produção: `curl -X POST -d '{}' https://cambrasmax.local:8484/wp-json/...` deve retornar 200, não 301.

## RISCO MÉDIO

### 4. Item 6 (update_meta_cache batch) — memory exhaustion em listings grandes

Listing JetEngine de 415 estudos (citado no roteiro) com 29 meta-keys/item = ~12k entries de cache em um único `update_meta_cache` call. Cada entry ~500 bytes → ~6MB de RAM por request, multiplicado por 20 workers = 120MB pico simultâneo. Aceitável, mas no **listing paginado sem limit**, se o filter capturar todos os 415 IDs em vez do batch da página atual, sobe para ~200MB em um único worker — risco de hit no `pm.max_children=20 × 245MB`.

**Mitigação:** o filter `jet-engine/listing/grid/posts` deve garantir que `$posts` é o batch da página, não o conjunto total. Validar em dev com listing grande antes de prod. Adicionar `if (count($ids) &gt; 100) { return $posts; }` como circuit breaker.

### 5. Item 10 (CF Function regex) — auto-DoS em 6-12 meses

**Cenário:** regex `Firefox\/(1[4-9]\d|[2-9]\d\d)` bloqueia Firefox 140+. Firefox stable em 2026-05 é 138, mas em 2026-11 será ~142, e em 2027-05 será ~146. Sem maintenance reminder, em 6-12 meses o site vai começar a bloquear usuários reais.

**Mitigação:**
- Criar **schedule remote agent** para semestralmente revisar e bumpar a regex.
- Adicionar logging na CF Function (CloudWatch) para alarmar se taxa de 403 subir &gt; 0.5% (sinal de que o regex pegou um browser real).
- Documentar a regex em comentário com data de revisão obrigatória: `// REVISAR ATÉ 2026-11-01`.

### 6. Item 11 (Bot Control COUNT 7d) — falsos positivos em VPN corporativa

**Cenário:** AWS Bot Control marca CGNAT/VPN corporativa como suspeita por causa de fingerprint compartilhado. Usuários internos do BIT e do cliente Concertação podem usar NordVPN/corporate VPN. 7 dias COUNT é mínimo razoável, mas **não há plano para excluir whitelisted IPs**.

**Mitigação:**
- Adicionar IP set de exceção contendo: ranges NordVPN BR/IT (já documentados em CLAUDE.md raiz), IPs estáticos do escritório do cliente (a colher), CIDR do BIT.
- Após 7 dias COUNT, exportar `SampledRequests` e filtrar por User-Agent e ASN antes de promover para BLOCK.

### 7. Item 14 (upgrade t3.xlarge) — Reserved Instance perdido

**Cenário:** spec não menciona se `i-059febbd720286cd4` está em Reserved Instance ou On-Demand. Se RI 1-yr/3-yr, upgrade para t3.xlarge gera **custo duplo** (RI pago sem uso + nova OD). Valor potencialmente perdido: USD 60-300/mês × meses restantes.

**Mitigação:**
- **Pré-requisito explícito do Item 14:** `aws ec2 describe-reserved-instances --filters Name=state,Values=active --profile Concertação` para identificar RI vinculada à instance.
- Se RI ativa → opções: (a) modificar RI via `modify-reserved-instances` para t3.xlarge (mesma família, possível), (b) vender no Marketplace, (c) postergar Item 14 até expirar.

### 8. Item 5 (WAF FORWARDED_IP + FallbackBehavior=MATCH) — bloqueio de Health Checks ALB

**Cenário:** ALB health checks para `/health` chegam à origin sem passar pelo CloudFront, logo **sem header X-Forwarded-For**. `FallbackBehavior=MATCH` faz a regra de rate limit aplicar a esses requests com fallback no edge IP. Em theory health checks vêm direto pro ALB e nunca passam pelo WAF do CF — confirmar topologia. Se houver path em que health check passa pelo CF, vai entrar na bucket de rate limit do edge.

**Mitigação:** confirmar topologia (CloudFront → ALB ou rotas paralelas?) e excluir health check user-agent do rate limit via scope-down statement.

### 9. Item 13 (surrogate cache Redis) — cache poisoning + stale crítico

**Cenário:** se a chave de cache não considerar **estado do post (publish/private/draft)**, um editor pode salvar um draft e o cache servir conteúdo antigo público por 6h+ (se TTL alto). Pior: se um editor logado vê preview e o cache key não tem `is_user_logged_in` (apesar de estar listado no spec), o preview vaza para visitor.

**Mitigação:** spec sub-document deve incluir teste de "edit-and-flush" de 30s, e TTL máximo deve ser ≤ 1h. Cache key deve incluir `post_status` filter resolvido na query.

## Item #1 — Redis drop-in + prefix

Status: AJUSTE necessário.

A spec corrige `WP_REDIS_PREFIX` e ativa o drop-in, mas **não trata o cache de strings WPML pós-ativação**. Após `wp redis enable`, todas as caches `icl_cache_class` (`locale_cache_class`, `flags_cache_class`, `language_name_cache_class`, `wpml_translatable_cache`) ficam vazias e populam sob demanda. Isso é benigno — `icl_cache` (`wordpress/wp-content/plugins/sitepress-multilingual-cms/inc/cache.php`) usa `wp_cache_get/set` com fallback graceful.

Pontos a adicionar:

- VERIFICAR: o WPML String Translation guarda transients em `wp_options` que podem virar stale (`_transient_wpml_translations_*`). Após ativar Redis, executar `wp transient delete --all` é seguro e elimina entries duplicadas que ficariam em DB.
- AJUSTE: **multisite** — `WP_REDIS_PREFIX='prd:'` é compartilhado entre blog 1 e blog 2 sem distinção. O plugin redis-cache já adiciona blog_id no namespace via `wp_cache_get` group, mas vale validar com `redis-cli keys 'prd:*'` que existem chaves para `2:*` (blog 2). Se não, problema.
- BUG potencial em `feedback_redis_postmeta_invalidation.md`: `wp_cache_incr("last_changed")` foi a estratégia adotada para invalidar postmeta. Ao ativar drop-in agora, validar se as invalidações de save_post existentes em `bit-tec-cache.php` continuam funcionando. **Não há risco crítico, mas precisa smoke pós-deploy**.
- Não existe `wp wpml-cli` para flush — o WPML usa apenas `icl_cache_clear()` interno e não expõe CLI público.

Risco do plano: BAIXO. Recomendação: adicionar passo 8 ao Item 1: `wp transient delete --all &amp;&amp; wp wpml-st refresh-translations` (sub-comando do wpml-string-translation).

## Item #6 — `update_meta_cache` batch — **BUG CRÍTICO**

Status: BUG. **O hook proposto não existe**.

A spec usa `add_filter('jet-engine/listing/grid/posts', ...)`. Auditoria do código real do JetEngine em `wordpress/wp-content/plugins/jet-engine/includes/components/listings/render/listing-grid.php` (linha 1594):

```
$query = apply_filters( 'jet-engine/listing/query/items', $query, $settings, $this );
```

O hook correto é **`jet-engine/listing/query/items`** (não `grid/posts`). Recebe `$query` como array de `WP_Post` e é o ponto certo para `update_meta_cache`. Reescrever o mu-plugin do Item 6:

```php
add_filter('jet-engine/listing/query/items', function ($query, $settings, $render) {
    if (!is_array($query) || empty($query)) return $query;
    $ids = array_filter(array_map(fn($p) =&gt; is_object($p) ? (int)$p-&gt;ID : 0, $query));
    if (count($ids) &gt; 1) update_meta_cache('post', $ids);
    return $query;
}, 10, 3);
```

Outros pontos:

- WPML: `update_meta_cache('post', $ids)` cobre apenas IDs do idioma corrente. Como `wp_query` do JetEngine já é filtrado por `lang`, **não é preciso aquecer IDs traduzidos** — eles só seriam carregados se houver `apply_filters('translate_post_meta', ...)` cross-language, o que não é o caso do listing 28187.
- **CCT (`wp_jet_cct_*`) NÃO usa `wp_postmeta`** — tabela própria com colunas. A intervenção `update_meta_cache` é inerte para CCT. Listing 28187 é de `estudos` (post-type CPT, `wp_postmeta` aplicável) — OK. Mas **se um dia migrarem para CCT, o fix vira inútil sem alarme**. Documentar limitação.
- Conflito com `bit-elementor-local-cache.php` v2.1.2: nenhum. O `debug_backtrace` daquele plugin roda em `wp_get_attachment_url`, não em `get_post_meta`. Independentes.
- `jet-engine/listing/query/items` dispara **antes** da resolução AJAX do JSF, então o batch funciona tanto no render inicial quanto em filtragens AJAX.

## Item #7 — `purge_cron_interval` 10h → 24h

Status: OK com ressalva.

`save_post` dispara `rocket_clean_post()` automaticamente via WP Rocket. **VERIFICAR**:

- TEC events salvos via REST API (`/wp-json/tribe/events/v1/events`) podem **não disparar** `save_post` em todos os casos — alguns sub-events do TEC usam `wp_insert_post` direto. Validar com `wp eval 'add_action("save_post_tribe_events", function($id){ error_log("save_post fired for $id"); });'` em dev.
- JetEngine CCT records (tabela própria) **NÃO disparam `save_post`** — usa hook próprio `jet-engine/custom-content-types/data/before-update`. WP Rocket NÃO conhece esse hook → CCTs **dependem** do purge horário/diário para invalidar. Aumentar para 24h pode cachear estado stale por 24h em UIs que listam CCTs (ex.: listagens de participantes).
- Listings dinâmicos (espiral) atualizam quando posts `estudos` mudam (save_post sim) ou quando termos de eixo mudam (`edited_term`/`created_term` — WP Rocket conecta via `clean_term`).

Recomendação: manter 24h, mas **adicionar hook explícito para CCT** em mu-plugin novo:
```php
add_action('jet-engine/custom-content-types/data/after-insert', 'rocket_clean_domain');
add_action('jet-engine/custom-content-types/data/after-update', 'rocket_clean_domain');
```
ou versão cirúrgica via `rocket_clean_post(get_option('page_on_front'))` se afetar apenas a home.

## Item #4 — nginx trailing-slash

Status: AJUSTE — múltiplos riscos com WPML/multisite.

- JetEngine CCT permalinks (se houver) usam `/{cct-slug}/{record-id}/` com trailing-slash padrão.

Recomendação: testar em dev com lista exaustiva de URLs (incluindo `/wp-sitemap.xml`, `/feed`, `/wp-json/...`, `/cultura/wp-admin`, `/cultura/atlas/?lang=en`, `/wp-content/uploads/foo.jpg`).

## Item #14 — t3.xlarge + Aurora

Status: VERIFICAR — gargalo Aurora subestimado.

- Aurora db.t3.medium (2 vCPU, 4GB) — `max_connections` padrão em Aurora MySQL 8 t3.medium é ~90 (`{DBInstanceClassMemory}/12582880`). Atual FPM=20 → 20 connections. Após upgrade FPM=40, ainda dentro de 90.
- Mas Aurora t3.medium é **burstable** — sustained CPU 35-40% durante pico esgota credits em 90min. Após esgotar, baseline 20% causa connection wait.
- Recomendação: subir Aurora para `db.t3.large` ou (preferível) `db.r6g.large` (4GB → 16GB, sustained, sem credit). Custo ~USD 80/mês delta — **considerar antes do EC2 upgrade**.

# Auditoria 2 — Análise Técnica/Correctness das Mitigações Aplicadas (2026-05-02 / 2026-05-03)

**Escopo:** Items 1 (Redis), 3 (Referer block), 8 (Slowlog), 16 (Dedupe `previous_ecp_versions`) já em produção. mu-plugin novo `bit-tec-versions-dedupe.php` v1.0.0.

---

## 1. mu-plugin `bit-tec-versions-dedupe.php`

**Verdict: correto com ressalvas — atenção**

Hook `pre_update_option_tribe_events_calendar_options` priority 5 está **OK** — TEC e WPML não registram filters ali (verificado em pesquisa de código TEC Pro 7.x; WPML opera nos `option_*`, não em `pre_update_option_*`). WP Rocket também não escuta esta option específica. Priority 5 dá folga para outro plugin se interpor depois (priority 10) sem race.

`return $value` para não-array é **defensivo correto** — `update_option` aceita scalar/null em casos exóticos (resets de instalação). Hook de filtro WP **sempre recebe `$value` completo**, nunca diff (confirmado em `wp-includes/option.php`).

`array_unique` default usa `SORT_STRING` — converte tudo para string antes de comparar. Versions tipo `"6.0.1.1"` (string) e `6.0.1.1` (numeric-string PHP) seriam tratadas como **iguais**. TEC armazena sempre como string serializada (`add_option` → `maybe_serialize`). Sem risco prático.

**Atenção 1 — threshold `&gt; 100` cego para drift incremental.** Se TEC inflar de 11→50 (cresce 39 entries em 6 meses), o log nunca dispara. Isso vai mascarar o início de uma reinflação lenta.

**Atenção 2 — `array_values()` reseta keys.** TEC consome via `version_compare()` em `tribe_events_is_new_install()` que itera com `foreach`/`usort` — não depende de keys numéricas específicas. Seguro.

**Atenção 3 — `array_unique` é O(n²) em worst-case** com PHP &lt; 8.0, O(n log n) com PHP 8+. Em prod (PHP 8.3) a performance é fine; com 11 entries é inerte.

---

## 2. Conflito potencial com `bit-tec-cache.php`

**Verdict: BUG operacional latente — atenção crítica**

O `bit-tec-cache.php` v1.2.0 cacheia `tribe_get_option('previous_ecp_versions')` por 24h via transient `bit_tec_ecp_versions`. **Se o transient já estava preenchido com o valor inflado (103.917 entries) ANTES do dedupe ser aplicado, ele continuou servindo o array gigante por até 24h.**

A invalidação manual descrita no spec do dedupe (`update_option(...)`) **não invalida o transient** — `set_transient` grava em `wp_options` com prefixo `_transient_`, e o filter `tribe_get_option_previous_ecp_versions` priority 1 retorna o transient antes de qualquer leitura ao option real. **Status updates log não menciona `delete_transient('bit_tec_ecp_versions')` no Item 16.**

Como o ganho de TTFB foi observado (-52% médio), provavelmente o transient não estava setado no momento (Redis recém-corrigido no Item 1, transients ficam no DB ou Redis), ou foi limpo coletoramente por `transient delete --all` no pós-aplicação do Item 1 (R3, linha 398 do spec). **Mas isso é coincidência — não está documentado como step.**

Ordem de carregamento alfabética: `bit-tec-cache.php` carrega antes de `bit-tec-versions-dedupe.php`. Os dois operam em hooks diferentes (`tribe_get_option_*` vs `pre_update_option_*`), sem race.

## 3. Item 1 (Redis) — validação técnica

## 4. Item 3 (nginx Referer) — edge cases

**Verdict: correto, com lacuna conhecida — atenção**

O `map` cobre 4 strings literais sem trailing slash. Edge cases analisados:

- **Referer com query string sem path** (`https://concertacaoamazonia.com.br?utm=x`) — **NÃO bloqueia** (não bate no map). Bot real provavelmente não usa esse padrão.
- **Referer com fragment** (`#anchor`) — fragmentos NÃO são enviados em Referer header pelo navegador (RFC). Não-issue.
- **Mixed-case** (`HTTPS://CONCERTACAOAMAZONIA.COM.BR`) — `map` é **case-sensitive por default** no nginx. Bot que mude case escapa. Trivial mitigar com `map ... { hostnames; }` ou regex `~*`.
- **`return 444`** fecha TCP RST. CloudFront/proxies CDN **não retentam 444** (é resposta nginx-internal sem status code HTTP visível). Sem amplificação.
- **Convivência com Item 4 (trailing-slash)** — `if` executa antes de `rewrite`, então o 444 fecha conexão antes do redirect. OK na ordem proposta no spec.

## 5. Item 8 (Slowlog) — segurança e privacidade

**Verdict: correto — atenção a logrotate**

`request_slowlog_trace_depth=30` grava **apenas backtrace de funções/arquivos PHP, NÃO request body nem $_POST/$_GET**. LGPD safe — não há PII.

Path `/var/log/php8.3-fpm-slow.log` (note: spec usa `php8.3-fpm-slow.log`, claudeMd usa `php-fpm-wordpress-slow.log` — divergência nominal a reconciliar). Perms 644 owned by www-data — **644 é world-readable**, qualquer usuário shell no servidor lê traces. Não é exfiltration crítica (zero PII), mas **prefer 640** alinhado com práticas Bureau IT.

Logrotate criado com `daily rotate 14 compress` — **OK**. Tamanho do log com `request_slowlog_timeout=5s` em produção saturada pode crescer rápido (cada trace ~3-5KB × 100 traces/h = ~10MB/dia). 14 dias × 10MB = 140MB — disk seguro.

**Verdict: explicação encontrada — atenção a path adicional**

Pós-dedupe, slowlog mostrou 21 traces de `tribe_events_is_new_install()` + `usort()`. **Hipótese mais provável:**

**Conclusão:** o `bit-tec-cache.php` não cobre 100% dos paths. Ainda assim, com array em 11 entries, o impacto é negligenciável — slowlog está reportando trace incidental, não causa raiz.
